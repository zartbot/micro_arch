SM Microarchitecture

This chapter presents the complete internal architecture of the SM_120 Streaming Multiprocessor, encompassing pipeline topology, dispatch model, co-issue rules, instruction encoding, and sub-pipe structure. The SM_120 employs a strict 4-SMSP (Sub-Core) single-issue architecture with 16 distinct execution pipes, a redesigned operand collector that eliminates bank-conflict penalties, and a compiler-guided scheduling model with 6 hardware scoreboard counters. The characterization is derived from 249 microbenchmark cases across Sections 11 (156 cases), 12 (37 cases), 13 (instruction encoding), and 22 (56 NCU pipe-verification cases). Key findings include: .reuse bits are functionally active (+85% ILP throughput impact), 10 of 15 pipe pairs achieve full co-issue, and the memory pipe queue depth (12-13) significantly exceeds prior architectures.

The SM_120 Sub-Core (SMSP) is the fundamental execution building block of the Streaming Multiprocessor — four identical, fully independent SMSPs compose each SM, collectively delivering up to 4 instructions per cycle at the SM level while each individual SMSP adheres to a strict single-issue-per-cycle discipline. Each SMSP hosts a pool of 12 resident warps (48 total per SM), a private register file of 16,384 × 32-bit entries, a dedicated warp scheduler, a decode stage, an operand collector with register file cache (RFC), and six distinct execution pipe groups: the FMA-lite pipe (handling FP32 multiply-add, FP16/BF16 packed arithmetic, and simple FP operations at 4-cycle latency and 1.0 ops/cycle throughput), the FMA-heavy pipe (integer multiply-add IMAD, bit manipulation LOP3/SHF/PRMT/BFE, and select/min-max operations at 4–5 cycles), the XU/SFU pipe (transcendental approximations MUFU.RCP/SIN/COS/EX2/TANH at 18–44 cycles with a pipeline depth of 5 allowing 5 simultaneous in-flight operations), the LSU pipe (global/shared/atomic loads and stores with a remarkably deep MSHR queue of 12–13 outstanding requests per warp), the U-pipe (warp-uniform operations executing once for the entire warp via a separate 64-entry uniform register file), and the TensorCore pipe (matrix multiply-accumulate HMMA/QMMA/IMMA supporting FP16, BF16, TF32, FP8/FP6/FP4 with MX scaling, and INT8 at 29-cycle latency with a sustained throughput of 15.88 cycles per MMA operation).

The instruction lifecycle within an SMSP proceeds through a well-defined pipeline: the warp scheduler selects one eligible warp per cycle using a hybrid greedy-then-youngest (CGGTY) policy with zero-cycle context-switch overhead (since all warp state resides locally in the RF), then fetches the next instruction from the per-SMSP L0 I-cache (~4 KB, backed by a shared 64 KB L1 I-cache with a streaming prefetcher that fully hides L0 miss latency for straight-line code). The 128-bit instruction word is decoded into its two logical halves: the lower 64-bit opcode word (operation and operands) and the upper 64-bit control word containing the 17-bit scheduling metadata — a 4-bit stall counter specifying minimum inter-issue delay, a 1-bit yield hint (inverted semantics), two 3-bit scoreboard IDs (WrBarrier for producers, RdBarrier for consumers), a 6-bit WaitBarrier bitmask gating issue on SB completion, and a 4-bit reuse field providing per-operand RFC caching hints. The scheduler trusts the compiler-set stall counter absolutely for fixed-latency instructions (wrong values produce silent data corruption, not stalls), while variable-latency operations (LDG, LDSM, TMA) are tracked by 6 hardware scoreboard counters that provide completion notification to dependent consumers.

The register file and operand delivery system represents a key area of SM_120's hardware-software co-design philosophy. The 16,384-entry RF employs a 2-bank architecture with bank assignment determined by simple register number parity (R%2: even registers in Bank 0, odd in Bank 1), providing 2 read ports per bank. When all three source operands reside in the same parity bank, a +1.0 cycle penalty is incurred from the additional read cycle needed (ceil(3/2) = 2 read phases). However, the ptxas compiler's parity-aware register allocator ensures that this conflict never manifests in practice — it is observable only through CuAssembler SASS binary patching that bypasses the compiler entirely. Fronting the RF is a Register File Cache (RFC) with ≥24 entries per warp (substantially larger than SM_70's ~8 or SM_80's ~16 entries), which caches recently-read operands and provides them directly to the operand collector on subsequent accesses. The RFC's generous capacity, combined with the 4-bit .reuse hints in the control word, delivers a measured +85% throughput improvement for ILP-rich ALU code by eliminating RF read cycles for cached operands. The operand collector itself acts as a staging buffer that gathers all source operands — from the RFC (0-cycle hit), the RF banks (1-cycle normal, 2-cycle on bank conflict), or the bypass network (forwarding results directly from execution units) — before dispatching the fully-formed operation to the target execution pipe.

Within the broader SM architecture, the four SMSPs operate as fully independent processing elements: each has its own scheduler, decode logic, RF, RFC, and complete set of execution pipe backends. Warps are statically bound to sub-cores at block launch via round-robin-by-4 assignment (warps 0–3 to SC0, 4–7 to SC1, etc.) and never migrate, eliminating cross-sub-core synchronization overhead at the cost of potential load imbalance. The only shared resources are the L1 I-cache (64 KB), the L1 data cache, the shared memory banks, and the TMA engine. This isolation means that cross-SMSP pipe contention is exactly zero — each SMSP's pipe backends operate independently (verified at <0.2% variance). The co-issue matrix measured across the six pipe groups reveals that 10 of 15 pipe pairs achieve full or near-full overlap: FMA-lite co-issues perfectly with FMA-heavy (1.001×), FMA with TensorCore (1.00×), FMA with XU/SFU (1.00×), and TensorCore with distributed shared memory (1.00×), while partial contention exists only for LSU + Tensor consumer paths (0.78×) due to writeback port sharing when LDS results feed into FMA consumers.

The performance characteristics of this design yield several notable advantages over prior architectures. The deep MSHR queue (12–13 entries vs SM_89's ~5–6 or SM_90's ~8) enables substantially higher memory-level parallelism per warp, reducing occupancy requirements for memory-bound kernels. The single-issue discipline with compiler-managed scheduling eliminates the area cost of hardware interlocks (saving ~48K comparator entries that a full hardware scoreboard would require for 48 warps × 256 registers), trading silicon area for compiler complexity. The 4-warp latency hiding threshold for FFMA (4 cycles × 1 issue/cycle = 4 warps to saturate) means that even modest occupancy achieves peak compute throughput. And the 29-cycle TensorCore MMA window provides up to 27 dispatch slots for non-MMA instructions to execute "for free" — sufficient to absorb typical GEMM epilogues (bias, activation, quantization) with negligible (<3%) overhead on the critical MMA chain. Together, these design choices position the SM_120 SMSP as an area-efficient, throughput-optimized execution engine that relies on deep compiler cooperation rather than expensive hardware speculation to maximize utilization of its 16 execution pipes.

3.2 128-bit Instruction Encoding

SM_120 uses 128-bit (16-byte) fixed-width instruction encoding. Each instruction word contains two logical halves: an opcode word (lower 64 bits) encoding the operation and operands, and a control word (upper 64 bits) encoding scheduling hints.

3.2.1 128-bit Instruction Layout and Control Code Decomposition

Every SM_120 instruction occupies exactly 128 bits (PC step = 0x10). The upper 64 bits carry scheduling metadata (control codes, reuse hints) while the lower 64 bits encode the operation and operands. The SVG diagram above visualizes the field partitioning.

The 17-bit control code [121:105] embeds all scheduling state inline: a 4-bit stall counter (minimum inter-issue delay), a 1-bit yield hint (inverted semantics: 0 = yield slot), two 3-bit scoreboard IDs (WrBarrier for producers, RdBarrier for consumers), and a 6-bit WaitBarrier one-hot mask gating issue on SB completion. The 4-bit reuse field [125:122] provides per-operand RFC caching hints — functionally active for ALU instructions (+85% ILP throughput) but true no-ops for TC instructions (HMMA/QMMA/OMMA.SF).

3.2.2 Control Code Semantic Summary

The diagram above illustrates how the 17-bit control code embedded in each 128-bit instruction governs the warp scheduler's issue decision. Every cycle, the scheduler evaluates a candidate warp's next instruction through a strict priority-ordered gate sequence:

Gate 1 — WaitBarrier check: The scheduler inspects the 6-bit WaitBarrier[121:116] one-hot bitmask. If any flagged scoreboard counter (SB0–SB5) has a non-zero value — meaning a prior variable-latency operation has not yet completed — the warp is immediately blocked and the scheduler moves to the next candidate. This is the primary mechanism for enforcing memory dependence correctness.

Gate 2 — Stall countdown: If the warp passes the WaitBarrier gate, the scheduler checks whether the stall countdown timer (initialized to stall[108:105] when the previous instruction issued) has reached zero. The countdown decrements by 1 each cycle, enforcing the minimum inter-issue delay that the ptxas compiler encoded for fixed-latency dependencies. A non-zero countdown blocks the warp for that cycle.

Gate 3 — Yield hint: With both hard gates passed, the scheduler consults the yield[109] bit. If yield=0 (inverted semantics: "yield this slot"), the scheduler is encouraged to switch to another eligible warp — though this is a hint, not a hard block. If yield=1, the scheduler prefers to continue issuing from the same warp, maximizing instruction-level locality.

Issue + Side-effects: Upon issuing the instruction, two side-effects occur simultaneously: (a) the stall countdown is loaded with the new instruction's stall value, and (b) if WrBarrier[112:110] ≠ 7, the specified scoreboard counter is incremented (SBn++), registering this instruction as a pending producer whose completion the memory subsystem will later signal by decrementing the same counter.

Reuse flags (orthogonal): The 4-bit reuse field [125:122] operates independently of the scheduling logic — it instructs the operand collector to cache the flagged source operands in the Register File Cache (RFC) so that subsequent instructions reading the same registers get a 0-cycle hit instead of a full RF bank read. This optimization delivers +85% throughput for ALU instructions in ILP-rich code, but is a true no-op for TensorCore instructions which use an independent operand fetch path.

3.2.3 Control Code Behavioral Rules (CuAssembler-Verified)

Stall counter rules (Section 11.8.2, 4 cases):

  1. stall=0: Triggers HW scoreboard fallback -- always correct even for dependencies [source: S11/case_129]
  2. stall=1-3 (< latency): SILENT stale register read -- consumer reads previous value, no trap, no error [source: S11/case_132-135]
  3. stall=4 (= FFMA latency): Correct boundary -- consumer reads valid output [source: S11/case_136]
  4. stall=N (independent): Exact N cycles delay, 1:1 linear, no saturation at stall=15 [source: S11/case_128,131]
  5. Stall counter replaces (not adds to) dependency wait -- stall=2 on 4-cyc dep results in reads after 2 cyc (stale) [source: S11/case_130]

Yield bit rules (Section 11.8.4, 7 cases):

  1. yield=0 + stall=0: Blocks latency hiding -- 32 warps regress to single-warp throughput [source: S11/case_142]
  2. yield=0 + stall>=latency: No effect -- stall already prevents re-issue before completion [source: S11/case_148]
  3. Partial yield (N% of instructions): Proportional blocking -- 25% yield results in 75% throughput [source: S11/case_145]
  4. Yield on producer only: ZERO effect -- it is the consumer's yield that matters [source: S11/case_146]

SB (Scoreboard) rules (Sections 12, 13):

  • 6 SB counters (SB0-SB5), dynamically assigned by ptxas
  • LDG typically uses SB3-4; LDS uses SB0-1 [source: S13/case_12]
  • Co-consumed loads share same SB -- 3 cyc savings vs distinct SB assignment [source: S21/case_105]
  • SB exhaustion cliff: N=5 to 6 outstanding variable-latency ops causes 44.8x stall (17 to 761 cyc, L1-cached path) [source: S11/case_44]; under L2-forced long latency the effective cliff shifts to N=10-11 → 12 (358 to 726 cyc, ~2x) [source: S12/case_24]
  • DEPBAR.LE: Explicit SB wait instruction, used ONLY for cp.async (not standard LDG) [source: S21/case_12]

Reuse bit rules (Section 11.8.5 + S12/case_1 + Case 155):

  • ALU instructions (FFMA/FADD/IMAD): .reuse bits [60:58] are functionally active — +85% throughput in ILP-rich code (stall=1); 0% in latency-bound dep chains (stall≥4) [source: S11/case_155, S12/case_1]
  • TC instructions (HMMA/QMMA/OMMA.SF): .reuse bits are true no-op — ptxas never sets them (bits [60:58]=000); forcing them to 111 produces 0% delta. TC pipeline uses independent operand fetch path, does not use RFC [source: S21/case_78 retest]
  • SM_120's operand collector absorbs RF bank conflicts (+1 cyc penalty) but RFC provides additional caching benefit for ALU pipes

The diagram above captures the complete instruction scheduling workflow of the SM_120. When the warp scheduler dispatches an instruction, it classifies the operation into one of two dependence-tracking paths based on latency determinism — a compile-time decision embedded directly into the 17-bit control word of each 128-bit instruction.

Path 1 — Fixed-latency instructions (left column): For operations whose execution time is known at compile time — FFMA (4 cycles), IADD3 (2 cycles), HMMA (29 cycles), MUFU transcendentals (18–44 cycles), and all other ALU/FP/TC instructions — the ptxas compiler encodes the exact inter-issue delay into the 4-bit stall[108:105] field. The hardware scheduler simply counts down this value before re-issuing the same warp; it performs no RAW hazard check whatsoever when stall ≥ 1. This "trust the compiler" model means that an incorrect stall value produces silent data corruption (the consumer reads stale register contents) rather than a pipeline stall or trap. The sole safety valve is stall=0, which triggers a full hardware scoreboard fallback — always correct but 17× slower than the optimal stall setting for IADD3 (34 cycles vs 2 cycles). In production code, ptxas never emits stall=0 for analyzable instructions; it is reserved as a conservative escape hatch for code paths where static latency analysis is infeasible.

Path 2 — Variable-latency instructions (right column): For operations whose completion time depends on runtime conditions — LDG (L1 hit: 44 cycles, L2: 354 cycles), LDSM, TMA (479+ cycles), cp.async, and all memory/atomic operations — the hardware provides 6 scoreboard counters (SB0–SB5) per warp. The producer instruction's control word sets WrBarrier[112:110] to a scoreboard ID (0–5), incrementing that counter on dispatch. When the memory subsystem completes the operation, it asynchronously decrements the counter. The consumer instruction encodes a WaitBarrier[121:116] bitmask specifying which SB counters must reach zero before issue. This mechanism allows up to 5 fully-overlapping variable-latency operations per warp (using SB1–SB5, with SB0 often reserved); the 6th operation triggers the catastrophic SB exhaustion cliff — a 44.8× stall from 17 to 761 cycles as the warp must serialize behind the oldest outstanding load. Under L2-dominated access patterns, ptxas can share a single SB counter across multiple co-consumed loads, effectively extending the cliff to N=10–12 outstanding operations.

Area efficiency: This hybrid design requires only 41 bits of scheduling state per warp (36 bits for 6×6 SB counters plus 4 stall bits plus 1 yield bit), compared to 256+ bits that a full hardware scoreboard would need for per-register dependency tracking with CAM comparators. Across 48 warps per SM, this represents a savings of over 10,000 transistor-bits — a compelling demonstration of hardware-software co-design where compiler intelligence substitutes for silicon complexity.

3.3.1 Microarchitectural Analysis of the Dispatch Model

The SM_120 dispatch architecture reveals a fundamental design trade-off: maximizing area efficiency through single-issue sub-cores while maintaining high aggregate throughput via warp-level parallelism. The measured data constrains a precise model of the internal pipeline structure.

Evidence for single-issue, not dual-issue: Case 11.1/01 measures INT32+FP32 interleaving at MIX/MAX = 0.076, meaning the mixed workload completes in almost exactly max(INT_time, FP_time) — no dual-issue benefit exists within a single warp. If SM_120 had a second issue slot (as some AMD architectures employ), mixed INT+FP streams would show MIX/MAX approaching 0.5 (50% speedup). The measured 7.6% "overlap" is entirely attributable to the 4-warp interleaving effect across SMSPs in the same block.

I-buffer depth constrains ILP exploitation: The instruction buffer holds approximately 4 entries per warp (Case 11.2/21). This means the scheduler can look ahead by at most 4 instructions when seeking independent operations to issue. With FFMA latency of 4 cycles, ILP=4 exactly fills the pipeline (4 independent chains × 1 cyc issue = 4 cyc covered). Beyond ILP=4, additional parallelism yields only marginal throughput improvement (1.08 → 1.01 cyc/op from ILP=4 to ILP=16). This I-buffer depth is consistent with the 128-bit instruction width: 4 × 16 bytes = 64 bytes, fitting within a single L0 I-cache line.

Result queue is non-blocking: Case 11.1/12 demonstrates zero write-back contention even at ILP=4 with IADD3 (2-cycle latency, 1 cyc/op throughput). This implies at least 4 write ports to the register file per SMSP (one per pipeline stage in flight), or equivalently, a bypass network that forwards results to dependent consumers without queueing. The absence of write-port contention is remarkable for a 4-deep FMA pipeline and suggests either (a) results bypass directly from execution units to the operand collector, or (b) the RF has separate write ports per pipe backend.

Execution unit latch occupancy: The measured IADD3=2 cyc, HADD2=4 cyc, VIADD.16x2=5 cyc pattern (Case 11.1/11) reveals that different ALU sub-pipes have distinct pipeline depths. The 2-cycle IADD3 path implies a simple 2-stage execute+writeback for basic integer addition, while the 4-cycle HADD2 path suggests the FP16×2 datapath uses the full FMA pipeline (likely packed through the fmalite backend). The 5-cycle VIADD path indicates an additional decode stage for the SIMD vector format. Crucially, no half-warp serialization is observed for any operation, confirming that ALL arithmetic units operate on full 32-thread warps in a single pass — there is no half-warp execution on SM_120 (unlike some older AMD RDNA architectures).

Memory pipe queue depth — a deep MSHR architecture: Case 11.6/97 reveals the most architecturally significant finding: SM_120's LSU can sustain 12-13 independent outstanding loads per warp before stalling. The stall cliff is dramatic — from 2 cycles at N=12 to 984 cycles at N=14 (a 492× increase). This behavior is precisely consistent with a fixed-size miss-status holding register (MSHR) that tracks outstanding cache misses:

This deep queue enables SM_120 to sustain much higher memory-level parallelism (MLP) per warp, reducing the occupancy requirements for memory-bound kernels. With 12-13 loads in flight and L2 latency of 354 cycles, the theoretical MLP bandwidth per warp is (12.5 × 128B) / 354 cyc = 4.5 B/cyc — already exceeding the per-SM L2 bandwidth allocation.

Operand Read Workflow — How the Three-Tier System Works: When the warp scheduler issues an instruction (e.g., FFMA R5, R2, R3, R7), the operand collector must gather three source operands before dispatching the operation to the FMA execution pipe. The read path proceeds through a priority-ordered three-tier hierarchy:

Tier 1 (RFC Hit, 0-cycle): The operand collector first checks the Register File Cache (≥24 entries per warp). If the instruction's control word has the corresponding .reuse bit set (bits [125:122]) and the operand was recently read, the RFC supplies it immediately with zero additional latency. This is the fast path that ptxas aggressively optimizes for — in ILP-rich ALU code, RFC hits deliver a measured +85% throughput improvement by entirely eliminating RF bank reads for cached operands.

Tier 2 (RF Normal Read, 1-cycle): On RFC miss, the operand collector reads from the physical register file. With the 2-bank parity architecture (even registers in Bank 0, odd in Bank 1), a 3-source instruction whose operands span both banks completes in a single cycle — each bank provides up to 2 reads simultaneously, and max(ceil(2/2), ceil(1/2)) = 1 cycle suffices. The ptxas compiler's parity-aware register allocator guarantees this configuration for all production code.

Tier 3 (RF Bank Conflict, 2-cycle): If all three source operands reside in the same parity bank (e.g., R4, R8, R12 — all even), the bank's 2 read ports cannot serve 3 requests in one cycle. The read splits into two phases: ceil(3/2) = 2 cycles, imposing a +1.0 cycle penalty. This scenario is observable only through CuAssembler SASS binary patching that bypasses ptxas's conflict-avoidance logic — it never occurs in normally compiled code.

Why zero penalty is observed at the PTX/application level: Case 11.3/46 measures FFMA with PTX-level register constraints and sees 0 cyc delta between "same-bank" and "cross-bank" configurations. This is NOT because bank conflicts don't exist — it is because ptxas's register allocator prevents all-same-parity configurations from ever occurring in compiled SASS. The SASS binary patching in Case 12/01 bypasses ptxas entirely to expose the underlying hardware behavior. The two results are completely consistent: conflicts exist in hardware but are perfectly hidden by the compiler. This represents a textbook example of hardware-software co-design: a simpler 2-bank RF with only 2 read ports per bank saves significant silicon area, while the compiler's intelligence (parity allocation + RFC caching) delivers performance equivalent to — or better than — a more complex multi-port design.

3.3.3 Warp Scheduling Policy: Hybrid GTO with Static Binding

The SM_120 warp scheduler implements a nuanced policy that cannot be described by a single label. The measured evidence constrains a multi-regime model:

Regime 1 — Low occupancy (≤4 warps per SM, 1 per sub-core): Pure greedy

At 1-4 warps, the scheduler behaves as a pure greedy/GTO scheduler: it continues issuing from the same warp until that warp stalls, then switches to the youngest eligible warp. Evidence:

  • Context switch overhead = 0 cycles for 1-16 warps (Case 47). Per-warp FFMA chain time is identical to single-warp baseline (4096 cyc at all warp counts 1-16).
  • Hot-warp bias ratio = 1.000 at 1-4 total warps (Case 25): a compute-ready warp gets 100% of dispatch slots while stalled warps contribute zero interference.
  • Asymmetric workload (1x vs 10x FFMA) produces duration ratio = 9.9x (Case 28), proving each warp progresses independently.

Regime 2 — Medium occupancy (5-16 warps): Fairness transition

As warp count exceeds one per sub-core, the scheduler begins sharing dispatch slots more evenly:

  • Hot-warp bias grows from 1.000 (4W) to 1.39 (8W) to 1.96 (16W) (Case 25). The ready warp no longer gets all slots; stalled warps receive occasional dispatch attempts.
  • Per-warp FFMA time remains constant up to 16W (Case 47), meaning the sharing doesn't hurt individual warp throughput yet — warp interleaving hides the dependency latency.

Regime 3 — High occupancy (17-48 warps): Resource contention

Beyond 16 warps (>4 per sub-core), contention becomes visible:

  • Per-warp FFMA dep chain at 24W: 6.25 cyc/op (+2.25 overhead, Case 47). The sub-core scheduler cannot interleave fast enough to serve all warps without pipeline bubble waste.
  • At 32W: 8.12 cyc/op (+4.12 overhead). Linear scaling with warp count suggests per-sub-core pipeline pressure, not a step function.
  • Hot-warp bias saturates at 2.01 (32W) — the ready warp gets only half the slots.

Static warp-to-sub-core binding (Case 33): SM_120 pins warps to sub-cores at block launch and never migrates them. Mapping rule: round-robin-by-4 (blockWarps 0-3 → SC0, 4-7 → SC1, etc.). Zero binding changes observed across 970 samples. This eliminates cross-sub-core load balancing but simplifies the hardware and ensures zero context-switch overhead (the per-warp RF is local to the sub-core).

3.4.5 Microarchitectural Analysis of Co-Issue Constraints

The co-issue matrix reveals that SM_120's 16 named pipes are NOT 16 fully independent execution ports. Rather, they share dispatch and write-back resources in a structured hierarchy. Three distinct constraint mechanisms emerge from the data:

Constraint 1 — LSU + Tensor consume-path contention (17.4% overhead)

This is the single most important structural hazard for GEMM/attention kernel optimization. The data reveals that the LSU pipe itself is fully independent from the Tensor pipe — ldmatrix + MMA shows 0% overhead (Case 11.7/106). The 17.4% penalty only appears when LDS results must be consumed via FADD/FFMA (Case 22/47).

The diagram above decomposes the single most important structural hazard on SM_120 for GEMM and attention kernel optimization. The key insight is that the 17.4% LSU+Tensor overhead is not caused by contention between the LSU pipe and the Tensor pipe — those are fully independent hardware units. Instead, the bottleneck arises from the consume path: how LDS results are processed after they arrive in the register file.

Path A (LDSM → HMMA, 0% overhead): The ldmatrix.sync.x4 instruction (SASS: LDSM.16x8.M88) loads 4×32-bit matrix fragments from shared memory and writes them directly into consecutive registers (e.g., R4–R7). These registers are then read by HMMA.16816.F32 as source operands. Crucially, no FMA-pipe instruction ever touches this data — the entire flow stays within LSU (load) → RF (buffer) → TC (consume). The RF write port used by LDSM and the one used by HMMA's accumulator writeback are architecturally distinct, so zero serialization occurs.

Path B (LDS → FADD → HMMA, 17.4% overhead): When shared memory values must be accumulated (e.g., softmax partial sums, quantization scale factors), the typical pattern is LDS R10, [addr] followed by FADD R11, R11, R10 (accumulate). The FADD instruction executes on the FMA pipe and writes its result through the FMA pipe's RF write port. Simultaneously, if an HMMA instruction is in flight, its 4×FP32 accumulator writeback also needs the same RF write port. When both FADD and HMMA attempt to write back in the same cycle, the hardware must serialize one — creating a +1 cycle stall per conflict. Over 512 LDS operations interleaved with 64 HMMA operations, this accumulates to 325 cycles (17.4% of the 1863-cycle TC-only baseline).

Production kernel design rule: Use ldmatrix (LDSM) for all shared memory → TC operand feeding in the MMA mainloop. This includes A-tile loads, B-tile loads, and any data that feeds directly into mma.sync operands. Reserve the LDS+FADD pattern for non-TC paths where TensorCore is idle — for example, epilogue scale factor loading after the MMA chain completes, or RMSNorm accumulation in a separate warp-specialization phase. By segregating the two patterns, the kernel achieves the 0%-overhead ideal (LDSM path) for the critical MMA throughput loop.

3.5 Hybrid Dependence Tracking: The Two-Mechanism Model

SM_120 employs a fundamentally different dependence-resolution architecture than textbook out-of-order processors. Instead of a unified hardware scoreboard that tracks ALL producer-consumer relationships, SM_120 splits dependence management into two parallel mechanisms with distinct correctness semantics.

3.5.1 Mechanism 1: Stall Counter (Fixed-Latency Instructions)

The 4-bit stall counter [108:105] encodes the minimum cycles before the scheduler re-issues the SAME warp. CuAssembler binary-patching experiments (Case 11.8.2, patching 511 IADD3 instructions) reveal the exact behavioral semantics:

The stall counter represents SM_120's most radical departure from conventional processor design. Unlike x86 or ARM out-of-order cores where hardware interlocks guarantee correctness regardless of instruction scheduling, SM_120 places absolute, unconditional trust in the compiler for fixed-latency instruction dependence resolution. The diagram above visualizes the three behavioral regimes discovered through CuAssembler binary patching of 511 IADD3 instructions:

Regime 1 (stall=0): Hardware Safety Fallback — When the stall field is zero, the hardware interprets this as "the compiler cannot determine the latency" and activates full dependency tracking via the hardware scoreboard. This is always correct but catastrophically slow: 34.055 cyc/insn for IADD3 vs the optimal 2.008 cyc/insn — a 17× penalty. In production ptxas output, stall=0 is never emitted for analyzable instructions; it exists solely as a conservative escape hatch for pathological code where static latency analysis fails. The counter-intuitive semantics (0 = "maximum delay") arise because zero triggers the special HW fallback path rather than representing a numeric delay.

Regime 2 (1 ≤ stall < latency): Silent Data Corruption — This is the dangerous regime that distinguishes SM_120 from all x86/ARM designs. If the stall counter is set too low (e.g., stall=1 on a 4-cycle FFMA), the consumer instruction issues before the producer's result reaches the register file. The consumer reads whatever value was previously in that register — a stale result from an earlier instruction. There is no trap, no exception, no error flag; cudaGetLastError() returns cudaSuccess. The corruption is silent and deterministic (reproducible across runs), making it extremely difficult to diagnose without knowledge of the stall counter mechanism. This regime is reachable only via CuAssembler SASS patching; ptxas never generates incorrect stall values for deterministic-latency instructions.

Regime 3 (stall ≥ latency): Correct Operation — The stall counter delays same-warp re-issue by exactly N cycles with perfect 1:1 linearity (no saturation, no rounding). At the boundary (stall = exact latency), the producer's result becomes available in the register file on the precise cycle the consumer reads it. Beyond the boundary (stall > latency), the extra delay wastes cycles but produces correct results. The ptxas compiler always targets the exact boundary to minimize wasted cycles while ensuring correctness.

Architectural significance of the trust model: The stall counter is a correctness requirement, not a performance hint. This is fundamentally different from x86's hardware interlocks, where wrong scheduling merely reduces throughput. On SM_120, wrong scheduling produces wrong results silently. This design eliminates the area cost of per-instruction dependency checking (no CAM lookup per source operand, no bypass detection logic) at the cost of placing absolute trust in the compiler. The area savings are substantial: a full hardware scoreboard for 48 warps × 256 registers would require ~48K comparator entries, while the 4-bit counter costs only 4 bits per warp per SMSP.

3.5.2 Mechanism 2: Scoreboard Counters (Variable-Latency Instructions)

The scoreboard system handles all operations whose completion time cannot be predicted at compile time — primarily memory loads (LDG, LDS, LDSM), TMA transfers, and cp.async operations. Unlike the stall counter which encodes a fixed delay, each scoreboard counter acts as a binary semaphore: the producer increments it on dispatch, the memory subsystem decrements it on completion (with 0-cycle visibility delay), and any consumer instruction with the matching WaitBarrier bit set remains blocked until the counter reaches zero.

The most critical performance characteristic of this system is the SB exhaustion cliff — a catastrophic throughput collapse when the number of outstanding variable-latency operations exceeds the available SB counters. SM_120 exhibits two distinct cliff profiles depending on ptxas's encoding strategy: for L1-cached short loads (1 LDG per SB), the cliff hits at N=5→6 with a 44.8× stall; for L2-forced long loads (counter-mode encoding), ptxas packs multiple loads into shared SB counters via DEPBAR.LE batch waits, extending the effective window to N=10-11 before a gentler 2× cliff at N=12. Both regimes are real and reproducible — the difference is entirely in ptxas's compile-time encoding decision.

3.5.3 Yield Bit: Inverted Semantics and Latency Hiding Control

The yield bit [109] controls whether the warp scheduler should consider switching to another eligible warp after dispatching the current instruction. Its semantics are inverted relative to the naive interpretation: yield=0 means "yield this slot to other warps" (release), while yield=1 means "do not yield, keep dispatching from me" (retain). This inverted encoding was confirmed by CuAssembler binary patching of 4232 FFMA instructions across a full yield×stall×warp-count test matrix (Case 142).

The most important behavioral finding is that the yield bit only has observable effect when stall=0 (the hardware scoreboard fallback mode). When the stall counter is set correctly (stall ≥ instruction latency), it already prevents same- warp re-issue for the required number of cycles — making the yield hint redundant. The dangerous combination is stall=0 + yield=0: this creates a pathological state where the hardware scoreboard activates (adding ~34 cyc/insn overhead) AND the scheduler refuses to switch warps, resulting in 2169 cycles regardless of how many warps are available. Even 32 warps cannot hide the latency because the scheduler is locked onto the stalled warp.

The yield bit has inverted semantics on SM_120, confirmed by comprehensive CuAssembler patching (Case 142):

             → Allows other warps to fill pipeline bubbles
             → Enables latency hiding via warp interleaving

  yield = 1: NO YIELD (retain the scheduler slot)
             → Warp keeps dispatching from its own instruction stream

  CRITICAL: This is OPPOSITE to the naive interpretation where

The yield bit only matters when stall=0 (hardware scoreboard mode). When stall ≥ instruction latency, the stall counter already prevents re-issue, making the yield bit irrelevant. The critical scenario is stall=0 + yield=0: this completely blocks latency hiding even with 32 warps available, because the scheduler keeps trying to dispatch from the stalled warp instead of switching.

ptxas default policy: yield=1 (no yield) on ALU instructions (greedy same-warp dispatch for throughput), yield=0 at barriers and EXIT (release slot for synchronization). This is the optimal policy for most workloads.

3.5.4 Interaction Between the Two Mechanisms

The diagram above shows how SM_120's two dependence-tracking mechanisms cooperate as a parallel AND gate: the stall counter provides a minimum delay floor (ensuring fixed-latency producers complete before consumers read), while the scoreboard counters provide a variable-latency completion gate (ensuring memory operations finish before dependent computation proceeds). Both conditions must simultaneously be TRUE for the scheduler to mark a warp as eligible for issue. If either gate fails, the warp remains blocked and the scheduler moves to the next candidate warp in its CGGTY priority queue.

The concrete example illustrates the typical GEMM scheduling pattern: an LDG producer fires with stall=1 (fire-and-forget, SB3 tracks it), followed by 3 independent ALU instructions that fill the latency window (each with stall=1, no WaitBarrier). When the FFMA consumer arrives, it encounters BOTH mechanisms: the WaitBarrier bitmask (0b001000) blocks it until SB3 reaches zero (LDG complete), and once issued, its own stall=4 field gates downstream consumers via the fixed-latency path. This demonstrates how a single instruction can be both an SB consumer (waiting for LDG) and a stall-counter producer (gating its own dependents) — the two mechanisms compose naturally without interference.

The stall counter and SB counters operate in parallel with a simple priority rule: 

  Issue condition:
    warp_eligible = (stall_counter == 0) AND (wait_mask & active_SB == 0)

  Both must be satisfied: the stall counter provides a minimum delay floor,
  while the SB provides a variable-latency completion gate.

  Typical scheduling pattern for an LDG → FFMA consumer:

    LDG R5, [R2]      ; WrBarrier=SB3, stall=1 (issue next insn after 1 cyc)
    ... (3 independent ALU ops, stall=1 each) ...
    FFMA R7, R5, R6   ; WaitBarrier=0b001000 (wait SB3), stall=4

  The LDG issues with stall=1, allowing subsequent independent instructions
  to issue quickly. The FFMA consumer waits for SB3 completion (variable-
  latency gate) AND then uses stall=4 for its own consumers (fixed-latency).

This two-mechanism design is remarkably efficient in silicon: only 6×6 = 36 bits of SB state + 4 bits of stall + 1 bit of yield per warp, versus the hundreds of bits a full scoreboard would require. It achieves correctness comparable to hardware interlocks while consuming a fraction of the die area.

3.6 Key Insights

KI-3.1: Unified INT/FP Pipeline with Separate Backends -- Not True Dual-Issue

Empirical Evidence: INT32+FP32 interleaving shows MIX/MAX=1.000 with zero throughput gain over max(INT,FP). Yet they do NOT block each other when independent -- FP32 and INT32 fully overlap at multi-warp level. [source: S4/case_5, S11/case_101]

Theoretical Analysis: SM_120 has a shared issue slot per SMSP feeding separate execution backends. Independent INT/FP streams overlap via warp interleaving but serialize within a single warp. Expected behavior: no single-warp dual-issue but full multi-warp parallelism.

Microarchitecture Speculation [Confidence: H]: The FMA group (pipe_fma, pipe_fmalite, pipe_fmaheavy) shares a single dispatch port that can only accept one instruction per cycle. The separate backends execute concurrently once dispatched, but the issue port is the bottleneck. This is architecturally simpler than true dual-issue (no second decoder, no dependency check between issue slots) while still achieving high throughput via warp-level parallelism. Patent US20230289189A1 Section 3.2 (Fig. 4) and US11768686 Section 5.1 describe this single-issue sub-core model.

Implication for Optimization: Do not expect INT+FP "free" co-issue within a single warp. Instead, ensure sufficient warp occupancy (>=4 warps per SMSP) to overlap INT and FP streams across warps. Kernel designs that rely on per-warp INT+FP co-issue (common on AMD RDNA) will not see benefits on SM_120.

KI-3.2: .reuse Bits Are Functionally Active — RFC Still Operational on SM_120

Empirical Evidence (Corrected): Cross-gen verification on SM_89 and SM_120 using correct bit positions [60:58] of the upper 64-bit control word demonstrates .reuse is functionally active on both architectures. SM_89: +126% (2.26×) in ILP=8; SM_120: +85% (1.85×) in ILP=8. Both show 0% delta in latency-bound dep chains (stall=4). [source: S11/case_155, SM_120 server1 ILP=8 verification]

Microarchitecture Interpretation [Confidence: H]: The RFC remains functional on SM_120 for ALU pipes and provides significant throughput benefit in ILP-rich code. In latency-bound dep chains (stall≥4), .reuse provides no benefit. For TC instructions (HMMA/QMMA/OMMA.SF), the pipeline uses an independent operand fetch mechanism — ptxas never sets .reuse bits on TC instructions (confirmed: 456 HMMA all have bits [60:58]=000), and forcing them to 111 produces 0% delta. Patent US11336476 describes the enhanced operand collector — it absorbs bank conflicts but the RFC caching mechanism remains active for ALU pipes only.

Implication for Optimization: .reuse bits ARE performance-critical for throughput-bound ALU (FFMA) code on SM_120. ptxas correctly sets these bits in ILP-rich regions. For CuAssembler-based SASS optimization, preserving .reuse bits on ALU instructions is essential — clearing them causes up to 85% throughput loss. For TC instructions, .reuse bits are irrelevant (never set, no effect).

KI-3.3: 10 of 15 Pipe Pairs Achieve <=2.2% Switching Overhead

Empirical Evidence: FMA+TC=2.4%, ALU+TC=2.2%, FMA+LSU=0%, ALU+LSU=0%, XU+LSU=0%, FMA+FP64=0.4%, FMA+TMA=0%, Tensor+XU=0%. Only XU-involved (132-143%) and LSU+Tensor (17.4%) show significant overhead. [source: S22/case_19-53]

Theoretical Analysis: The 29-cycle MMA execution window provides ample time for other pipes to execute without contention. At IPC=1, up to 27 non-MMA instructions can issue during one MMA. The XU penalty comes from micro-op decomposition (MUFU requires multiple internal cycles that occupy the issue slot), not from switching. The LSU+Tensor penalty arises from LDS consumer instructions (FADD after ldmatrix) that contend with HMMA writeback to the register file.

Microarchitecture Speculation [Confidence: H]: Each SMSP has physically independent execution backends connected to a shared issue port via a pipe-selection mux. The mux settles in <1 cycle, making switching penalty zero. The observed overheads are from structural hazards (shared RF write port for LSU+TC, XU occupying issue slot for multi-cycle) rather than switching cost.

Implication for Optimization: In TC-dominated kernels (>80% of cycles in MMA), all non-TC computation (epilogue FFMA, address calculation, data conversion) is effectively free. The only pairs to avoid co-scheduling are: (1) XU-heavy sequences alongside FMA/ALU (use MMA stall window for MUFU instead), and (2) ldmatrix immediately followed by TC -- separate them by >=2 independent instructions.

KI-3.4: Finer Instruction Interleaving Always Better -- 0% Switching Penalty

Empirical Evidence: Switching overhead does NOT increase with interleave granularity (group=1 same as group=8). The 8.3% FMA+ALU penalty is from shared issue slot contention, not switching cost. [source: S22/case_26]

Theoretical Analysis: If switching cost existed, fine-grained interleaving (alternating every instruction) would show higher overhead than coarse batching. The uniform 0% penalty across all group sizes confirms the mux model.

Microarchitecture Speculation [Confidence: H]: The pipe selection mux settles within 1 cycle; no pipeline flush or reconfiguration is needed when switching between pipe targets.

Implication for Optimization: Instruction scheduling should prioritize dependency satisfaction and latency hiding over grouping same-pipe instructions. ptxas already does this effectively. For manual SASS optimization, interleave different pipe types freely without penalty concerns.

KI-3.5: Memory Pipe Queue Depth Unexpectedly Large (12-13, Not 5)

Empirical Evidence: LDG queue stall cliff at N=14 (2 to 984 cyc); effective depth 12-13. Exceeds Ampere's reported ~5. Memory-Level Parallelism (MLP) approximately 5.3x from N=1 to N=16. [source: S11/case_97, S11/case_41]

Theoretical Analysis: With L2 latency of 354 cycles and an issue rate of 2.27 cyc/LDG, a queue depth of 12-13 provides effective MLP of 354/(354/13) = 13x theoretical. Measured 5.3x accounts for cache misses and pipeline stalls.

Microarchitecture Speculation [Confidence: H]: SM_120's LSU has a deeper miss-status holding register (MSHR) or equivalent request buffer than prior generations. This may be enabled by the larger L2 (96 MB) which can absorb more outstanding requests without thrashing. The 6 SB counters can track at most 6 independent variable-latency ops per warp, so the 12-13 queue depth is shared across all warps on an SMSP.

Implication for Optimization: Software prefetch is counterproductive (see Chapter 2, KI-2.4) because the hardware can already sustain 12-13 outstanding loads. Kernels should instead focus on ensuring that loads are issued early enough (via instruction reordering) to fill the queue, rather than adding explicit prefetch instructions that compete for issue slots.

KI-3.6: The 1-Issue-Per-Cycle Model Creates a "Dispatch Tax" That ILP Cannot Avoid

Empirical Evidence: FFMA IPC=0.92 (not 1.0) even at ILP=16 with zero dependencies. The 8% dispatch tax is from loop control (BRA/ISETP) consuming the same single issue slot. 4-SMSP aggregate IPC=3.66 (not 4.0). [source: S22/case_2-4, S11/case_15]

Theoretical Analysis: With 29 cyc MMA latency and 15.88 cyc/mma throughput, pipeline depth = 29/15.88 = 1.83. Only 1.83 MMA can be in flight per SMSP. The remaining 27.17 cycles of the MMA window are available for non-MMA instructions. At IPC=1 dispatch rate, up to 27 non-MMA instructions can be dispatched per MMA cycle -- more than sufficient for typical GEMM epilogues (256 FFMA = 0.25 cyc/FFMA alongside MMA, per Section 4.7).

Microarchitecture Speculation [Confidence: H]: The single-issue model is a deliberate area/power optimization -- removing the second decoder, dependency checker, and issue port saves significant silicon while warp-level parallelism (4 SMSPs x 12 warps each = 48 warps) provides aggregate throughput.

Implication for Optimization: Kernel optimization should focus on reducing TOTAL instruction count (including control flow overhead) rather than pipe-specific throughput. ptxas loop unrolling directly eliminates BRA/ISETP overhead at the cost of I-cache pressure (cliff at 8-16KB per Chapter 7). The optimal unroll factor balances dispatch tax reduction against I-cache miss penalty.

KI-3.7: Register File Has 2 Banks (R%2) with +1 cyc Conflict — Fully Hidden by ptxas

Empirical Evidence: CuAssembler SASS binary patching of 3,520 FFMA instructions across 22 register configurations proves SM_120 has 2 RF banks (bank = R%2, even/odd parity). All-even or all-odd source operands cost 3.034 cyc/FFMA (+1.0 cyc penalty). Mixed parity costs 2.038 cyc/FFMA (baseline). BUT: PTX-level tests show zero delta because ptxas always assigns mixed-parity operands. This contradicts Huerta 2025's 4-bank (R%4) model. [source: S12/case_1 (retest), S11/case_46, S11/case_149-156]

Theoretical Analysis: The RF has 2 read ports per bank. Reading 3 operands from the same bank requires ceil(3/2) = 2 read cycles, adding +1 cycle. Mixed parity allows max(ceil(2/2), ceil(1/2)) = 1 cycle, no penalty. The RFC (≥24 entries per warp) provides an additional bypass path that allows 1.342 cyc/FFMA when .reuse bits are active (34% faster than conflict-free-no-reuse at 2.038 cyc).

Microarchitecture Model [Confidence: H]: SM_120's operand delivery is a three-tier system: (1) RFC hit → 0 cycles added, (2) RF read with mixed parity → 0 cycles added, (3) RF read with all-same parity → +1 cycle. ptxas guarantees Tier 1 or Tier 2 for all production code, making Tier 3 unreachable without SASS binary patching. The system-level result is zero observable penalty despite the simpler (2-bank, 2-port) RF hardware. This is a remarkable example of hardware-software co-design.

Implication for Optimization: For CuAssembler-based optimization on SM_120, register bank avoidance is irrelevant FOR NORMAL CODE (ptxas handles it). For SASS binary patching, maintain mixed-parity operand assignments or the +1 cyc penalty manifests. The optimization headroom from manual SASS patching is LIMITED to: (1) stall counter adjustment (+6.1% for HMMA chains), (2) SB counter grouping (+3 cyc savings), and (3) yield bit manipulation. The .reuse bits are confirmed no-ops for TC instructions only (ALU .reuse remains functional at +85%).

3.8 Summary

The SM_120 microarchitecture represents an evolution toward simplicity-at-scale: a single-issue sub-core model with 16 pipes, zero bank-conflict overhead, and deep memory queues. The key architectural insight is that warp-level parallelism (48 warps across 4 SMSPs) substitutes for instruction-level complexity (dual-issue, OoO). This makes occupancy and warp diversity the primary performance levers, while traditional SASS-level optimizations (.reuse, bank avoidance) are rendered obsolete by the redesigned operand collector.
