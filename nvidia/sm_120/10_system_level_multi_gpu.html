System-Level & Multi-GPU

This chapter characterizes the system-level behavior of the NVIDIA SM_120 (GB202, RTX PRO 5000 72GB) in an 8-GPU PCIe Gen5 configuration without NVLink. Drawing from 29 host-device transfer cases (Section 14), 12 GPU-to-GPU P2P cases (Section 15), 49 inter-SM communication cases (Section 17), 6 CTA scheduling cases (Section 21.11), and 6 GDDR7 characterization cases (Section 21.12), we present a complete picture encompassing: PCIe interconnect performance with protocol-level overhead decomposition, host-device data transfer pipeline architecture, peer-to-peer topology with BAR aperture failure analysis, kernel launch latency hierarchy, CTA scheduling microarchitecture with GPC-aware stride-20 distribution, inter-SM signaling cost models, DSMEM-vs-L2 architectural trade-offs, tile scheduling primitives, and multi-GPU collective communication patterns. The absence of NVLink fundamentally shapes multi-GPU strategy: PCIe Gen5 x16 achieves 88-90% of theoretical bandwidth per direction, but the single-link bottleneck (55.8 GB/s) and cross-NUMA penalty (24% degradation) make point-to-point parallelism and pipeline parallelism preferable to AllReduce-heavy tensor parallelism.

10.1 System Topology and PCIe Fabric Architecture

10.1.1 Physical Topology

The 8-GPU server employs a dual-socket CPU with two PCIe switch domains, each hosting 4 GPUs. This creates a 2x4 NUMA topology where intra-NUMA GPU pairs share a PCIe switch and cross-NUMA pairs must traverse the inter-socket QPI/UPI link.

The topology diagram above reveals the three-tier hierarchy that governs all multi-GPU data movement on this platform. At the top level, two CPU sockets define NUMA Node 0 (GPUs 0-3, violet boundary) and NUMA Node 1 (GPUs 4-7, blue boundary), connected through the QPI/UPI inter-socket link. Within each NUMA node, a dedicated PCIe Switch (orange) fans out to four GPUs via independent PCIe Gen5 x16 links, each delivering a measured 55.8 GB/s unidirectional bandwidth. The critical architectural distinction is between the two data paths annotated at the bottom of the figure:

Intra-switch path (e.g., GPU0 ↔ GPU1): Data traverses a single PCIe Switch hop, achieving the full 55.8 GB/s PCIe Gen5 x16 rate. All 12 intra-NUMA pairs (6 per NUMA node) fall into this category, and measurements confirm <0.3% directional asymmetry, indicating the PCIe switch fabric itself is perfectly symmetric. This is the fastest possible GPU-to-GPU path on this platform.

Cross-NUMA path (e.g., GPU0 ↔ GPU4): Data must traverse six hops — GPU → Switch 0 → Root Complex 0 → QPI/UPI → Root Complex 1 → Switch 1 → GPU — incurring a 24% bandwidth penalty (41-45 GB/s vs 55.8 GB/s). The QPI/UPI link introduces both a capacity bottleneck and a directional asymmetry of up to 8.9%, because the QPI port-dependent routing selects different physical paths depending on source-destination address mapping. At small transfer sizes (64 KB), this asymmetry peaks at 23.4%, exposing per-TLP routing overhead that large transfers amortize.

The absence of NVLink (highlighted by the red badge in the figure) is the single most consequential architectural constraint: without GPU-to-GPU dedicated interconnect, all 56 P2P pairs are forced onto the shared PCIe fabric, making topology-aware placement essential. The cudaDevP2PAttrPerformanceRank returns 0 for all pairs, falsely suggesting a flat topology — software must explicitly account for the NUMA boundary to avoid the 24% cross-NUMA penalty. Furthermore, native PCIe atomics are unsupported across all 56 P2P pairs (cudaDevP2PAttrNativeAtomicSupported = 0), meaning cross-GPU atomic operations fall back to system-scope serialization with catastrophic latency penalties (see Section 10.9.5).

10.1.2 PCIe Protocol Overhead Decomposition

The measured PCIe Gen5 x16 throughput of 56.9 GB/s represents 88.9% of the theoretical 64 GB/s. The 11.1% gap decomposes into identifiable protocol-level components:

The waterfall chart above decomposes the 11.1% gap between theoretical and measured PCIe Gen5 x16 bandwidth into two categories. Protocol overhead (orange bars, 4.0 GB/s / 6.3%) encompasses the PCIe specification's own framing costs: the 128b/130b line encoding (−1.54%), which converts the raw 32 GT/s lane rate into usable data bits; TLP (Transaction Layer Packet) headers (−3.1%), where each 16–24 byte header is prepended to every payload transfer; DLLP (Data Link Layer Packet) overhead (−1.2%) for ACK/NAK flow control and credit exchange; and ECRC (−0.6%) for optional end-to-end CRC integrity. These four protocol components are architecturally fixed — they cannot be reduced by software tuning.

System overhead (violet bars, 3.1 GB/s / 4.8%) represents the GPU and host-side DMA pipeline costs that are partially tunable. DMA engine scheduling (−2.0%) reflects the time-multiplexing of the copy engine's command FIFO across pending transfer requests. Driver/staging overhead (−3.3%) captures the CUDA driver's per-transfer setup: command buffer insertion, doorbell writes to the GPU, and (for pageable transfers) the pin-on-demand bounce-buffer cycle. For large pinned transfers, this staging cost is amortized to near-zero; for small or pageable transfers, it dominates total latency (see Section 10.2.1).

The near-symmetric H2D/D2H performance (56.89 vs 56.54 GB/s, 0.6% delta) [source: S14/case_1,2] contrasts with HBM-based systems where read/write asymmetry is inherent. PCIe Gen5 uses independent TX/RX lanes with no bus turnaround penalty, enabling symmetric transfers. The slight H2D advantage likely comes from the host DMA engine's read-ahead prefetcher being more effective than the GPU's write-push path.

Why H2D is 1.5 us slower than D2H: The asymmetry in latency floor (6.5 vs 5.0 us) is the mirror image of the bandwidth asymmetry (H2D slightly faster at saturation). The host-side DMA engine requires additional setup for H2D:

  1. CPU cache line invalidation (H2D): Before reading from host memory, the DMA engine must snoop the CPU cache hierarchy to ensure coherency. This adds ~0.5 us.
  2. PCIe completion ordering (H2D): The host root complex enforces stronger ordering for CPU-initiated reads, requiring an explicit completion TLP before the GPU acknowledges. For D2H, the GPU's fire-and-forget write path skips this handshake.
  3. IOMMU translation (H2D): The host IOMMU translates virtual addresses to physical for the DMA engine. For D2H, the GPU has already resolved addresses via its own TLB.

The combined 1.5 us delta is consistent across all sizes up to 1KB, confirming it is a fixed per-transfer overhead rather than a per-byte cost. [source: S14/case_3]

10.4 Kernel Launch Pipeline

10.4.1 Launch Latency Hierarchy and Decomposition

The diagram above illustrates the complete kernel launch pipeline on SM_120, decomposed into host-side submission and GPU-side dispatch phases separated by the PCIe interface boundary. On the host side, cudaLaunchKernel() executes three sequential stages: argument validation (0.2 µs), command buffer insertion (0.3 µs baseline, scaling to +0.8 µs with 4 KB argument payloads), and a PCIe MMIO doorbell write (0.1 µs) that triggers the GPU’s Command Processor. The total host-side submission overhead is approximately 0.6 µs — a negligible fraction of the end-to-end round-trip.

On the GPU side, the Command Processor hardware frontend executes a four-stage pipeline: command decode (0.2 µs), CTA allocation from the hardware allocation table (0.3 µs), warp creation including register file reservation (0.5 µs), and first instruction issue (0.2 µs). This yields a total GPU-side dispatch latency of approximately 1.5 µs from doorbell receipt to first warp execution. The dominant cost in the host round-trip (5.0–6.4 µs) is the synchronous wait for kernel completion and the return signal traversal across PCIe, which accounts for 3.5–5.0 µs of dead time. Cold-start scenarios introduce dramatic additional latency: the first-ever kernel launch adds +103 µs for PTX→SASS JIT verification, module loading, and GPU resource pool initialization (108.7 µs total, 72.5× warm), while each subsequent unique function incurs +5–7 µs for binary location, symbol resolution, and execution context setup.

10.6 CTA Scheduling Microarchitecture

10.6.1 GPC-Aware Round-Robin Distribution

SM_120 employs a GPC-interleaved round-robin CTA assignment with stride-20 GPC affinity. The first 20 CTAs are assigned to the first SM pair in each of the 10 GPCs:

The diagram above visualizes the hardware CTA scheduler’s assignment pattern for the first 40 CTAs across 5 of 10 GPCs (GPCs 5–9 are symmetric). The scheduler distributes CTA pairs in a stride-2 within TPC pattern: consecutive even/odd CTA pairs (0,1), (2,3), (10,11), etc. always land on the two SMs that share a single TPC. Between GPCs, SM IDs advance by 20 (the number of SMs per GPC), creating a stride-20 inter-GPC interleave that distributes load uniformly across all 10 GPCs before advancing to the next TPC within each GPC. The formal assignment rule is SM_i = CTA_id % 110 (after GPC reordering), with multi-block grids using stride-110 so that block 0, 110, 220, 330 all execute on SM 0. This explains the zero-cost cluster launch measured in Section 10.4.1 (1.54 µs, identical to non-cluster). The DSMEM fabric connecting TPC-local SMs provides the low-latency inter-CTA communication channel (86 cyc barrier) that cluster scheduling is designed to exploit.

MEMBAR cost decomposition: The 549 cyc GPU-scope MEMBAR performs three operations that must complete sequentially:

  1. L1 write-back drain (~120 cyc): All dirty L1 cache lines are flushed to L2. The SM's 89 KB L1 (default carveout, 100 KB physical SRAM) can hold up to 712-768 dirty lines; the L1-to-L2 write path processes ~6 lines/cyc.

The SYS-scope MEMBAR adds ~370 cyc on top of GPU-scope for the PCIe ordering point acknowledgment, which requires a round-trip through the root complex to ensure stores are visible to the CPU. [source: S17/case_7, S8/case_64-66]

The fundamental question: why is DSMEM slower for raw reads but faster for synchronized exchanges?

The two-fabric explanation: SM_120 has two physically separate on-chip networks:

  1. SM2SM fabric (DSMEM path): Low-latency, narrow-bandwidth point-to-point mesh connecting SMs within a cluster. One-way traverse: ~85 cyc. No cache coherency protocol overhead. Barrier completion uses dedicated virtual channels (VC_NONBLOCKING).

  2. GXBAR (L2/global path): High-bandwidth, fully-connected crossbar connecting all SMs to all 12 L2 partitions. Higher latency (~354 cyc round-trip) but higher bandwidth (116 B/cyc per SM at L1 hit). Requires MEMBAR for ordering guarantees.

For raw reads, L2 wins because the GXBAR has higher bandwidth and the L2 cache provides a fast hit path. A single L2 read traverses: SM -> GXBAR -> L2 partition -> GXBAR -> SM in ~354 cyc. A DSMEM read traverses: SM -> uTLB -> TPCXRB -> QX -> EG -> dest SM SRAM -> EG -> QX -> TPCXRB -> SM in ~225 cyc for the smaller fabric, but requires sequential request/response phases that cannot be pipelined as effectively.

For synchronized exchanges, DSMEM wins because the cluster barrier (barrier.cluster.relaxed) costs only 86 cyc and stays entirely on the SM2SM fabric. The equivalent L2-based synchronization requires st.release.gpu (555 cyc MEMBAR) + ld.acquire.gpu (393 cyc) = 948 cyc minimum for a one-way notification, plus the data transfer itself. The barrier.cluster.relaxed is O(1) for 2-8 CTAs because the SM2SM fabric delivers the arrive signal as a single broadcast, not a point-to-point chain. [source: S17/case_26-38]

st.async as the optimal DSMEM primitive: The st.async instruction fires at 2 cyc (issue-only, no scoreboard wait) and delivers data through the SM2SM fabric with hardware completion tracking via mbarrier. The consumer waits on mbarrier.try_wait.parity, which polls the local mbarrier word without traversing any network. Total producer-to-consumer latency: 242 cyc (vs 2045 cyc for L2 release/acquire). The 290x producer-side speedup (2 vs 580 cyc) comes from completely eliminating MEMBAR from the critical path. [source: S17/case_37]

DSMEM sync overhead breakdown: 95% of DSMEM round-trip cost is barrier.cluster.relaxed (85.7 cyc). The mapa address translation is compile-time resolved (0 cyc runtime). The actual DSMEM read throughput (pipelined) is only 2.5 cyc/op -- the 225 cyc latency is entirely from the dependent-chain round-trip through the SM2SM fabric. [source: S17/case_38]

  3. Cross-channel row-activation pipelining: while channel C0 is in tRCD/tRP, channels C1-C11 are servicing other accesses; the MC overlaps row management latency across channels, exposing only ~10 cyc of single-channel tRP at very large WS.

10.13.1 Detailed Cliff Analysis — Identification, Root Cause, and Prevention

Tier 1: Catastrophic Failures (100x+ penalty)

Cross-PCIe atomicCAS (15,000x) — The most extreme cliff on the platform. A atomicCAS targeting peer GPU memory with 32-thread contention costs 42 µs per operation (vs 2.9 ns for local). The root cause is non-commutative: each CAS may fail and retry, and each attempt requires a full PCIe round-trip (read-modify-write). With 32 threads serialized on a single remote L2 partition, total cost = 32 × ~1,300 ns (including retries) ≈ 42 µs. Commutative atomics (ADD) avoid retries and cost only 789 ns (275x, not 15,000x). Detection: SASS shows ATOMS.STRONG.SYS.CAS targeting peer addresses; NCU reports near-zero L2 atomic throughput. Mitigation: Replace CAS-based lock-free structures with commutative reductions, or centralize synchronization on a single GPU.

cudaFree synchronous (244x) — cudaFree() performs an implicit cudaDeviceSynchronize (waits for ALL pending GPU work), then unmaps virtual addresses and returns physical pages. The 61 µs cost includes full pipeline drain. cudaFreeAsync() merely marks the allocation as “available for reuse” in the mempool metadata (0.25 µs). Detection: Nsight Systems timeline shows repeated short cudaDeviceSynchronize calls interleaved with allocation API. Mitigation: Use cudaMallocAsync/cudaFreeAsync with memory pools, or pre-allocate all buffers at initialization and never free during execution.

Tier 2: Severe Degradation (5–100x penalty)

L2 sync barrier vs DSMEM (26x) — Using st.release.gpu + ld.acquire.gpu for intra-cluster synchronization costs 2,242 cyc because each release triggers MEMBAR.ALL.GPU (549 cyc: L1 drain → L2 ACK → L1 invalidate). The DSMEM barrier.cluster.relaxed stays on the SM2SM fabric’s dedicated virtual channel at only 86 cyc. Detection: NCU stall breakdown shows dominant stall_membar percentage in cluster kernels. Mitigation: For cluster-local communication, always use st.async + mbarrier.try_wait.parity (242 cyc total). Reserve L2 path for cross-cluster data reads only (where DSMEM is unavailable).

Kernel cold launch (20x) — First-ever launch: 108.7 µs (PTX→SASS JIT verification + module load + resource pool init). Subsequent unique functions: 10–12 µs (binary lookup + symbol resolve + context setup). Warm launch: 5.4 µs. Detection: First-request latency spike in serving systems; timeline shows single 100 µs+ gap at start. Mitigation: Execute all kernel functions with a dummy grid (1 block, 1 thread) during initialization warmup. Use CUDA Graphs to eliminate per-invocation dispatch overhead for repeated hot-path kernels.

Custom AllReduce on PCIe (14x) — NVLink-optimized AllReduce kernels (e.g., vLLM’s single-block approach) achieve only 2.4 GB/s on PCIe because: (1) st.global.sys triggers MEMBAR.ALL.SYS (~920 cyc/fence), (2) no topology awareness to avoid cross-NUMA serialization, (3) single-block execution bottlenecks on one warp’s memory bandwidth, not the full PCIe link. NCCL’s tree/ring DMA-based algorithm achieves 35.2 GB/s by avoiding all three issues. Detection: AllReduce bandwidth << 35 GB/s; high MEMBAR.ALL.SYS stall count. Mitigation: On PCIe-only systems, always use NCCL for collectives. For MoE scatter/gather, use raw cudaMemcpyPeerAsync (53 GB/s). Prefer pipeline parallelism with griddepcontrol (544 ns inter-stage).

BAR aperture exhaustion (7.8x — SILENT) — The most dangerous cliff due to its complete lack of error reporting. When cumulative P2P BAR1 mappings exceed the 256 MB hardware aperture, affected pairs silently fall back to host-memory-bounce routing (source GPU → CPU DRAM → destination GPU), yielding 7.18 GB/s. The sequential FIFO exhaustion pattern means later-enabled GPU pairs fail first. Detection: Benchmark actual P2P bandwidth for all enabled pairs; look for the characteristic 7.18 GB/s signature. Mitigation: Audit total P2P buffer allocations: max_per_pair = 256 MB / (num_peers) = 37 MB for 8-GPU. Enable only actually-needed P2P pairs. Use IPC handles with demand mapping/release for large transfers.

GDDR7 bank group conflict (6.3x) — GDDR7 uses 4-way bank group interleaving at 128B granularity. Strides that are 3× multiples of 128B (384, 768, 1536 bytes) map repeatedly to the same bank group because gcd(3, 4) = 1, serializing all accesses. Power-of-2 strides ≥ 512B distribute uniformly across all 4 bank groups. Detection: NCU reports DRAM throughput far below 1,150 GB/s at high occupancy; struct size is 384/768/1536 B. Mitigation: Pad data structures to power-of-2 sizes; avoid sizeof(T) being 3× multiples of 128B. Use 1,150 GB/s as the roofline DRAM BW reference.

Tier 3: Moderate Penalties (1–5x)

UVM first-touch without prefetch (4.9x) — Managed memory first-touch triggers GPU page faults: each 64 KB page requires interrupt → host driver → page allocation → PCIe DMA migration → page table update. At 256 MB working set, total fault cost is 4.0 ms vs 5.4 ms for native (672x speedup with prefetch vs first-touch). Mitigation: Always call cudaMemPrefetchAsync before kernel launch. For performance-critical code, prefer explicit cudaMalloc + cudaMemcpy.

Tail effect (up to ~2x at R=1) — SM_120 has 110 SMs. Grid sizes of N×110 + 1 leave one CTA occupying one SM while 109 SMs sit idle for the final wave. Formula: SM_waste = (1 - R/110) × 100% where R is the remainder CTA count. Mitigation: Always use grid = ceil(work_units / 110) * 110. For irregular workloads, employ Stream-K decomposition (measured 3.71x speedup at optimal K-slice). Minimum 220 blocks (2/SM) for >90% efficiency; 440 blocks (4/SM) for >97%.

Pageable memory (1.2–1.6x) — The CUDA driver stages pageable transfers through a pinned bounce buffer (pin → DMA → unpin per chunk). Peak pageable BW occurs at 16 MB (49.43 GB/s); larger transfers degrade due to TLB pressure. Mitigation: Use cudaMallocHost for all buffers involved in host-device transfers. WC memory provides zero benefit on Blackwell (DMA engine bypasses CPU cache regardless).

P2P cross-NUMA (23% per-pair, 1.3x) — At 4+ concurrent P2P ring pairs, traffic necessarily crosses the QPI/UPI inter-socket link, reducing per-pair bandwidth from ~48 to ~37 GB/s. The root complex serializes cross-domain transactions. Mitigation: Partition communication-intensive GPU groups within NUMA domains (4 GPUs per socket). Use topology-aware placement: adjacent pipeline stages or expert groups on intra-NUMA GPU pairs.

10.13.2 Optimization Priority Framework

Given the 4+ orders of magnitude range (1.3x to 15,000x), a triage-based priority system is essential:

  1. Priority 0 (ELIMINATE): Cross-PCIe CAS, synchronous cudaFree, BAR exhaustion — these produce catastrophic or silent failures. Any occurrence in production code is a bug.
  2. Priority 1 (REDESIGN): L2 sync in clusters, custom AllReduce on PCIe, cold launch — these require architectural redesign (different synchronization primitive, different parallelism strategy, warmup phase).
  3. Priority 2 (TUNE): GDDR7 bank conflicts, tail effect — addressable with data layout tuning or grid sizing without algorithmic changes.
  4. Priority 3 (BEST PRACTICE): UVM prefetch, pinned memory, NUMA-aware placement — standard CUDA best practices that should be default in any production code.

KI-10.1: PCIe Gen5 x16 Protocol Overhead Decomposition Reveals 11% Irreducible Gap

Empirical Evidence: H2D 56.89 GB/s (88.9%), D2H 56.54 GB/s (88.3%), bidirectional 101.5 GB/s (80.6%). Near-symmetric H2D/D2H within 0.6%. [source: S14/case_1,2,6,7,8]

Theoretical Analysis: The 11.1% gap decomposes into: 128b/130b encoding (1.5%), TLP headers (3.1%), DLLP flow control (1.2%), ECRC (0.6%), DMA scheduling (2.0%), driver staging (3.3%). Each component is intrinsic to the PCIe Gen5 protocol and cannot be reduced by software optimization.

Microarchitecture Speculation [Confidence: H]: The near-symmetric H2D/D2H performance is architecturally significant: PCIe Gen5 uses independent TX/RX lanes with separate credit pools, unlike shared-bus architectures (e.g., HBM) where direction switching incurs bus turnaround penalties. The 0.6% H2D advantage suggests the host DMA engine's read-ahead prefetcher is marginally more effective than the GPU's write-push path.

Implication for Optimization: 56.9 GB/s is the hard ceiling for single-direction PCIe transfers. Bidirectional transfers provide 1.79x (not 2x) due to credit pool contention. For workloads requiring >57 GB/s host-device bandwidth, use persistent GPU memory with asynchronous prefetch rather than streaming transfers.

KI-10.2: BAR Exhaustion at 256 MB Is a Silent Catastrophic Failure (7.8x Penalty)

Empirical Evidence: GPUs 0-2 retain 55.8 GB/s while GPUs 3-7 degrade to 7.18 GB/s when total P2P BAR mappings exceed 256 MB per GPU. No error, no warning -- silent performance cliff. [source: S15/case_5]

Theoretical Analysis: The 7.18 GB/s fallback matches host-memory-bounced transfer rate: source GPU writes to host DRAM at ~7 GB/s (limited by root complex arbitration), then destination GPU reads at ~7 GB/s. The sequential exhaustion pattern (GPU3 fails first at cross-NUMA) reveals FIFO BAR allocation across the P2P enable order.

Microarchitecture Speculation [Confidence: H]: PCIe BAR1 has a fixed 256 MB hardware aperture on GB202. This cannot be enlarged without silicon changes. The driver could potentially implement BAR virtualization (demand-paging BAR space) but this would add per-TLP address translation latency.

Implication for Optimization: Multi-GPU applications must audit total P2P buffer allocations and stay within 256 MB cumulative per GPU. For 8-GPU all-pairs P2P, this limits per-pair allocation to ~37 MB (256 MB / 7 peers). Exceeding this silently degrades to 11% of peak bandwidth. This is the most critical system-level failure mode to guard against on this platform.

KI-10.3: Tail Effect Formula -- (1 - R/110) x 100% SM Waste

Empirical Evidence: R=1: 98.2% waste. R=55: 33.3%. R=109: 0.5%. Perfect wave quantization confirmed across 27 grid sizes. Single full wave (110 blocks) achieves only 55.3% efficiency due to insufficient occupancy. [source: S21/case_68, S14/case_29]

Theoretical Analysis: The CTA scheduler assigns round-robin across 110 SMs with no work subdivision. A partial last wave leaves (110-R) SMs completely idle. The 55.3% efficiency at 110 blocks (1/SM) reflects 8 warps of occupancy per SM, well below the 16-32 warps needed for compute saturation (Chapter 7, KI-7.5).

Microarchitecture Speculation [Confidence: H]: The non-monotonic dip at 256 blocks (65.7% vs 93.2% at 220) directly confirms wave quantization: 256/110 = 2.33 waves creates a tail of 36 blocks wasting 67% of SMs in the last wave. This interacts with the occupancy model: minimum 220-440 blocks (2-4/SM) for >90% efficiency.

Implication for Optimization: Grid sizes must be multiples of 110 or close to it. Stream-K decomposition (3.71x speedup at optimal slice) eliminates tail effects entirely. For non-persistent kernels, the formula optimal_grid = ceil(work_units / 110) * 110 should be applied. A single extra CTA beyond a wave boundary nearly doubles execution time.

KI-10.4: DSMEM Wins 26x Over L2 for Synchronization Because It Bypasses MEMBAR.ALL.GPU

Empirical Evidence: Cluster barrier.relaxed = 86 cyc vs L2 acquire/release = 2,242 cyc. st.async fire-and-forget = 2 cyc vs st.release.gpu = 555 cyc. But raw DSMEM read is 4.5x slower than L2 cached read. [source: S17/case_26-38]

Theoretical Analysis: The 26x sync advantage comes from bypassing the 549-cyc MEMBAR.ALL.GPU that dominates all L2-based synchronization. DSMEM's cluster barrier uses the SM2SM fabric's dedicated VC_NONBLOCKING virtual channel, which provides direct point-to-point notification without requiring L1 drain, L2 partition acknowledgment, or L1 invalidation. The barrier.cluster.relaxed is O(1) for 2-8 CTAs. 95% of DSMEM overhead is the barrier itself (85.7 cyc), with mapa resolution at 0 cyc (compile-time) and pipelined reads at 2.5 cyc/op.

Microarchitecture Speculation [Confidence: H]: The SM2SM fabric and GXBAR are physically separate interconnects serving different roles. DSMEM's advantage is structural -- it simply does not traverse the L2 coherence protocol. Patent US20230315655A1 describes "0.5 RTT" composite write+arrive that eliminates write-completion acknowledgment.

Implication for Optimization: Use DSMEM exclusively for producer-consumer tile handoff within clusters. Use L2 for unsynchronized cross-SM reads. Never use L2 acquire/release for cluster-local synchronization -- it is 26x more expensive than the dedicated barrier path. The st.async + mbarrier pattern (242 cyc total) should be the default for all cluster-based data redistribution.

KI-10.5: GDDR7 4-Way Bank Group Interleaving Creates Unique Stride Sensitivities

Empirical Evidence: Strides 384B, 768B, 1536B (all 3x multiples) produce 258-310 GB/s -- 6.3x worse than stride 1024B (1,633 GB/s). Power-of-2 strides are naturally conflict-free (2 and 3 are coprime). Latency is stride-invariant at ~843 cyc. [source: S21/case_74]

Theoretical Analysis: GDDR7 uses 4-way bank group interleaving at 128B granularity [S24/case_10] (see §10.6 / KI-10.5 above). The 4-way structure means strides that are 3× multiples of 128B (384, 768, 1536) hit the same bank group repeatedly because gcd(3, 4) = 1, serializing what would otherwise be parallel bank accesses; conversely, multiples of 4×128B = 512B distribute uniformly across all 4 bank groups. Earlier draft text describing this as "3-way bank group" was a transcription error — superseded by [S24/case_10] fine-grained probe; OQ-2 (Ch16) confirms 4-way as canonical.

Microarchitecture Speculation [Confidence: H]: The sustained BW of 1,150 GB/s (85.6% of theoretical 1,344 GB/s) at stride-16 includes row buffer hits that compensate for partial bank utilization. Full bank saturation at stride 1024B gives 1,633 GB/s -- higher than sustained BW because the L2 cache absorbs some re-reads in the steady-state measurement. The P99/P50 refresh ratio of 2.49x corresponds to ~5% duty cycle, consistent with GDDR7 specifications.

Implication for Optimization: Avoid strides that are multiples of 384B, 768B, or 1536B in memory access patterns. Power-of-2 strides are safe. For roofline models, use 1,150 GB/s as the sustained DRAM bandwidth reference. The 0.7% variation across 0.87-100ms confirms no thermal or power management interference with memory throughput.

Theoretical Analysis: The 768 ns advantage represents the eliminated host round-trip: ~400 ns for host CPU to detect kernel completion via MMIO polling + ~370 ns for host to submit next launch via doorbell register. griddepcontrol's GRIDDEPCONTROL.LAUNCH_DEPENDENTS instruction signals the GPU command processor directly, staying entirely on-chip.

Microarchitecture Speculation [Confidence: H]: The PREEXIT/ACQBULK SASS opcodes implement a hardware notification channel between the last-exiting warp and the command processor's scheduling queue. This bypasses both the host CPU and the GPU's interrupt/completion-reporting path.

Implication for Optimization: For pipeline-parallel inference (multiple stages across kernels), griddepcontrol reduces inter-stage latency by 1.23-2.4x. Combined with the 544 ns gap (vs 1,696 ns median for cudaStreamWaitEvent), this enables efficient pipeline parallelism with sub-microsecond inter-kernel coupling. This is the recommended inter-kernel dependency mechanism for all new SM_120 production code.

KI-10.7: PCIe-Only Topology Fundamentally Shapes Multi-GPU Strategy

Empirical Evidence: NCCL AllReduce: 35.2 GB/s (PCIe-bounded). Custom P2P AllReduce: 2.4 GB/s (14x worse -- NVLink-optimized design fails on PCIe). MoE AllToAll via raw P2P: 53 GB/s (1.3-2.1x better than NCCL AllToAll). Cross-NUMA penalty: 24% (41 vs 55.8 GB/s). 3->4 concurrent pair cliff at 23% per-pair degradation. [source: S17/case_14-16, S15/case_1,12]

Theoretical Analysis: On NVLink systems (H100), the AllReduce bottleneck is ~900 GB/s NVLink aggregate BW. On SM_120's PCIe-only topology, AllReduce is fundamentally limited by ~56 GB/s per GPU link -- a 16x lower ceiling. This inverts the optimal parallelism strategy: tensor parallelism (AllReduce-heavy) is viable on NVLink but wasteful on PCIe; pipeline parallelism (minimal cross-GPU communication) and expert parallelism (point-to-point scatter/gather) are optimal for PCIe.

Microarchitecture Speculation [Confidence: H]: The 2x4 NUMA topology creates a natural partition boundary: intra-NUMA communication (55.8 GB/s, 95% aggregate efficiency) is 36% faster than cross-NUMA (41 GB/s). For 8-GPU inference, the optimal strategy places pipeline stages or expert groups within NUMA domains to minimize cross-NUMA traffic.

Implication for Optimization: For LLM inference on 8x SM_120: (1) Prefer pipeline parallelism with griddepcontrol (544 ns inter-stage) or expert parallelism with raw P2P memcpy (53 GB/s). (2) Never implement custom AllReduce over PCIe -- use NCCL exclusively. (3) Partition work to minimize cross-NUMA communication. (4) Monitor BAR aperture usage to prevent the 7.8x silent degradation cliff.

10.15 Methodology Notes

  • Host-side timing: cudaEventRecord/cudaEventElapsedTime for GPU-inclusive measurements; std::chrono::high_resolution_clock for host-side-only latencies (cudaMalloc/cudaFree). Minimum of N iterations reported (N=20-10000 depending on measurement target).
  • GPU-side timing: clock64() and inline PTX %globaltimer for kernel-side measurements. Single-warp (32 threads) per CTA for clean timing in inter-SM cases.
  • Transfer sizes: Sweep 4B to 1GB (powers of 2 + intermediate sizes) for bandwidth; 4B-1KB for latency floor.
  • Clock pinning: nvidia-smi -lgc on all GPUs to suppress DVFS. GPU clock locked at 2100-2497 MHz depending on test section.
  • P2P: cudaDeviceEnablePeerAccess for all 56 pairs. Minimum 100-200 iterations after 20-50 warmup.
  • Section 15 Case 9 retest: Fork-before-CUDA design for statistically rigorous IPC handle measurements (1000+ iterations per configuration).
  • Section 17: 200-2000 measured iterations per test point. Topology invariance confirmed across Cases 1, 2, 3, 10, 20, 31 with <4% variance.
  • NCCL: Standard NCCL as shipped with the CUDA toolkit; tree/ring algorithms selected automatically.
  • GDDR7 tests: 128 MB+ working sets to ensure L2 miss. 10,000-sample distributions for refresh interference. SASS verified for correct store qualifiers.
