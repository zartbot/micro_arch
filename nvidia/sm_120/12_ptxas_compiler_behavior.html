ptxas Compiler Behavior

The ptxas assembler is the final optimization pass in the CUDA compilation pipeline, transforming PTX intermediate representation into machine-level SASS instructions with control codes, scoreboard assignments, and register allocation. Because ptxas is a closed-source black box, its behavior directly affects both measurement trustworthiness and optimization headroom. This chapter documents all empirically observed ptxas behaviors on SM_120 (GB202, RTX PRO 5000 72GB), organized into five categories: register allocation, instruction scheduling, known optimizations and side-effects, missed optimizations, and implications for measurement validity. Every finding is backed by CuAssembler binary patching experiments and cuobjdump SASS verification.

12.1 Register Allocation

12.1.1 The 24-Register Floor

ptxas enforces an absolute minimum of 24 registers per thread on SM_120. Any maxrregcount value below 24 triggers the ptxas warning: 

For profile sm_120a adjusting per thread register count of N to lower bound of 24

The floor affects the limit, not the actual allocation. Kernels that naturally require fewer registers (e.g., 8 or 16) continue to use their natural allocation regardless of the maxrregcount setting.

maxrregcount ptxas Warning Actual Regs (trivial) Actual Regs (moderate)
8 YES 8 16
16 YES 8 16
20 YES 8 16
24 No 8 16
32 No 8 16
48 No 8 16

[source: S21/case_111]

12.1.2 Restructuring Preference Over Spilling

When maxrregcount is set low (e.g., 24) but the kernel naturally needs more registers, ptxas exceeds the requested limit rather than introducing spill code:

  • Kernel compiled with maxrregcount=24 actually uses 28-30 registers
  • Zero spill instructions (STL/LDL) generated
  • Zero performance degradation compared to unconstrained compilation

This demonstrates that ptxas treats maxrregcount as a soft target for occupancy guidance, not a hard constraint. The compiler prefers register restructuring (rematerialization, recomputation) over spilling when the excess is modest.

[source: S21/case_112]

12.1.3 Spill Onset and Cost

When register pressure genuinely exceeds hardware capacity (e.g., 32 accumulator registers with maxreg=24), ptxas is forced to spill:

Configuration Total Cycles Spill Ops Slowdown Spill Cost
10-FMA (no spill) 35 0 1.0x --
32-FMA + maxreg=24 (forced spill) 183 233 STL + 231 LDL 5.23x 0.32 cyc/op
40-FMA (natural, unconstrained) 74 0 1.0x --

Key observations: - Spill operations use scalar STL and LDL.LU (not vectorized STL.128) - Per-spill-operation cost: 0.32-0.43 cyc (amortized, fire-and-forget for stores) - Local memory frame allocation adds negligible overhead (linear FMA scaling preserved) - The 5.23x slowdown at 32 accumulators with maxreg=24 is catastrophic -- avoid at all costs

[source: S21/case_113-115]

12.1.4 launch_bounds Impact

__launch_bounds__(maxThreadsPerBlock, minBlocksPerSM) has zero performance impact when the kernel's natural register usage already fits within occupancy constraints. The annotation serves only as a hint to ptxas for cases where register pressure is borderline.

  • Measured delta with and without launch_bounds(256, 2): 0%
  • ptxas uses launch_bounds to relax or tighten the register target (40-56 reg range observed)
  • Spill onset when maxreg<=36 with complex kernels: up to 7.75x slowdown

[source: S11/case_124, S21/case_10]

12.2 Instruction Scheduling

12.2.1 Stall Counter Semantics

The stall counter field (bits [44:41] of the 128-bit SASS instruction word) controls the minimum number of cycles between consecutive instruction issues for the same warp. SM_120 uses three special sentinel values:

Stall Value Behavior Measured Cost (cyc/FFMA) vs Optimal
0 HW sentinel: forces full replay 31.10 7.15x (22x replay)
1 (ptxas default for FFMA) Optimal for dependent chains 1.45 1.0x (baseline)
2 Direct interval control 2.05 1.41x
4 (ptxas default for loops) Direct interval control 4.05 2.79x
8 Direct interval control 8.05 5.55x
11 (ptxas default for HMMA) Suboptimal for MMA chains 34.34 --
12 (CuAssembler optimal) Optimal for MMA chains 32.23 6.1% better than 11
13 Same as 12 for MMA 32.23 6.1% better than 11
14+ HW sentinel: dep resolution 55.51 ~1.7x vs stall=12
15 HW sentinel: full dep replay 32.05 23x vs stall=1

Critical findings:

  1. stall=0 and stall=15 are hardware sentinels that trigger full dependency resolution replay. Both produce identical timing (641 cyc for 20 FFMAs), indicating they bypass the normal scheduling mechanism.

  2. Stall values 1-13 provide direct interval control. The relationship is exactly linear: total_cycles = stall * N_instructions + constant. Each stall unit adds exactly 1 cycle of minimum inter-issue delay.

  3. stall>=14 triggers a different hardware mechanism (possibly scoreboard-based dependency resolution), producing 55.51 cyc/HMMA -- significantly worse than the optimal stall=12.

[source: S21/case_9,16,17,82]

FFMA chains (compute-bound) 1 1 0% S21/case_9

12.2.2a Microarchitectural Analysis: The Stall Counter Hardware Model

The stall counter sweep data (Cases 9, 16, 17, 82) constrains a precise model of the SM_120 dispatch hardware. The 4-bit stall counter field (bits [44:41] of the 128-bit SASS instruction word) does not operate as a uniform "delay N cycles" knob. Instead, the hardware enters three completely different operating modes depending on the encoded value — a critical distinction that any CuAssembler-based optimization must respect:

Mode A — Linear Interval Control (stall ∈ [1..13])

In this mode, each stall unit corresponds to exactly one cycle of inter-issue delay for the same warp. The relationship is perfectly linear: total_cycles = stall × N_instructions + constant, where the constant (+1-2 cyc) represents clock64 measurement overhead. This is confirmed by Case 17's sweep across 5 FFMAs:

Stall Value Total (5 FFMAs) Per-FFMA Derivation
1 7 cyc 1.40 5×1 + 2 (overhead)
2 11 cyc 2.20 5×2 + 1
4 21 cyc 4.20 5×4 + 1
8 41 cyc 8.20 5×8 + 1

ptxas is hardware-optimal throughout this range for ALU instructions: FFMA dependency chains use stall=1, tight loops use stall=4. No CuAssembler-based improvement is possible here.

Mode B — Hardware Dependency Resolution Sentinel (stall ∈ {0, 14, 15})

These three values are hardware sentinels that bypass the linear dispatch-interval mechanism entirely and invoke a full hardware scoreboard dependency resolver. The resolver walks the complete dependency chain before allowing the next instruction to issue — producing catastrophic slowdowns:

Instruction Stall Value cyc/op Slowdown vs Optimal Source
FFMA 0 31.10 22× (vs stall=1) Case 9
FFMA 15 32.20 23× (vs stall=1) Case 17
HMMA 14 55.51 72% (vs stall=12) Case 82
HMMA 15 55.51 same as 14 Case 82

ptxas never emits stall values 0, 14, or 15 — these are its "forbidden zone." SASS-level patchers must exercise extreme caution: accidentally landing on a sentinel value produces silent catastrophic slowdown with no error or exception, just dramatically slower code.

Mode C — Pipeline-Aligned Sweet Spot (stall=12 for HMMA)

The most architecturally interesting finding emerges from the HMMA stall sweep (Case 82, 16-deep dependency chain). The results are counter-intuitive: stall=12 is faster than stall=11 (the ptxas default), even though it imposes a longer inter-issue delay:

Stall cyc/HMMA Interpretation
8 32.30 HW scoreboard auto-resolves (stall too short → falls into Mode B behavior)
9 32.32 Same as stall=8
10 33.33 Slightly slower — partial misalignment begins
11 34.34 ptxas default — SUBOPTIMAL (1-cyc writeback misalignment)
12 32.23 ★ OPTIMAL — aligns with TC accumulator writeback slot
13 32.23 Same as stall=12 (within alignment window)
14 55.51 Sentinel triggered — 72% penalty

The 6.1% HMMA optimization opportunity explained: The stall counter only controls the SMSP-level dispatch interval — it does not directly regulate the Tensor Core pipeline's internal accumulator writeback timing. The TC pipeline has a 29-cycle latency with a pipeline depth of ~1.81 slots. At stall=11, the next HMMA instruction issues 11 cycles after the previous one dispatches, but the TC accumulator writeback is not yet ready — producing a 1-cycle partial replay bubble as the hardware stalls waiting for the accumulator register to become available. At stall=12, the dispatch aligns precisely with the TC writeback slot, eliminating the bubble.

At stall=8-9, the stall is so short that it falls below the TC pipeline's natural throughput threshold — the hardware scoreboard detects the premature issue and automatically resolves the dependency via its full-replay path, accidentally achieving the same throughput as the aligned stall=12. This creates the non-monotonic U-shaped performance curve visible in the chart above.

ptxas's internal latency model is calibrated conservatively to stall=11 (the "safe" side of the alignment boundary). CuAssembler binary patching from stall=11 to stall=12 immediately recovers the 6.1% throughput gap. This improvement is consistent across all MMA families (HMMA, QMMA, OMMA.SF), confirming that ptxas uses a single stall heuristic for all Tensor Core instructions that is calibrated exactly one unit below the hardware optimum.

[source: S21/case_9,16,17,82]

12.2.3 Instruction Reorder Distance

ptxas reorders instructions within a bounded window to improve scheduling:

Metric Value
Maximum reorder distance 20 positions
Mean reorder distance 10.4 positions
Minimum reorder distance 0 positions
Median reorder distance ~11 positions

The reordering exhibits a two-wave pattern driven by register availability: ptxas processes instructions in alternating waves (even-indexed first, odd-indexed second), creating a characteristic interleaved scheduling output. This pattern maximizes the dependency distance between operations that share register operands.

Impact on measurement: Dep-chain measurements may have +/-1-2 cyc uncertainty from ptxas reordering within the 20-instruction window. Measurements using inline PTX asm with explicit output dependencies are immune to this effect.

[source: S21/case_11]

12.2.4 DEPBAR Insertion Heuristic

ptxas inserts zero DEPBAR instructions for regular memory loads on SM_120, regardless of the number of outstanding loads (tested 1-16 loads). Instead, ptxas relies exclusively on inline scoreboard control-code bits (bits [57:54] of the instruction word) for load dependency tracking.

DEPBAR is reserved exclusively for cp.async operations, where the asynchronous nature requires explicit barrier-based completion tracking that cannot be expressed through inline scoreboard bits alone.

Outstanding Loads DEPBAR Count Mechanism Used
1 0 Inline SB bits
4 0 Inline SB bits
8 0 Inline SB bits
16 0 Inline SB bits
cp.async (any count) >=1 DEPBAR

[source: S21/case_12]

12.3 Known ptxas Optimizations and Side-Effects

12.3.1 Constant Folding

ptxas performs aggressive constant folding that can eliminate entire instruction chains when operands are compile-time visible:

Pattern ptxas Behavior Evidence
add.s32 %0, %0, 1 chain (N=128) Folds to single IADD3 R, R, 128 Only 24 IADD3 in SASS vs expected 256+
Integer address arithmetic (constant offsets) Eliminates redundant IMAD/IADD3 Tested at -O3 vs -O0
FP32 immediate constants (0.5f, 1.0f) Folds MOV+FFMA into single FFMA with immediate slot MOV produces 0 cyc overhead
HFMA2 constant materialization Zero overhead (fully hidden) 4.00 cyc/FMA with/without constants
Polynomial chain constants Zero overhead; 28.0 cyc/7-FMA regardless of constant source S21/case_91

Critical implication for benchmarking: Any microbenchmark using compile-time visible integer arithmetic as a "workload" will be constant-folded to a trivial computation. Use volatile variables or global memory loads to prevent this optimization.

[source: S21/case_13, S21/case_87-91]

12.3.2 Loop Full Unrolling

ptxas fully unrolls ALL constant-trip-count loops on SM_120, with no upper threshold found up to 8192 iterations:

Trip Count Unrolled? Speedup vs Non-Unrolled
1 Yes 1.24x
64 Yes 6.45x
512 Yes 12.22x
2048 Yes 13.67x
8192 Yes 14.05x

Key details: - No threshold found: ptxas unrolls up to at least 8192 iterations with a single-instruction body - nvcc frontend also unrolls: #pragma unroll is redundant -- nvcc auto-unrolls regardless - PTX-level loops also unrolled: Even explicit bra back-edges in PTX are eliminated by ptxas - Only volatile/runtime trip counts prevent unrolling: To force loop behavior, use volatile loop counters - Loop overhead eliminated: ~53 cyc/iteration (IADD + ISETP + BRA + pipeline stalls) removed by unrolling

[source: S21/case_14]

12.3.3 Dead-Code Elimination (DCE)

ptxas performs its own DCE pass, independent of nvcc's C++ optimizations:

Anti-DCE Method Status Mechanism
Volatile global store SAFE ptxas preserves side-effect
Non-volatile global store (out[tid] = val) SAFE ptxas preserves store
printf conditional sink SAFE ptxas preserves call dependency
asm volatile("" :: "f"(val) : "memory") DCE'd ptxas eliminates (no visible side-effect)

Critical finding: The common pattern asm volatile("" :: "f"(val) : "memory") with only register clobbers is NOT sufficient to prevent ptxas DCE on SM_120. ptxas eliminates the entire dependent computation chain because no visible memory side-effect is produced.

Recommended safe pattern: Always store results to global memory (out[threadIdx.x] = val) or use a sink function that ptxas cannot prove dead.

[source: S21/case_15]

12.3.4 .reuse Bit Behavior

The .reuse bits are encoded at bits [60:58] of the upper 64-bit control word (= bits [124:122] of the full 128-bit instruction). They are functionally active on SM_120 with significant throughput impact in ILP-rich workloads: FFMA ILP=8 (stall=1) shows +85% penalty when .reuse is cleared. In latency-bound dep chains (stall≥4), .reuse has 0% impact.

Workload Type .reuse ON .reuse OFF Delta Source
FFMA ILP=8 (stall=1) 1.070 cyc/FFMA 1.984 cyc/FFMA +85% (1.85×) SM_120 server1
FFMA dep chain (stall=4) 4.00 cyc/FFMA 4.00 cyc/FFMA 0% S11/case_149

The RFC (Register File Cache) provides operand caching in throughput-bound regions where multiple ALU instructions issue back-to-back (stall=1). ptxas sets .reuse bits correctly for ILP-rich ALU code. In latency-bound dep chains (stall≥4), .reuse provides no benefit. For TC instructions (HMMA/QMMA/OMMA.SF), ptxas never sets .reuse bits — the TC pipeline uses an independent operand fetch path that does not rely on RFC. Forcing .reuse=111 on HMMA produces 0% delta, confirming TC .reuse is a true no-op.

[source: S11/case_155, SM_120 server1 ILP=8 verification]

12.3.4a Microarchitectural Analysis: ptxas as an SM_120-Aware Scheduler

Viewing ptxas as a "generic compiler pipeline" (constant folding / DCE / register allocation / scheduling / loop unrolling) sells the tool short — those are table stakes that every compiler since the LLVM era performs. The real value of ptxas on SM_120 lies in being a microarchitecture-aware scheduler: for every SASS instruction it emits, ptxas must populate five control-code fields whose semantics are tightly coupled to specific SM_120 hardware mechanisms; for several PTX operations it must additionally make SM_120-specific lowering choices (REDG vs ATOMG, UFFMA promotion, descriptor-mode addressing, and so on).

The four diagrams below cover the depth this chapter actually wants to convey.

(i) The Five SASS Control-Code Fields ptxas Sets per Instruction

Each 128-bit SASS instruction reserves five distinct control fields in its upper 64-bit word. Each field maps to a separate SM_120 hardware mechanism, and any field that is mis-set causes a silent performance loss — no error, no exception, just slower code.

Of these five fields, ptxas already does four of them well (see the "Key takeaway" panel at the bottom of the figure): parity-aware register allocation always avoids same-parity 3-source patterns, the SB wait mask correctly synchronizes with the 6-counter scoreboard, the yield bit defaults to 1 to optimize warp interleaving, and predicate dependencies are routed through a dedicated hardware resolver. The single systematic miscalibration is the stall counter on HMMA chains (11 → 12, detailed in the next subsection). The .reuse field deserves a special note — it is a dual-natured field: on ALU pipes, ptxas actively sets it and the hardware honors it (+85% throughput in ILP-rich workloads); on TC pipes, ptxas never sets it and the hardware ignores any value (true no-op). Earlier patchers that targeted the wrong bits [48:46] entirely missed this distinction and concluded ".reuse is universally inert" — a cautionary tale for SASS-level reverse engineering.

(ii) The Three Hardware Modes of the Stall Counter

The stall counter is 4 bits wide (0-15), but it is not a uniform "delay N cycles" knob. The hardware enters three completely different modes depending on the value:

Mode A (Linear, stall ∈ [1..13]) is the conventional dispatch-interval control: each stall unit corresponds to exactly one cycle of inter-issue delay. ptxas is at hardware optimum throughout this range for ALU pipes — FFMA chains use stall=1, tight loops use stall=4, and the Case 17 sweep confirms total cycles follow the strict linear model stall × N + 1.

Mode B (Sentinel, stall ∈ {0, 14, 15}) triggers the hardware dependency resolver's full-replay path, bypassing the linear interval mechanism entirely. stall=0 makes FFMA crawl at 31 cyc/op (22× slowdown); HMMA stall=14 measures 55.51 cyc/op. These values are ptxas's forbidden zone — it never produces them. SASS-level patchers must be careful here: accidentally landing on a sentinel value silently triggers catastrophic slowdown.

Mode C (Pipeline-Aligned) is the most interesting finding. The HMMA stall sweep shows a counter-intuitive result: stall=11 (ptxas default) takes 34.34 cyc, while stall=12 is faster at 32.23 cyc. The cause is that the stall counter only governs the SMSP dispatch interval — it does not directly control the TC pipeline's internal re-issue logic. stall=11 mis-aligns with the TC writeback's 1-cycle granularity, producing a 1-cyc partial-replay bubble; stall=12 aligns with the writeback slot. ptxas's internal latency model does not capture this granularity. CuAssembler patching from stall=11 to stall=12 immediately recovers 6.1% throughput — this is the only SASS-level optimization headroom on the entire SM_120 microarchitecture.

(iii) Operand Delivery: 3-Tier Model with ALU/TC Bifurcation

SM_120's register-operand path operates as a three-tier hierarchy. ptxas actively selects the tier — by combining register parity (which controls RF bank routing) with .reuse bits (which control RFC caching) — rather than relying on hardware fallback:

  • Tier 1 (RFC Hit, 1.34 cyc/FFMA) — .reuse bits set and the RFC cache hits. Operands are read directly from the cache, completely bypassing the RF read ports. This is the default path ptxas takes for ILP-rich ALU code, delivering +85% over Tier 2.
  • Tier 2 (RF Mixed Parity, 2.04 cyc/FFMA) — three source operands are split across even/odd banks, so the two RF read ports can serve all three reads in a single OC cycle. This is the baseline.
  • Tier 3 (RF Same Parity, 3.03 cyc/FFMA) — three sources are all even (or all odd), forcing ⌈3/2⌉ = 2 OC cycles to serialize the reads. ptxas never produces this pattern; observing it requires CuAssembler binary patching to force same-parity registers.

The lower half of the figure highlights how ALU and TC pipes interpret the same .reuse field differently. ALU pipes consult the RFC and .reuse is functionally active. TC pipes have their own operand fetch network and do not query the RFC at all. This is why all 456 HMMA instructions in the test cubin have .reuse bits = 000, and why forcing them to 111 produces zero performance delta.

(iv) Optimization Headroom Map: 4 Bottlenecks ptxas Cannot Auto-Resolve

After all the microarchitecture-aware decisions are accounted for, ptxas is near-optimal on SM_120. The remaining headroom concentrates in four specific scenarios:

These four gaps split into two categories by intervention level:

  • SASS-level patch class — ① HMMA stall calibration. No kernel source changes required; CuAssembler binary patching from stall=11 to stall=12 recovers +6.1% throughput. This is the only headroom that pure post-processing can capture.
  • Kernel-design class — ② missing auto-vectorization (developers must explicitly use float4 / int4); ③ non-vectorized spills (must be avoided via tile / launch_bounds tuning rather than fixed at the spill site); ④ helper-function overhead (replace __cuda_sm20_* calls with inline PTX). These are not bugs in ptxas — they are deliberate design boundaries: ptxas does not rewrite the instruction selection that the developer explicitly wrote.

Overall, ptxas is a deeply microarchitecture-aware code generator on SM_120: it knows the RF has 2 banks, that the OC tolerates cross-parity reads, that the RFC has ≥24 entries, that the 6 SB counters hit a cliff at N=6, that TC pipes do not participate in the RFC, and that REDG and ATOMG are two physically distinct hardware paths. This level of SM_120-specific awareness is far beyond what any "generic compiler optimization pipeline" can express — and it is the lens developers should bring when reading the SASS that ptxas emits.

12.3.5 nvcc -O0 Pass-Through

The nvcc -O0/-O3 flag controls only the host C++ compiler (cicc/cudafe). The PTX-to-SASS compilation by ptxas runs at its own default optimization level regardless of nvcc's flag:

Compilation Mode Effect on Inline Asm Clock Overhead
nvcc -O3 (default) Baseline 1 cyc
nvcc -O0 Zero effect (bit-identical SASS) 1 cyc
nvcc -O3 + --ptxas-options=-O0 8.6-31.3x inflation 102 cyc

To disable ptxas optimization, the explicit --ptxas-options=-O0 flag must be used. This inflates measured latencies by: - ALU operations: 8.6x (4 cyc -> 34 cyc) - SFU operations: 12.7-31.3x (18-44 cyc -> 559 cyc) - SEL (SETP+SELP): 6.8x (10 cyc -> 68 cyc)

[source: S21/case_13]

12.3.6 Uniform Datapath Promotion (UFFMA/UIADD3)

ptxas promotes operations on uniform (warp-invariant) data to the uniform datapath, using UFFMA and UIADD3 instructions. These uniform operations dispatch to a dedicated execution unit with free (zero-cost) issue, not competing with the main FP32/INT pipe.

  • Detected via SASS: UFFMA, UIADD3, UIMAD prefixed instructions
  • Uniform registers: UR0-UR63 (distinct from per-thread R0-R255)
  • Transfer cost: R2UR optimized away by ptxas; REDUX-UR-R = 45 cyc when explicit transfer is needed

[source: S11/case_85, S21/case_120]

12.4 Missed Optimizations (ptxas Limitations)

12.4.1 No Auto-Vectorization

ptxas does NOT merge contiguous scalar loads into wider vector loads, even when the pattern is trivially recognizable: 

  Source pattern: 8 consecutive float loads from base+0 to base+28 (32B total)

  ptxas output (8 scalar loads):
    LDG.E.CONSTANT R0, desc[UR4][R2.64]
    LDG.E.CONSTANT R7, desc[UR4][R2.64+0x4]
    LDG.E.CONSTANT R6, desc[UR4][R2.64+0x8]
    ...
    LDG.E.CONSTANT R12, desc[UR4][R2.64+0x1c]

  Optimal output (2 vector loads):
    LDG.E.128.CONSTANT R12, desc[UR4][R2.64]
    LDG.E.128.CONSTANT R4, desc[UR4][R2.64+0x10]

Performance impact: Manual float4 vectorization provides 1.27-1.31x speedup for L2-resident data, and 1.02x for DRAM-bound workloads (where the memory controller already handles request coalescing).

Recommendation: Always use float4, int4, or __int128 types for contiguous memory access patterns. Do not rely on the compiler to vectorize scalar access patterns.

[source: S21/case_116-117]

12.4.2 Suboptimal MMA Chain Stall Assignment

ptxas assigns stall=11 to HMMA instructions in dependency chains, but the hardware-optimal value is stall=12-13:

Configuration Throughput (cyc/HMMA) Delta vs Optimal
ptxas default (stall=11) 34.34 -6.1%
CuAssembler stall=12 32.23 Optimal
CuAssembler stall=13 32.23 Optimal
CuAssembler stall>=14 55.51 HW sentinel (broken)

The 6.1% throughput loss is consistent across HMMA, QMMA, and OMMA.SF chains, suggesting ptxas uses a single stall heuristic for all MMA families that is calibrated slightly below the hardware optimum.

Recommendation for kernel developers: Use CuAssembler binary patching to set stall=12 on HMMA/QMMA/OMMA.SF instructions in performance-critical MMA chains. The 6.1% improvement is free (no code change required, only control-code patching).

[source: S21/case_82]

12.4.3 Integer Chain Constant Folding Masks Real SASS

ptxas constant-folds integer chains with compile-time visible operands, which can produce SASS that does not reflect the intended computation:

PTX Pattern Expected SASS Actual SASS
128x add.s32 %0, %0, 1 128 IADD3 Single IADD3 (add 128)
Address arithmetic (constant stride) Multiple IMAD Single IMAD with folded offset
Loop counter increment (constant trip) Per-iteration IADD3 Fully unrolled (no counter)

This makes it impossible to measure integer ALU latency using the naive pattern add.s32 %0, %0, 1 in a loop -- ptxas folds the entire chain. To measure true integer latency, use a runtime-variable addend loaded from global memory.

[source: S21/case_13]

12.4.4 No Spill Vectorization

When ptxas does spill registers to local memory, it uses scalar STL and LDL.LU instructions exclusively -- never vectorized STL.128 or LDL.128:

Spill Pattern Instructions Generated Theoretical Alternative
32 accumulators at maxreg=24 233 STL + 231 LDL.LU (scalar) 58 STL.128 + 58 LDL.128
Cost per spill operation 0.32-0.43 cyc (would be similar per 128-bit op)

While the per-operation cost is low (0.32 cyc), the 4x instruction count overhead compared to vectorized spills increases I-cache pressure and adds decode overhead for spill-heavy kernels.

[source: S21/case_113-114]

12.5 SASS Encoding and Control Code Structure

12.5.1 Control Code Field Layout

Each 128-bit SASS instruction on SM_120 contains a 17-bit control code field at bits [121:105] of the instruction word, which maps to the following layout in the upper 64-bit word:

Field Bits (upper word) Width Purpose
Stall count [44:41] 4 bits Minimum inter-issue delay (0-15 cycles)
Yield bit [45] 1 bit Warp scheduling yield hint
Reuse bits [60:58] 3 bits Operand reuse hint (functionally active on SM_120 for throughput-bound code)
SB wait mask [57:54] 4 bits Scoreboard dependency wait bitmask

[source: S21/case_110]

12.5.2 Yield Bit Effect

The yield bit (bit [45]) provides a scheduling hint to the warp scheduler:

Yield Value Effect Performance Delta
0 Do not yield after this instruction 2.4-4.7% slower
1 (ptxas default) Yield after this instruction Baseline (faster)

ptxas sets yield=1 as its default for most instructions. Setting yield=0 via CuAssembler causes 2.4-4.7% degradation that grows with warp count, suggesting the yield bit enables better warp interleaving at the scheduler level.

[source: S21/case_106]

12.5.3 Scoreboard Mechanics

Finding Value / Evidence Source
Co-consumed SB grouping saves 3 cyc (1 vs 4 cyc for L1-cached loads) S21/case_105
Multi-SB wait bitmask cost 0% delta vs single-SB wait S21/case_110
Fire-and-forget STG 4.3 cyc/store; no SB wait required S21/case_107
@!PT LDS RZ,[RZ] marker 0.00 cyc overhead; scheduling hint only S21/case_108
Address IMAD interleaving Each dep IMAD adds ~4-5 cyc/load (blocks hiding) S21/case_109

Key finding on co-consumed grouping: When multiple loads are consumed by the same instruction, grouping them under a single scoreboard bit (co-consumed) saves 3 cycles compared to assigning distinct SB bits. This occurs because a single SB wait resolves all loads simultaneously, while distinct SB bits require sequential resolution.

12.5.4 Predicate Dependencies Bypass Stall Counters

Hardware predicate resolution is independent of the stall counter field. All stall values 0-15 produce identical timing (37.00 cyc/iter) for predicate-consuming instructions:

  • Predicate dependencies resolve via dedicated hardware, not the stall-based scheduling
  • ptxas conservatively sets stall=13 before predicate consumers (costs nothing because HW handles it)
  • No stall-counter optimization is possible for predicate-dependent code paths

[source: S21/case_103]

12.6 Descriptor-Based Addressing

12.6.1 Descriptor Is the ONLY Addressing Mode

On SM_120, all ld.global and st.global PTX instructions are lowered to descriptor-based addressing (desc[UR][R.64] format), regardless of the source-level addressing pattern:

PTX Source SASS Output Delta
ld.global.f32 %f0, [%rd0] (pointer) LDG.E.CONSTANT desc[UR4][R2.64] 0 cyc
ld.global.f32 %f0, [%rd0] (TMA desc) LDG.E.CONSTANT desc[UR4][R2.64] 0 cyc
st.global.f32 [%rd0], %f0 (pointer) STG.E desc[UR4][R2.64] 0 cyc

There is zero performance delta between "pointer-based" and "descriptor-based" global memory access at the source level, because both lower to identical SASS. The descriptor register pair (UR + R.64) encodes the base address and offset.

[source: S21/case_118-119]

12.6.2 R2UR Transfer

Register-to-uniform-register transfers (R2UR) are optimized away by ptxas in most cases. When an explicit transfer is required (e.g., broadcasting a per-thread value to all threads), the mechanism is REDUX-UR-R with a measured latency of 45 cycles.

[source: S21/case_120]

12.7 Global Reduction (REDG) Native Support

ptxas lowers atomic reductions to native REDG instructions on SM_120, providing efficient fire-and-forget semantics:

12.7.1 Integer Variants

Operation SASS Instruction Cost (cyc) Semantics
atomicAdd (S32) REDG.E.ADD.STRONG.GPU.S32 2 Fire-and-forget
atomicAdd (U32) REDG.E.ADD.STRONG.GPU.U32 2 Fire-and-forget
atomicMin (S32) REDG.E.MIN.STRONG.GPU.S32 2 Fire-and-forget
atomicMin (U32) REDG.E.MIN.STRONG.GPU.U32 2 Fire-and-forget
atomicMax (S32) REDG.E.MAX.STRONG.GPU.S32 2 Fire-and-forget
atomicMax (U32) REDG.E.MAX.STRONG.GPU.U32 2 Fire-and-forget

All six integer REDG variants achieve identical 2 cyc fire-and-forget dispatch cost.

[source: S21/case_121]

12.7.2 FP16/BF16 Variants

Operation SASS Instruction Cost (cyc) Note
atomicAdd (FP16x2) REDG.E.ADD.F16x2 ~6 Native, no CAS fallback
atomicAdd (BF16x2) REDG.E.ADD.BF16x2 ~6 Native, no CAS fallback
ATOMG (dep chain) ATOMG.E.ADD.STRONG.GPU 409 L2 round-trip (compare-and-swap)

Critical correction: Section 7's finding that FP16 atomicAdd falls back to ATOM+CAS is corrected here. REDG provides the native FP16x2 and BF16x2 reduction path at ~6 cyc fire-and-forget. The CAS-based ATOMG path (409 cyc) is used only when the return value is consumed (dependency chain semantics required).

The distinction is: - REDG (fire-and-forget): Used when the old value is NOT needed. 2-6 cyc dispatch, no scoreboard wait. - ATOMG (read-modify-write): Used when the old value IS consumed. 409 cyc L2 round-trip.

[source: S21/case_121-123]

12.8 Immediate Encoding Capabilities

12.8.1 FFMA Immediate Slots

SM_120 FFMA supports immediate constants in both the multiplier and addend operand slots, contradicting some earlier architectural assumptions:

Operand Slot Encoding Opcode Cost Delta vs Register
Addend (c in a*b+c) 20-bit immediate 0x7423 0 cyc
Multiplier (b in a*b+c) 20-bit immediate 0x7823 0 cyc

ptxas exploits this by folding MOV immediate + FFMA into a single FFMA with the constant in the immediate slot. The MOV produces zero overhead -- it is not a separate instruction in the SASS output.

[source: S21/case_88-90]

12.8.2 Constant Materialization Strategy

For polynomial evaluation and other patterns requiring multiple constants:

Strategy Measured Cost Mechanism
HFMA2 constant materialization 0 cyc overhead Hidden behind dep-chain latency
FFMA immediate for 0.5f/1.0f 0 cyc Folded into instruction encoding
Preloaded register constants 0 cyc Same as inline immediate
Polynomial with 7 constants 28.0 cyc (4.0 cyc/FMA) Identical to pure register

All constant materialization strategies produce identical throughput (4.00 cyc/FMA), confirming that constant loading is fully hidden behind the FMA dependency chain on SM_120.

[source: S21/case_87-91]

12.9.2 Helper Function Cost

Compiler-generated helper functions (e.g., __cuda_sm20_rem_u16) incur significant overhead:

Operation Cost (cyc) vs Inline PTX
__cuda_sm20_rem_u16 helper call 174 2.16x slower
Inline PTX equivalent 80 Baseline
Integer division (general i32) 75 19x FP32 FMA
Integer modulo (general i32) 75 Same as division
Power-of-2 division 12 6.25x faster than general

Recommendation: Replace compiler-generated helper function calls with inline PTX for performance-critical paths. Avoid general integer division in hot loops (75 cyc = 19x FP32 FMA cost); use power-of-2 divisions where possible (12 cyc).

12.10 Impact on Measurement Trustworthiness

12.10.1 Residual Uncertainty Budget

Source of Uncertainty Magnitude Mitigation
ptxas reordering (max distance 20) +/-1-2 cyc Use dep-chain constraints in inline asm
Constant folding (integer chains) Complete elimination Use global-memory-loaded operands
DCE (asm clobber without store) Complete elimination Always use global store sink
Stall counter sensitivity +/-6.1% for MMA chains Accept or patch via CuAssembler
Clock overhead variation 1 cyc (O3) vs 102 cyc (ptxas-O0) Always compile with -O3

12.10.2 Recommended Measurement Practices

Based on the ptxas behaviors documented in this chapter:

  1. Anti-DCE: Always use out[threadIdx.x] = val (global store sink). Never rely on asm volatile with only register clobbers.

  2. Prevent constant folding: Load all operands from global memory. Use volatile loop counters for non-unrolled loops.

  3. Compiler flags: Always use -O3 (default ptxas optimization). Never use -O0 for timing measurements. The nvcc -O0/-O3 flag has zero effect on inline asm -- only --ptxas-options=-O0 matters.

  4. SASS verification: Always confirm instruction emission via cuobjdump --dump-sass before trusting timing results. ptxas may eliminate, fold, or restructure code in non-obvious ways.

  5. Stall sensitivity: For MMA chain measurements, be aware of the stall=11 suboptimality (6.1%). Report ptxas-default timing and note the CuAssembler-optimal bound where applicable.

12.10.3 Quantified Measurement Confidence

Given the above mitigations, the residual measurement uncertainty for this report's data is:

  • Cycle-precise measurements (dep-chain latency): +/-2 cyc maximum
  • Throughput measurements (multi-instruction bodies): +/-5% maximum
  • MMA chain throughput: +/-6.1% (ptxas stall suboptimality bound)

All measurements in this report use the recommended practices above, achieving the stated confidence bounds.

12.10.4 Microarchitectural Analysis: The Complete CuAssembler Optimization Surface

The comprehensive CuAssembler patching experiments across Sections 11, 12, and 21 establish the complete optimization surface available for SASS-level tuning on SM_120:

Why ptxas is near-optimal on SM_120: The combination of three architectural features makes SASS-level optimization uniquely low-reward on this architecture:

  1. Zero bank-conflict cost: The operand collector absorbs ALL RF bank conflicts (Chapter 3, §3.3.2), eliminating the historically largest optimization opportunity (5-10% on SM_80/SM_89)

  2. .reuse bits are functionally active: The RFC provides significant throughput benefit (+85%) in ILP-rich throughput-bound code (stall=1); ptxas correctly sets these bits in such regions

  3. Linear stall counter semantics: The 1:1 cycle-per-stall-count model leaves no room for non-obvious scheduling tricks — the optimal stall is simply the instruction's known latency

The only remaining optimization headroom comes from ptxas using stall=11 instead of the hardware-optimal stall=12 for MMA instructions — a single-value calibration error that CuAssembler can correct with zero risk of corruption.
