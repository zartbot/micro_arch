CHAPTER 06 · SM_120 · GB202 BLACKWELL

On-Chip Network & Async

This chapter characterizes the on-chip interconnect and asynchronous data movement subsystems of the SM_120 (GB202, RTX PRO 5000 72GB) microarchitecture. The SM_120 features a dual-fabric design: a dedicated SM-to-SM (SM2SM) interconnect supporting Distributed Shared Memory (DSMEM) with integrated cluster barriers, and a global crossbar (GXBAR) providing L2-coherent communication. The Tensor Memory Access (TMA) engine delivers descriptor-based multi-dimensional loads with deep pipelining (32+ in-flight), but its multicast capability is completely absent in hardware on this die -- a limitation with severe implications for cluster-based GEMM. Measurements spanning 117 TMA experiments, 24 cluster cases, and 38 inter-SM communication experiments establish the complete performance model for asynchronous data movement on SM_120.

6.1 Tensor Memory Access (TMA)

6.1.0 TMA Architectural Overview

The Tensor Memory Access (TMA) engine is a dedicated hardware accelerator tightly coupled to each SM on SM_120 (one TMAU per SM), first introduced in the Hopper architecture (SM_90) and significantly enhanced in Blackwell. Its architectural purpose is to offload the entire address-computation and data-movement burden from the SM's warp scheduler and register file, enabling fully asynchronous multi-dimensional data transfers between global memory and shared memory with zero register-file pressure.

The TMA design is motivated by a fundamental bottleneck in traditional GPU data movement: for workloads like GEMM and convolution, SM threads historically spent >10% of execution time on address computation alone ([US 2023/0289292 A1, §Background]), consuming hundreds of registers and thousands of instruction slots merely to calculate source and destination addresses for each cache line. The TMA eliminates this overhead entirely — the SM provides only a tensor descriptor pointer and tile coordinates in a single instruction (UTMALDG / cp.async.bulk.tensor in PTX), and the TMA engine hardware autonomously:

  1. Decodes the tensor descriptor — a 128-byte structure in global memory containing the tensor's base address, dimensionality (1D–5D), per-dimension sizes and strides, tile (box) dimensions, traversal strides, swizzle mode, and out-of-bounds fill policy.
  2. Traverses the multi-dimensional tile space — using a 5-level nested hardware loop that iterates over all elements within the requested tile, computing the global memory address for each element from coordinates × strides.
  3. Segments into L2 cache-line requests — each generated address is checked against tensor bounds (with automatic zero-fill for OOB elements), swizzled for bank-conflict-free shared memory layout, and packaged into cache-line-sized L2 requests.
  4. DMA-writes data directly to shared memory — bypassing the register file entirely. The data path is: Global Memory → L2 Cache → GXBAR → SM's shared memory SRAM.
  5. Signals completion via mbarrier — each request carries the address of an mbarrier structure in shared memory; upon final data arrival, the hardware atomically decrements the barrier's expected-count, enabling the SM to detect completion without polling or software tracking.

This five-stage hardware pipeline operates fully asynchronously — the SM's warp scheduler continues issuing compute instructions (FMA, HMMA, etc.) immediately after dispatching the TMA request, achieving 100% compute-data overlap as confirmed by co-issue measurements ([S9/case_29]: TMA + TC overlap = 100%). The TMA engine processes requests from its internal queue (≥32 entries deep on SM_120, a 4× increase over SM_90's ~8 entries), enabling deep software pipelining without complex multi-buffering logic.

The TMA employs a three-layer parameter hierarchy that minimizes per-request data transfer between the SM and TMA engine:

The tensormap.replace instruction (measured at 39 cycles total for all 4 fields, [S10/case_98-99]) enables dynamic descriptor updates for grouped GEMM without requiring full descriptor reconstruction.

6.1.0c TMA vs Traditional Data Movement: Architectural Trade-offs

The TMA's architectural contribution is not raw latency — a single TMA 1D load (479 cyc) is 16.7× slower than a cached ld.global (32 cyc) for hot L1-resident data. Instead, the TMA provides three complementary advantages that dominate at production-kernel scale:

  1. Zero register-file pressure: Traditional ld.global requires 32 registers per warp to hold a 1KB tile (32 threads × 32B/thread), occupying these registers for the entire memory latency (~350 cycles). The TMA bypasses the RF entirely, freeing those registers for computation and improving occupancy.
  2. Hardware-managed deep pipelining: With ≥32 queue entries, the TMA achieves 11× throughput improvement at saturation (1537 → 139 cyc/ld) without requiring any software loop management. The equivalent software double-buffering requires complex mbarrier phase tracking code that itself consumes instruction cache and scheduler bandwidth.
  3. Descriptor-based multi-dimensional addressing: For a 2D tile load, software would need ~20 instructions per row for stride computation, bounds checking, and address formation. For a 64×64 tile with 16B elements (64 rows), this is 1280+ ALU instructions eliminated — measured as the >10% performance loss cited in the TMA patent (US 2023/0289292 A1).

The practical crossover point where TMA becomes superior to manual loads is approximately 4KB of transfer per operation: below this threshold, the 479-cycle fixed TMA initiation cost dominates; above it, the linear 0.031 cyc/B transfer rate and pipelining benefits make TMA the clear winner. For production GEMM/attention kernels (16–64KB tiles), TMA provides 2–3× net efficiency improvement over manual cp.async-based pipelines.

The TMA engine on SM_120 provides hardware-accelerated descriptor-based data movement between global and shared memory, supporting 1D through 5D addressing, im2col mode, and atomic reductions. The following subsections present detailed measurement results across 117 TMA experiments covering latency, throughput, pipelining depth, store/reduction operations, multicast behavior, and additional features.

6.1.2a Microarchitectural Analysis: TMA Queue Saturation Model

The TMA pipelining curve follows a classic MSHR-like saturation model where effective throughput is bounded by min(queue_depth × issue_rate, service_rate):

The TMA engine operates as a decoupled two-phase pipeline with a deep hardware-managed descriptor FIFO separating the issue side from the service side. On the issue side, the SM dispatches UTMALDG instructions into the TMA engine at a measured rate of 148 cycles per operation ([S22/case_40]), representing the descriptor injection bandwidth. On the service side, each queued descriptor undergoes two sequential phases: a fixed-cost descriptor decode (~440 cycles) that computes the multi-dimensional source addresses, followed by a linear data transfer phase at 16.1 B/cycle DMA write rate (0.062 cyc/byte). The total per-tile service time is ~479 cycles for tiles up to 1KB.

The queue saturation model follows classic MSHR-style throughput scaling: at depth d, effective per-load cost is bounded by max(service_time / d, DMA_write_floor). The 62-cycle DMA floor (1KB ÷ 16.1 B/cyc) explains why measured throughput at d=32 (139 cyc/ld) significantly exceeds the naive prediction (479/32 ≈ 15 cyc) — the smem write port bandwidth, not queue depth, becomes the true bottleneck at high occupancy. Additional overhead from mbarrier signaling and residual descriptor decode serialization accounts for the remaining gap between the 62-cycle theoretical floor and the 139-cycle measured value.

The practical implication for GEMM kernel designers: the d=1→d=2 transition (double buffering) captures the single largest throughput gain (3.1×), making it the minimum viable pipelining strategy. Each subsequent doubling yields diminishing returns following a hyperbolic decay curve — d=2→d=4 adds 1.9×, d=4→d=8 adds 1.4×, d=16→d=32 adds only 1.12×. For production kernels, the 3-4 stage sweet spot (corresponding to d=4–8 in the saturation model) provides optimal shared-memory budget versus throughput trade-off. Beyond 4 stages, the marginal smem cost (one additional tile buffer) far exceeds the diminishing latency reduction.

Generationally, SM_120's ≥32-entry queue represents a 4× depth increase over SM_90 (Hopper, ~8 entries). This deeper hardware queue is architecturally necessary because SM_120 lacks Tensor Memory (TM) — all operands must transit through shared memory, increasing the latency that must be hidden. The deeper queue compensates by allowing more outstanding descriptors to overlap their service phases, achieving near-saturated DMA bandwidth without requiring complex software-managed multi-buffering beyond 3 stages.

Practical takeaway for GEMM designers: The 3.1× speedup from d=1 to d=2 captures the largest single improvement — implementing double buffering is the minimum viable pipelining strategy. Each subsequent doubling yields diminishing returns: d=4 adds 1.87×, d=8 adds 1.38×, d=16 adds 1.21×, d=32 adds 1.12×. For production GEMM, 3-4 pipeline stages provide optimal smem-budget-to-speedup trade-off.

The die-level cause: SM_120 (GB202) is a consumer/workstation die that lacks the multicast crossbar (LRC -- L2 Residency Controller with multicast tracking table) present in datacenter Blackwell dies (GB100/sm_100a, GB102/sm_110a). This is a physical silicon limitation.

6.1.5a Microarchitectural Analysis: The Multicast Emulation Mechanism

The multicast penalty measurements reveal two fundamentally different data-flow architectures between datacenter Blackwell dies (GB100/GB102) and the consumer GB202. Understanding this distinction is critical for any kernel porting effort from SM_100 to SM_120.

The hardware multicast path (SM_100/SM_110) operates as a single-read, multi-write transaction: the TMA engine issues one L2 cache read, and the LRC (L2 Residency Controller) — a dedicated hardware block at the L2 interface — maintains a multicast tracking table that maps each in-flight multicast descriptor to a set of destination SMs. As the L2 cache line is read, the LRC simultaneously initiates DMA writes to all target SMs' shared memory via the GXBAR crossbar, using per-SM write pointers and completion counters stored in the tracking table. The cost is effectively 1× TMA_latency (479 cycles for a 1KB tile) regardless of cluster size, achieving a memory bandwidth reduction of 1/K where K is the number of receiving CTAs.

The software emulation path (SM_120) is fundamentally different. Because the GB202 consumer die physically lacks the LRC multicast tracking table, ptxas cannot emit a hardware UTMALDG instruction for multicast requests. Instead, it lowers cp.async.bulk.tensor...multicast::cluster to a CALL.ABS.NOINC — a trap into a firmware routine that executes the following loop for each target CTA in the multicast mask:

  1. Step 1: TMA unicast load — issue a standard unicast UTMALDG to load the tile from L2/DRAM into the local SM's shared memory (~479 cyc for 1KB).
  2. Step 2: DSMEM byte-copy — copy the tile from local shared memory to the target CTA's shared memory via the SM2SM (DSMEM) fabric, one cache line at a time. At the measured DSMEM bandwidth of ~0.00023 B/cyc per byte, this dominates at ~4,348 cyc/byte.

The measured cost model (linear regression across 256B–4KB tiles, [S10/case_28], [S16/case_16]) decomposes cleanly into:

Component Cost Source
Firmware dispatch + setup ~97,000 cyc (base) Y-intercept of linear fit
DSMEM byte-copy (per byte) ~4,142 cyc/byte Slope of linear fit (99.5% of marginal cost)
TMA unicast (per tile) ~479 cyc/tile Fixed per-iteration overhead (0.5% of cost)
Total (1KB tile, mask=0x3) ~4,328,800 cyc (7,463× penalty) Measured end-to-end

The critical insight is that the DSMEM byte-copy dominates 99.5% of the per-byte cost. The per-byte marginal cost (~4,142 cyc/byte) matches the independently measured DSMEM bandwidth (~0.00023 B/cyc → 4,348 cyc/byte, within 5% measurement error), confirming the two-stage firmware model. The firmware cannot optimize this path because DSMEM has no bulk-copy primitive — each 16-byte st.async.shared::cluster incurs the full SM2SM fabric transit latency.

Die-level root cause analysis: The LRC on SM_100/SM_110 contains significant dedicated silicon: the multicast tracking table must maintain per-SM write pointers, completion counters, and buffer allocation state for each in-flight multicast (estimated 32+ simultaneous multicasts based on TMA queue depth). On the consumer GB202 die, NVIDIA chose to repurpose this die area for additional L2 cache capacity — the GB202's 96MB L2 is 50% larger than the B200's 64MB, strongly supporting the hypothesis that LRC multicast silicon was traded for L2 SRAM banks. This is a deliberate product segmentation decision: consumer/workstation workloads (rendering, inference) benefit more from L2 hit rate than from cluster GEMM bandwidth reduction.

Impact on GEMM design patterns: Without hardware multicast, the SM_100 "cluster GEMM" pattern — where one L2 read of the A-operand tile simultaneously serves all K CTAs in a cluster — is impossible on SM_120. Each CTA must independently issue its own TMA load, multiplying L2/DRAM memory traffic by the cluster size. For a cluster-4 GEMM at memory-bound problem sizes, this means 4× the L2 bandwidth consumption for the A operand versus SM_100. The recommended workaround is the DSMEM-based sharing pattern: one CTA performs the TMA unicast load, then distributes via st.async.shared::cluster (+220 cyc per cross-CTA read) — expensive, but still 5 orders of magnitude cheaper than the firmware-emulated multicast path (>100K cyc). In practice, CUTLASS, FlashInfer, and DeepGeMM all enforce ClusterShape<1,1,1> on SM_120, forgoing cluster GEMM entirely.

6.2.1a Microarchitectural Analysis: Cross-GPC Cluster Placement

The cluster CTA placement measurements ([S16/case_21], 100 launches, 100% deterministic) reveal that SM_120's DSMEM fabric is chip-wide — a significant architectural departure from SM_90 (Hopper) where clusters were confined to a single GPC. This finding has fundamental implications for the SM2SM interconnect topology and cluster-based kernel design.

The measured placement pattern follows a strict stride-20 rule: for cluster sizes >2, the hardware scheduler assigns CTA pairs to the first TPC of successive GPCs (SM 0,1 → SM 20,21 → SM 40,41 → SM 60,61). This stride precisely equals one GPC width (20 SMs), confirming the inferred die topology of ~5.5 GPCs with 10 TPCs (20 SMs) each, totaling 110 active SMs. The partial GPC 5 (SM 100–109) accommodates the remaining 10 SMs, consistent with a harvested die where one half-GPC is fused off.

Why cross-GPC placement is architecturally efficient: Despite spanning multiple GPCs (larger physical distances on die), the measured DSMEM latency is completely uniform at 220 cycles regardless of cluster size or span. SM pair {0, 61} (spanning 4 GPCs, ~15mm physical distance on a 750mm² die) shows identical latency to SM pair {0, 1} (same TPC, ~0.5mm distance). This is the definitive proof that the SM2SM fabric is a flat, non-hierarchical interconnect — there is no GPC-level routing hierarchy that would add per-hop latency for cross-GPC communication. The fabric behaves as a full crossbar (or functionally equivalent topology) with uniform O(1) latency to any destination SM on the die.

Multi-cluster contention is elegantly handled: When 4 simultaneous 8-CTA clusters launch, the scheduler assigns each cluster the next TPC pair within each GPC — Cluster 0 gets {0,1,20,21,...}, Cluster 1 gets {2,3,22,23,...}, Cluster 2 gets {4,5,24,25,...}, Cluster 3 gets {6,7,26,27,...}. This round-robin TPC assignment distributes fabric traffic uniformly across all physical links, preventing hotspots while maintaining the cross-GPC spread for maximum L2 partition coverage within each individual cluster. The cost of this spreading is zero — since DSMEM latency is uniform, there is no penalty for cross-GPC placement versus intra-GPC placement.

Generational departure from SM_90: On Hopper (SM_90), clusters were GPC-confined — the DSMEM fabric only connected SMs within a single GPC (max 16 SMs). SM_120's chip-wide DSMEM represents a fundamental interconnect redesign, likely enabled by the dedicated SM2SM fabric described in patent US20250173152A1 (crossbar topology with integrated barrier support). This chip-wide reach is what enables the measured 8-CTA and 12-CTA (non-portable) cluster sizes that span the entire die, and explains why the cluster barrier achieves O(1) scaling — the centralized CGA controller has equal-cost access to all participating SMs regardless of their physical location.

6.2.3a Microarchitectural Analysis: Cluster Barrier Hardware Decomposition

SASS disassembly of both barrier.cluster variants reveals a precise, measurable instruction-level cost decomposition that exposes the underlying hardware synchronization mechanism. The full barrier is overwhelmingly dominated by a single instruction — MEMBAR.ALL.GPU — which accounts for 77% of total cost, while the actual cluster synchronization hardware (the CGA controller) contributes only ~80 cycles regardless of cluster size.

The CGA controller is a centralized hardware counter: The O(1) scaling — 85 cycles constant for 2, 4, and 8 CTAs ([S16/case_5], [S7/case_53]) — is the definitive evidence for a centralized (non-tree) implementation. A tree-reduction barrier would scale as O(log N), adding ~40 cycles per doubling of cluster size. The measured constant-cost behavior proves the CGA controller maintains a simple per-phase hardware counter: each CTA atomically increments the counter on UCGABAR_ARV (via the SM2SM fabric sideband), and when the counter matches the cluster size, a broadcast release signal simultaneously unblocks all warps waiting at UCGABAR_WAIT. This centralized design is described in patent US20230289242A1 ("hardware-accelerated synchronization with O(1) barrier completion").

Only 6–8 cycles of useful ALU work fit between ARV and WAIT: This was verified by inserting independent IADD3 instructions between the barrier arrive and wait ([S16/case_9]). The short gap corresponds to the SMSP instruction issue pipeline depth — the warp is stalled as soon as the UCGABAR_WAIT instruction enters the issue pipeline, regardless of how many instructions are queued after ARV. This means overlapping useful computation with barrier latency is impractical; the arrive-to-wait window is effectively a fixed hardware property, not a tunable parameter.

The MEMBAR.ALL.GPU accounts for 77% of full barrier cost: This single instruction performs three sequential operations: (1) drains the SM's L1 write buffer to L2 by flushing all pending store requests, (2) waits for acknowledgment from all 12 L2 cache partitions confirming coherence (a GXBAR round-trip), and (3) invalidates stale L1 cache entries. Its 286-cycle cost is identical to the standalone fence.acq_rel.gpu measured in Chapter 2, §2.5 — confirming they share the same hardware path. Applications that can tolerate relaxed memory ordering (i.e., where cross-CTA data visibility is handled explicitly via st.async.shared::cluster + mbarrier, not implicitly via cache coherence) should ALWAYS use barrier.cluster.arrive.relaxed + barrier.cluster.wait to save 286 cycles per barrier invocation — a 4.4× cost reduction that directly translates to higher mainloop throughput in pipelined GEMM kernels.

6.2.4a Microarchitectural Analysis: DSMEM Bandwidth Scaling Evidence

The per-CTA bandwidth constancy across cluster sizes — 8.23, 8.23, and 8.15 GB/s for 2, 4, and 8 CTAs respectively (<1% variance) — provides the strongest measurement-based evidence for the DSMEM fabric's underlying topology. By formulating three competing hypotheses and testing each against the measured scaling behavior, we can definitively eliminate the shared-bus model and confirm an independent-channel architecture.

The three-hypothesis elimination proceeds as follows. Under a shared bus model (Hypothesis A), all CTAs in a cluster would share a single communication channel; as cluster size increases from 2 to 8, per-CTA bandwidth should decrease proportionally (8.2/4 ≈ 2.0 GB/s at 8 CTAs). The measured 8.15 GB/s at 8 CTAs — identical to the 2-CTA value — categorically falsifies this model. Under a point-to-point mesh (Hypothesis B), each CTA pair has its own dedicated channel, meaning aggregate bandwidth scales linearly with cluster size while per-CTA bandwidth remains constant. This precisely matches the measurements (16.46, 32.9, 65.2 GB/s aggregate for 2/4/8 CTAs). The non-blocking crossbar (Hypothesis C) also predicts constant per-CTA bandwidth for non-conflicting traffic patterns (ring), but may exhibit degradation under all-to-all traffic due to internal switching contention — consistent with the measured 68% BW retention under all-to-all versus ring.

The vector width scaling provides additional insight: v4.u32 (16B per operation) delivers 14.7× the bandwidth of scalar u32 (4B) for a 2-CTA pair (16.46 vs 1.12 GB/s). This sub-linear scaling (4× width → 14.7× BW) indicates that the per-operation overhead (instruction issue, fabric injection) is significant relative to the data transfer time — widening the operand amortizes this fixed overhead across more bytes per cycle.

A subtle directional asymmetry was observed in the ring traffic pattern: even-ranked CTAs consistently take ~4% more cycles than odd-ranked CTAs for the same amount of data transfer. This suggests an asymmetry in the physical routing — possibly upstream vs downstream on a ring-like physical wire layout, even though the logical topology provides point-to-point semantics. The physical wires connecting SMs may form a bidirectional ring at the silicon level, with the SM2SM fabric protocol abstracting this into a logically flat point-to-point interface. This is consistent with the uniform 220-cyc latency observed in §6.2.1a — the ring physical distance is negligible compared to the protocol/synchronization overhead that dominates the fixed latency.

6.3 DSMEM vs L2: Fabric Comparison

This section directly compares the two available inter-SM communication fabrics on SM_120: the dedicated SM2SM (DSMEM) fabric operating via point-to-point links, and the global GXBAR crossbar providing L2-coherent access. The choice between them is not a simple latency comparison — it is a fundamental "sync domain" decision that depends on whether the communication pattern requires ordering guarantees.

The fundamental trade-off is architectural: the L2 (GXBAR) fabric provides lower raw read latency (49.5 vs 225 cyc for 4B) because data traverses the high-bandwidth cache crossbar without protocol overhead, but any ordering guarantee requires a MEMBAR.ALL.GPU (286 cyc) that drains the L1 write buffer through all 12 L2 partitions. The DSMEM (SM2SM) fabric has higher raw latency (220 cyc one-way) due to fabric injection and transit, but provides native O(1) barrier synchronization at only 85 cycles — integrating ordering into the same fabric that carries data. This creates a 26× cost advantage for DSMEM on pure synchronization (86 vs 2,242 cyc) and a 3.5× advantage on combined data+signal patterns (952 vs 3,312 cyc/iter).

Both fabrics exhibit topology-independent latency: DSMEM shows 0% variance across all 8 cluster configurations (§6.2.1a confirmed the flat SM2SM fabric), while L2 shows <4% variance across all SM pair distances. This means the fabric choice can be made purely on the basis of communication semantics, without consideration of physical SM placement.

6.6 Microarchitectural Models

6.6.1 TMA Engine Internal Architecture

The TMA latency decomposition, combined with the internal pipeline structure disclosed in patent US 2023/0289292 A1 (Fig. 6), reveals a cleanly separable two-phase internal architecture: a fixed-cost descriptor decode phase (~440 cycles) followed by a linear data transfer phase (0.031 cyc/byte). This decomposition explains why TMA latency is size-independent for small tiles (≤1KB) but grows linearly for larger tiles, and why the deep hardware queue (≥32 entries) is architecturally essential for throughput.

The patent-disclosed pipeline (Fig. 6) consists of six sequential hardware stages, which map to the measured two-phase latency model:

  1. MIOC 604 (Memory I/O Controller): The SM-to-TMA interface that accepts the single UTMALDG instruction and routes it into the TMA processing pipeline.
  2. Request Queue 606: A deep FIFO (≥32 entries on SM_120) that buffers incoming requests, accepting both tensor requests (with descriptor pointer) and non-tensor requests (descriptor-less bulk copies). The queue enables decoupling of SM issue rate from TMA service rate.
  3. Descriptor Cache 608: Caches recently-used 128B tensor descriptors from shared memory, and speculatively prefetches the next request's descriptor to hide the smem read latency. A cache hit saves ~30 cycles of descriptor re-read.
  4. Setup Block 610: Collects all parameters (from descriptor + instruction encoding), validates correctness, and prepares the address computation state for the request generator. Operates in pipeline with the generator (Setup for request N+1 overlaps with generation for request N).
  5. Request Generator 616: The core computational engine — executes a hardware-implemented 5D nested loop that iterates over all elements in the requested tile, translates multi-dimensional coordinates to linear global addresses, checks out-of-bounds conditions, applies swizzle patterns for smem write addresses, and segments the tile into L2 cache-line-sized sub-requests.
  6. GNIC Interface 614 + Completion Tracker 618: Issues the segmented requests to the memory subsystem via the Generic Network Interface Controller, and tracks outstanding requests to signal mbarrier completion when all data has arrived at the destination shared memory.

Stages 1–5 collectively produce the Phase 1 fixed cost of ~440 cycles — dominated by descriptor decode and address generation, with zero data-size dependency (confirmed by the flat 479–482 cyc latency for 128B through 1KB tiles, and the zero additional cost for 5D vs 1D at the same tile size). Stage 6 produces the Phase 2 linear cost of ~0.031 cyc/byte, corresponding to the DMA write rate of 16.1 B/cyc — which precisely matches the shared memory write port bandwidth (128B per 8 cycles = 16 B/cyc). The complete latency model is therefore:

TMA_latency(size) = 440 + size × 0.031 cyc

TMA queue depth architecture: The measured ≥32 in-flight queue explains why TMA latency can be hidden despite the 479-cyc floor. With 32 outstanding descriptors being serviced concurrently, the effective per-tile throughput at saturation approaches the DMA write floor (139 cyc/ld measured at depth 32, consistent with the ~62 cyc DMA floor plus residual overhead). The 148-cyc issue rate ([S22/case_40]) means the SM can inject a new descriptor every 148 cycles; with ≥32 queue slots, the queue can accumulate requests during the long descriptor-decode phase without stalling the SM.

Architectural independence from compute pipes: Three independent measurements confirm the TMA is a physically separate DMA controller, not a mode of the existing LSU or TC hardware: (1) TMA + FP32 co-issue shows 0% overhead ([S22/case_53]) — TMA dispatch does not consume FMA issue slots, proving a separate issue port; (2) TMA + Tensor Core co-issue achieves 100% overlap ([S9/case_29]) — TMA operates fully asynchronously alongside HMMA with no stall interaction, proving physical separation from the TC datapath; (3) TMA issue rate of 148 cyc/op is 65× slower than the LSU rate (2.27 cyc/LDG), confirming that the descriptor decode pipeline (not instruction dispatch) is the throughput bottleneck — and that the deep queue exists precisely to hide this decode latency through overlapped service.

6.6.2 DSMEM Fabric Latency Decomposition

The 219–225 cycle cross-CTA read latency (6.5× the 34-cycle local shared memory baseline) can be cleanly decomposed into a five-step round-trip model. More significantly, the patent-disclosed compound request mechanism (US 2023/0315655 A1) reduces the producer-side synchronization cost from two full round-trips (~450 cyc) down to a single fire-and-forget injection of 5.77 cycles — a 78× reduction that fundamentally enables the efficient producer-consumer patterns measured in §6.3.

The five-step round-trip model decomposes the measured 225-cycle latency as follows:

  1. mapa address translation (4 cyc): The mapa instruction performs a PRMT-based address remapping that converts a cluster-scope virtual shared memory address into a (target_CTA_rank, segment_offset) pair. This is implemented as a simple register permutation — no TLB walk required at the source SM because the uTLB at the target SM handles the actual physical translation (patent US 2025/0173152 A1, §12: "uTLB executes translation at destination").
  2. Request injection (~8 cyc): The source SM's LSU identifies this as a remote-segment access (via the getctarank check), formats a request packet containing source CTA ID, destination segment index, offset, operation type, and data payload, then injects it into the SM2SM fabric's request FIFO. The 8-cycle cost includes the arbitration into the dedicated virtual channel.
  3. SM-to-SM fabric transit (~145 cyc): The dominant cost — the one-way fabric transit through the dedicated SM2SM interconnect. This is constant regardless of cluster size (2/4/8 CTAs) or physical SM distance (adjacent vs. cross-GPC), confirming the point-to-point mesh topology disclosed in patent US 2025/0173152 A1 where "SM-to-SM interconnect is independent of the L2 cache path." The 145-cycle transit includes the target SM's uTLB translation of segment index to physical SMEM address.
  4. Remote smem read (~34 cyc): At the target SM, the request is serviced by reading the local shared memory at the translated physical address — incurring the same 34-cycle access latency as a local smem read.
  5. Response return transit (~34 cyc): The read data is packaged into a response and sent back through the SM2SM fabric's response virtual channel. The asymmetry between forward (145 cyc) and return (34 cyc) paths reflects that the forward path includes uTLB translation overhead not needed on the return.

Fire-and-forget compound request model (patent US 2023/0315655 A1): The 5.77–6.0 cyc issue cost for st.async.shared::cluster represents only the time to inject the write request into the SM2SM fabric FIFO. The patent discloses that data writes and barrier-arrive signals are bundled into a single compound request packet, transmitted atomically through a dedicated FIFO-ordered virtual channel. This provides two critical hardware guarantees:

  1. Data-before-barrier ordering: Within the single VC, the data write to the destination SMEM completes before the barrier's atomic update (currentTxCnt increment). The consumer, upon seeing the mbarrier phase flip, is guaranteed that all data is already visible in its local SMEM — no additional fence or memory barrier is needed.
  2. No-ACK semantics: The producer does not wait for a write-completion acknowledgment. The interconnect hardware guarantees delivery via the FIFO channel; the producer proceeds immediately after the 5.77-cycle injection. The actual write completes 145 cycles later at the destination, but this latency is entirely hidden from the producer.

This compound request mechanism transforms the synchronization cost from the traditional two-RTT sequence (write → wait ACK → send barrier → wait ACK ≈ 450 cyc) to a single forward delivery of ~145 cyc visible to the consumer, with only 5.77 cyc charged to the producer. This is what makes DSMEM valuable for pipelined producer-consumer patterns: the producer pays only the injection cost, not the transit time. Combined with the topology-invariant 145-cycle transit (0% variance across all 8 cluster configurations), this confirms that the SM2SM fabric is a dedicated point-to-point mesh with per-pair FIFO channels — architecturally independent of the GXBAR/L2 global memory path.

6.7.1 Microarchitectural Analysis: Pipeline Stage Decomposition

The 608 cyc/tile 1-stage cost (Case 49) reveals that TMA latency dominates 78.8% of serial execution, making software pipelining via double/triple buffering the primary latency-hiding strategy on SM_120. The decomposition into five measurable sub-components quantifies each pipeline stage's contribution and explains why 3-stage triple buffering achieves near-optimal overlap (2.58×) while 4-stage regresses due to shared memory pressure.

The 1-stage serial decomposition breaks down as follows, with each component identified by its SASS instruction:

  1. TMA 2D Load — 479 cyc (78.8%): UTMALDG.2D performs the full descriptor-driven tile load: descriptor decode (~440 cyc fixed) plus DMA transfer. This is the Phase 1 + Phase 2 cost from §6.6.1, and represents the pipeline's critical path. The TMA operates completely asynchronously — the SM issues the instruction and blocks only at the subsequent mbarrier wait.
  2. mbarrier Wait — 5 cyc (0.8%): SYNCS.PHASECHK.TRANS64.TRYWAIT polls the transaction barrier for TMA completion. The near-zero cost (5 cyc) confirms that in serial execution the TMA has already completed by the time the wait is reached — the barrier flips immediately.
  3. ldmatrix.x4 — 33 cyc (5.4%): LDSM.16.M88.4 loads four matrix fragments from shared memory into registers. This is the smem→RF transfer that stages operands for the Tensor Core — a mandatory step on SM_120 where mma.sync reads from RF (unlike SM_100/110's wgmma which reads directly from Tensor Memory).
  4. HMMA.16816.F32 — 29 cyc (4.8%): A single 16×8×16 half-precision matrix multiply-accumulate. The Tensor Core execution is remarkably fast relative to the data delivery cost — compute is only 4.8% of total pipeline time, confirming that this workload is entirely memory-bound.
  5. Barrier reinit + overhead — 62 cyc (10.2%): SYNCS.EXCH.64 (mbarrier phase reset) + SYNCS.ARRIVE (signal completion) + loop control instructions. This overhead is irreducible and represents the minimum pipeline management cost per tile.

2-stage double buffering (322 cyc, 1.89×): The overlap model issues TMA(i+1) concurrently with Compute(i) + Barrier(i). The theoretical per-tile cost is max(479, 33+29+62) = max(479, 124) = 479 cyc, but the measured 322 cyc is better because the steady-state average amortizes the first tile's non-overlapped TMA across many iterations. Critically, the variance drops from σ=92 cyc (1-stage, exposing every TMA jitter spike) to σ=1.0 cyc (2-stage, buffer absorbs all variability) — confirming true steady-state pipelining is achieved.

3-stage triple buffering (236 cyc, 2.58×): With two TMA loads simultaneously in flight, the effective TMA cost halves: max(479/2, 124) = max(240, 124) = 240 cyc. The measured 236 cyc matches this prediction within 2%, validating the model. This is near-optimal for SM_120 because the TMA queue (≥32 entries) can easily sustain two outstanding requests, and the compute fraction (124 cyc) remains below the halved TMA cost.

4-stage regression (249 cyc, 2.44×): Adding a fourth buffer stage actually increases per-tile cost by 5.5% versus 3-stage. The root cause is shared memory pressure: four concurrent tile buffers increase bank conflict probability for ldmatrix loads, and the additional mbarrier management (four barriers to track/reinitialize) adds overhead that exceeds the marginal TMA overlap gain. This establishes 3-stage as the practical optimum for single-warp TMA-dominated pipelines on SM_120.

6.7.2 Why Warp Specialization Fails on SM_120

The 1.06× speedup for warp specialization (vs 1.89–2.58× for software pipelining) is the clearest evidence of SM_120's fundamental architectural difference from SM_100/SM_110. The root cause is the absence of Tensor Memory (TM/TMEM) on SM_120: without a dedicated operand buffer that separates producer and consumer hardware resources, warp specialization not only fails to help but actively destroys 68–73% of throughput through two compounding loss mechanisms.

The architectural divergence between SM_100/110 and SM_120 creates fundamentally different optimal scheduling strategies:

SM_100/SM_110 (wgmma + Tensor Memory): The wgmma / tcgen05 instructions read operands directly from a dedicated Tensor Memory (TM) buffer — a hardware SRAM separate from both shared memory and the register file. TMA loads can be programmed to write directly into TM, completely bypassing the smem→ldmatrix→RF path. This creates a natural producer-consumer split where the producer warpgroup manages the TMA→TM pipeline (using no TC resources) and the consumer warpgroup issues wgmma that reads from TM (using no LSU resources). The two roles operate on different physical hardware, making warp specialization essential for keeping both pipelines saturated.

SM_120 (mma.sync + Register File): The mma.sync instruction reads operands exclusively from the register file. Data must traverse the mandatory serial path: TMA → shared memory → ldmatrix → RF → MMA. Since every warp has access to both the LSU pipe (for TMA/ldmatrix) and the TC pipe (for MMA), and since TC+LSU co-issue achieves 100% overlap (§4, §9/case_29), every warp can both load and compute simultaneously. Dedicating a warp to "loading only" wastes its Tensor Core time slot — the TC hardware sits idle during that warp's scheduling turn.

Quantitative cost breakdown (Case 33, S9): The measured throughput loss of 68–73% results from two compounding factors:

  1. TC slot waste (−50%): In a 2P+2C configuration, the two producer warps never issue MMA instructions, leaving their TC time slots idle. With 4-warp round-robin scheduling across 4 SMSPs, this directly halves TC throughput from 0.0384 to ~0.019 mma/cyc. On SM_100, this loss doesn't occur because producer warps fundamentally cannot use the TC pipe (wgmma operands aren't in their registers).
  2. __syncthreads serialization (−18–23%): The producer-consumer handoff requires two __syncthreads barriers per tile iteration (after produce, after consume), adding 259 cyc of serialization overhead. This destroys the overlap that software pipelining (§6.7.1) achieves naturally without inter-warp synchronization — each warp independently manages its own TMA(i+1) ‖ Compute(i) overlap.

Combined: 0.0384 mma/cyc (all-compute baseline) → 0.0102 mma/cyc (2P+2C warp-specialized) = 73% throughput destruction. Even the less aggressive 1P+3C split yields only 0.0122 mma/cyc (−68%). The conclusion is definitive: on SM_120, software pipelining via per-warp double/triple buffering (§6.7.1) is the only effective latency-hiding strategy. Warp specialization — the cornerstone of SM_100/110 kernel design (CUTLASS 3.x, FlashAttention-3) — is not merely suboptimal but actively harmful on SM_120.

6.8 Key Insights

KI-6.1: DSMEM Wins on Sync, Loses on Raw Data Access

  • Phenomenon: DSMEM read latency is 4.5x worse than L2 (225 vs 49.5 cyc for 4B). No crossover point exists at any payload size. But for synchronized producer-consumer patterns: DSMEM + cluster barrier is 2.5-26x faster than L2 + acquire/release polling. Combined data+signal: DSMEM 3.5x faster (952 vs 3312 cyc/iter).
  • Mechanism: DSMEM's advantage is synchronization integration. The cluster barrier hardware (85 cyc relaxed, O(1) scaling across 2-8 CTAs) is 23-26x cheaper than L2 coherence protocol (2242 cyc per sync notification). The barrier accounts for 95% of all DSMEM sync overhead (85.7 out of ~90 cyc).
  • Patent basis: US20230289242A1 describes the hardware-accelerated cluster barrier arbiter (CGAERRBAR/UCGABAR_ARV). US20250173152A1 describes the DSMEM crossbar topology with integrated barrier support.
  • Confidence: H

KI-6.2: TMA Value Is Descriptor Addressing + Deep Pipelining, Not Raw Latency

  • Phenomenon: TMA 1D 1KB = 479 cyc. Manual ld.global (L1 cached) = 32 cyc. cp.async = 78 cyc. TMA is 16.7x slower than manual for single-operation cached data. But TMA in-flight queue >= 32 enables 11x pipelining speedup (1537 to 139 cyc/ld at d=32).
  • Mechanism: TMA's architectural contribution is: (1) descriptor-based multi-dimensional addressing without register-file pressure, (2) deep hardware-managed pipelining without software loop overhead, (3) mbarrier-integrated completion tracking for producer-consumer decoupling.
  • Patent basis: US12499052 describes the TMA descriptor-based addressing spine for efficient multi-dimensional data access.
  • Confidence: H

KI-6.3: Cluster Barrier O(1) Scaling -- 23-26x Cheaper Than L2 Acquire/Release

  • Phenomenon: Cluster barrier (relaxed) = 85 cyc constant for 2/4/8 CTAs. L2 atomic polling = 2242 cyc. mbarrier bilateral = 160 cyc vs L2 acquire/release 1977 cyc (12.4x).
  • Mechanism: Dedicated cluster barrier hardware resolves synchronization without traversing the L2 coherence protocol. The hardware arbiter achieves O(1) scaling because arrival notifications propagate via a dedicated sideband (SM2SM fabric), not through the memory crossbar (GXBAR).
  • Patent basis: US20230289242A1 describes the hardware-accelerated synchronization with O(1) barrier completion.
  • Confidence: H

KI-6.4: Named Barriers 14x Cheaper Than mbarrier for Intra-CTA Sync

  • Phenomenon: Named barrier round-trip 15 cyc vs mbarrier 211 cyc (14x). mbarrier.arrive alone costs 31 cyc. Named barrier is the correct primitive for intra-CTA warp specialization when TMA tracking is not needed.
  • Mechanism: Named barriers use the lightweight BAR.SYNC hardware path. mbarrier adds smem-based phase tracking (SYNCS.EXCH + SYNCS.ARRIVE + SYNCS.PHASECHK) which is necessary only for TMA completion signals or multi-CTA coordination.
  • Confidence: H

KI-6.5: SM_120 Is 4.4x TMA-Starved -- Deep Pipelining Essential

  • Phenomenon: TMA delivers 6.5 B/cyc vs MMA consuming 28.9 B/cyc -- a 4.4x data starvation ratio. 3-stage software pipeline achieves 1.92x speedup; 4th stage adds only 5.5%.
  • Mechanism: SM_120 has no Tensor Memory (TM) — all TC operands must pass through shared memory (smem -> ldmatrix -> RF -> MMA). This creates a fundamental bottleneck. The 4.4x starvation ratio requires at least 3-stage software pipelining to keep tensor cores utilized.
  • Confidence: H

KI-6.6: fence.proxy.async Dominates Sync Cost at Global Scope

  • Phenomenon: CTA-scope fence = 7 cyc; global scope = 163 cyc (+2229%). In TMA+mbarrier pipeline: sync fraction = 34% of mainloop (546/1615 cyc), with fence.proxy.async as dominant contributor.
  • Mechanism: Global-scope proxy fence forces L1-to-L2 coherence visibility, traversing the memory ordering hardware. CTA-scope stays within local SRAM. Applications should minimize global-scope fences and prefer CTA-scope when cross-CTA visibility is not required.
  • Confidence: H

KI-6.7: Warp Specialization -- Constant 259 cyc/iter Overhead

  • Phenomenon: Warp specialization adds fixed +259 cyc/iter vs cooperative (constant regardless of compute). Producer 39% idle. Consumer stall: 41.5% at 10 FMA, dropping to 1.9% at 2000 FMA. Producer:consumer ratio has <2% impact.
  • Mechanism: Sync overhead is amortized by compute. At production tile sizes (>2000 cyc compute), specialization overhead drops to <13%. Below ~100 cyc compute/tile, cooperative design is preferred. DeepGemm-style sync fraction: 20.6% at ~2000 cyc, 12.4% at ~4000 cyc.
  • Confidence: H

KI-6.8: Two Physically Separate Fabrics Create a "Sync Domain" Design Choice

  • Phenomenon: SM2SM fabric (DSMEM): 85 cyc one-way, 220 cyc round-trip. GXBAR fabric (L2): 354 cyc round-trip. Full cluster barrier: 371 cyc = MEMBAR.ALL.GPU (285, GXBAR) + arrive (85, SM2SM, overlapped).
  • Mechanism: The architectural choice between DSMEM (SM2SM fabric) and L2 (GXBAR fabric) for inter-SM communication is not a simple latency comparison. It is a "sync domain" choice: DSMEM provides O(1) barrier + data in one fabric, while L2 provides faster unsynchronized reads but requires MEMBAR.ALL.GPU (285 cyc, GXBAR) for any ordering guarantee.
  • Design decision tree: (1) If data+sync together is needed: DSMEM (3.5x faster pipeline). (2) If data-only without ordering: L2 (4.5x lower latency). (3) Never use L2 acquire/release for inter-SM sync -- it is 26x slower than cluster barrier.
  • Patent basis: US20230289189A1 Claim 6 states "SM-to-SM interconnect independent of the cache path." US20230315655A1 proposes composite write+arrive (0.5 RTT) on the SM2SM fabric -- this is the measured relaxed arrive at 85 cyc.
  • Confidence: H

KI-6.9: TMA Multicast Absence Creates "Cluster GEMM Impossibility" on SM_120

  • Phenomenon: All TMA multicast calls are SW-emulated (CALL.ABS.NOINC): 2,000-50,000x penalty. CUTLASS enforces ClusterShape<1,1,1>. Even the degenerate self-only mask triggers the emulation path.
  • Mechanism: On SM_100, cluster GEMM shares the A-operand tile across K CTAs via TMA multicast -- a single L2 read serves K copies (BW reduction by 1/K). On SM_120, each CTA must independently load its own A-tile from DRAM/L2, multiplying memory pressure by cluster_size. The DSMEM workaround (unicast + st.async.shared::cluster) adds 220 cyc per cross-CTA read, but is still cheaper than the SW-emulated multicast (>100K cyc).
  • Implication: SM_120 cluster GEMM is fundamentally limited to DSMEM-based B-operand sharing (not A-operand multicast). The optimal cluster strategy on SM_120 is cluster-2 for DSMEM-based attention KV sharing (FlashInfer MLA pattern), NOT for GEMM A-tile multicast. Any SM_100 kernel using TMA multicast must be restructured to unicast+DSMEM on SM_120.
  • Patent basis: US20230289190A1 describes the multicast tracking table at the L2 interface (LRC). SM_120 (GB202) lacks this LRC hardware -- the CALL.ABS.NOINC trap handler is the firmware fallback path. [SPECULATIVE: The LRC die area may have been repurposed for additional L2 cache or GDDR7 I/O on the consumer die.]
  • Confidence: H

6.9 Practical Design Guidelines

Based on the measured data, the following decision framework governs on-chip network and async primitives on SM_120:

Design Decision Recommendation Rationale
Inter-SM data + sync Use DSMEM + cluster barrier 3.5x faster than L2 pipeline
Inter-SM data only (no ordering) Use L2 global loads 4.5x lower raw read latency
Inter-SM sync notification Use barrier.cluster.relaxed 26x cheaper than L2 flag polling
TMA pipeline depth 3 stages (minimum) 2.58x speedup; 4th stage marginal
Intra-CTA warp sync Named barriers 14x cheaper than mbarrier
Global-scope fence Minimize; prefer CTA-scope 23x cost difference (7 vs 163 cyc)
GEMM cluster strategy Cluster-2 for KV sharing only TMA multicast hardware absent
Bulk data copy (>128B) cp.async.bulk or TMA cp.async.bulk 31x faster than cp.async at 16KB
Small cached data (<128B) cp.async.ca or manual ld.global 59 cyc (cp.async.ca) vs 479 cyc (TMA)
Epilogue reductions TMA cp.reduce.async.bulk.tensor 11.5x faster than scattered atomics
