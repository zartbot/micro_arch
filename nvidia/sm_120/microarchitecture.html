CHAPTER 01架构概览

SM_120 架构概览

SM_120 是 Blackwell 家族的工作站/消费级芯片 — 保留 mma.sync 全指令集，新增 FP6/FP4/MX 与 SIMD 8x4，但不支持 tcgen05、wgmma、Tensor Memory。

本章要点
  1. SM_120 (GB203) 是 Blackwell 家族的工作站/消费级芯片，保留 mma.sync 全指令集但不支持 tcgen05 / wgmma / Tensor Memory。
  2. CUDA 13.3 (PTX 9.3) 解锁了 setmaxnreg、SIMD 8x4、cvt.pack、FP8 cvt、MX narrow cvt、tensormap.replace、L2::cache_hint — 完善了 SM_120 的功能栈。
  3. 新增 FP6/FP4 dense MMA 和 MX block-scale MMA（Blackwell 独有 SASS：QMMA.SF / OMMA.SF）。
  4. 理论 DRAM 带宽 1344 GB/s（GDDR7 PAM3 28 Gbps/pin），实测单向 ~1176 GB/s（87.5%）。
  5. FP64 TC 从 pipe_tensor 移至 pipe_fp64_op_dmma，NCU 脚本需相应修改。

1.5 · 测试方法学简述

  • 计时基础：clock64（1 cycle overhead，0 stdev），单 warp 实验无需统计降噪。
  • Anti-DCE：所有测量结果写入 sink[] + asm volatile，防止 -O3 消除测量循环。
  • 循环控制：#pragma unroll 1 抑制编译器展开。
  • 取值策略：100+ 次测量取最小值/中位数（非均值），排除 OS 中断/DVFS 尾部噪声。
  • SASS 验证：每个 case 至少一次 cuobjdump --dump-sass 确认 PTX → SASS 映射。
  • 远程执行：CUDA_VISIBLE_DEVICES=N timeout 60 ./benchmark，sftp 传输。
Ch.02chip topology & SM structure

芯片拓扑与 SM 结构

110 SM 稠密分布在 9–10 个 GPC 内，每 SM 拆分为 4 个 sub-core，warp 按 warpid · 4 静态绑定——这是理解调度器、占用率与资源隔离的起点。

● 本章要点
  1. 110 SM · smid 稠密 [0,109]：每 GPC 12 SM，每 TPC 2 SM。
  2. 4 sub-core / SM：warp 按 warpid % 4 静态绑定，生命期内不迁移。
  3. Occupancy 上限：48 warps / 24 blocks / 65536 regs / 100 KB smem per SM。
  4. Boost 2580 MHz 持续稳定：抖动 < 0.3%，满载 5 分钟无 throttle。

2.1 · GB203 芯片拓扑与 GPC 结构

GB203 集成 110 个 SM，smid 空间为稠密 [0,109]——与 SM_100 (GB100) 的稀疏 smid 不同。拓扑遵循标准 Blackwell布局：2 SM/TPC，6 TPC/GPC，12 SM/GPC。Cluster 上限 12 息变顶了 GPC 容量。

2.2 · SM 内部 4 sub-core 所

每个 SM 包含 4 个 sub-core（亦称 warp scheduler partition），warp 按 warpid % 4 静态绑定。验证方法：FFMA 依赖链，W=1→4 elapsed 线性增长，W=5 跳至第二平台（ceil(5/4)=2）。

Ch.03execution pipelines & issue

执行管线与 指令发射

SM_120 每个 sub-core 提供 5 条独立管线，加一个完全独立的 V-Pipe 子单元；所有 9 个 pipe-pair 均支持 2-way co-issue，但调度器不支持真正的 3-way。

● 本章要点
  1. 5 条标量 pipeline：P0 (INT∪FP32 共享) · P1 (FP64) · P2 (SFU) · P3 (LSU) · P4 (Tensor)。
  2. V-Pipe 是 P0 上的独立子单元，VIADD/VIMNMX 吞吐比 IADD3 高 61%。
  3. 9 个 pipe-pair co-issue 确认，SFU+ALU 双发射，但不支持三路发射。
  4. Warp 调度器：Round-Robin · 0 cyc 切换 · CGGTY 阈值 5 warps。

Ch.04integer ALU

整数 ALU

第 6 章 · Memory Hierarchy

从 SMEM 到 DRAM 的七级阶梯

SM_120 的存储层级在 SM_100 的基础上微调：L1 进一步加速、L2 切换为 96 MB 单切片但被显式 split 为 “near” 与 “far” 两段，DRAM 升级为 GDDR7 384-bit。整套体系呈现 SMEM 33 → L1 33 → Const 34 → L2-near 79 → L2-far 180 → DRAM 372 → stride DRAM 900 的清晰阶梯。

  • Shared Memory — 100 KB/SM、32 banks × 4B、± 2 cyc/way conflict。
  • L1 / Texture — ~24 KB、128 B line、≥32-way、hit ≈33 cyc。
  • L2 — 96 MB total，near-half 79 cyc · far-half 180 cyc，ld.cg 在 SM_120 上默认 bypass 至 DRAM。
  • DRAM — GDDR7 384-bit、1344 GB/s peak、实测 ~1186 GB/s、372 cyc latency。

6.1 Shared Memory — 100 KB / SM · 32 banks

  • 每个 SM 独享 100 KB SMEM，与 SM_100 持平；bank 宽 4B，结构与 Hopper 一致。
  • 无 conflict 延迟 33 cyc；N-way 冲突 ≈ (N-1) × 2 cyc 额外惩罚。
  • ldmatrix.x4 / stmatrix.x4 为 Tensor Core 提供 16×16 / 16×8 块拼装加载，128 B 宽、与 bank 调阶对齐。

6.2 L1 / Texture Cache — 被“升级”后的近隔邻

  • SM_120 的 L1 从 SM_100 的 ≈41 cyc 加速至 33 cyc，与 SMEM 持平。
  • 容量~24 KB、128 B line、≥32-way。default policy → L1::ca。
  • 与 SM_100 的 L1.5 (未公开中间层) 被去除；ld.ca 后直接进 L2。

6.3 L2 — 96 MB “near/far” 二分

第 7 章 · Tensor Core (Blackwell 5th-gen)

RTX PRO 5000 上的 统一管线 Tensor Core

SM_120 (GB203) 是 Blackwell 工作站/消费级芯片，Tensor Core 路径与 数据中心级 SM_100 (GB100) 显著分化：只支持 mma.sync 全套指令，不支持 wgmma / tcgen05.mma / Tensor Memory / 2-SM cooperative GEMM。所有非 FP64 精度共享统一 TC 管线，实测 lat = 29 cyc、tput = 23 cyc/mma。

  • 仅 mma.sync — SM_120 没有 wgmma 也没有 tcgen05；CUTLASS 4.5 在该芯片上走 mma.sync + CLC scheduler 路径。
  • 统一 TC 管线 — FP16/BF16/TF32/FP8/FP6/FP4/MX/INT8 指令延迟均为 29 cyc，吞吐 23 cyc/mma；FP64 走独立 DMMA (lat=175, tput=449 cyc/mma)。

第 8 章 · Async Memory Engine

TMA 与 cp.async.bulk：488 cyc 装载、33 cyc 卸货、multicast 报废

SM_120 (RTX PRO 5000 72GB) 保留了 cp.async、cp.async.bulk 与 cp.async.bulk.tensor (TMA) 三套异步引擎接口，1D bulk 实测 488 cyc、TMA 2D load 620 cyc、TMA 2D store 极快 33 cyc (19× faster than load)。但 TMA multicast 完全报废 (~10000× 退化)、TMA im2col 不支持、cp.async.bulk smem→smem (cluster) 会触发 illegal instruction。CUDA 13.3 解锁 tensormap.replace (26-45 cyc/op) 与 L2::cache_hint 让 TMA 描述符可在 device 上动态修改。

  • cp.async.bulk 1D — 488 cyc latency · max 4096 B/次 · throughput 75 cyc/iter。
  • TMA 2D store 极速 — 仅 33 cyc fire-and-forget · 19× faster than 2D load · epilogue 首选。

  • pipeline ≥32 outstanding — 才能达到 90% peak · multi-CTA 完美线性 · 110 CTA 聚合 1483 GB/s。

第 9 章 · Warp Scheduler & Control Bits

128-bit 指令字与 17-bit 控制码：stall 严格 1:1，reuse 是 no-op

SM_120 采用 128-bit (16 B) 固定宽度指令编码，Upper 64-bit 中 bits [57:41] = 17-bit Control Code (stall / yield / R-W scoreboard / wait-barrier mask)。 Stall counter 严格 1:1 线性 — 若 stall < 指令 write-back latency 会 静默读取 stale 寄存器 (不报错、不 trap、仅读旧值)。 yield=1 完全阻断 latency hiding，即使 32 warps 也回退到 336 cyc/iter。 而 .reuse 位在 SM_120 上是 hardware no-op，operand collector 已 完全吸收 bank conflict，WAR hazard overhead = 0。

  • 128-bit 指令 · 控制码 17 b — [44:41] stall (4) · [45] yield (1) · [48:46] write-sb (3) · [51:49] read-sb (3) · [57:52] wait-mask (6)。
  • Stall 严格 1:1 — stall=4 是 IADD3 依赖链的最小安全值 · stall=0 退化到 hardware scoreboard fallback (~25 cyc/inst)。
  • Scoreboard 深度 ≥12 — load queue ≥32 · load 代价从 401→83 cyc 递减 · SB + stall 可共存。
  • Reuse bit = 0 效果 — SM_120 上为 hardware no-op · RFC 无可观测 hit/miss · operand collector 完全吸收 bank conflict。
  • Barrier 线性 — bar.sync cost ≈ 28 + 1.9×warps cyc · 32 warps = 87.5 cyc。
关键安全规则：control code 的 stall counter 必须 ≥ 目标指令的 write-back latency，否则读取结果会 静默损坏 — 不报错、不 trap、仅读取 stale register value。生产代码请勿下调 stall 求快。

10.3 多 GPU 调优要点

友好场景

  • Pipeline parallel：比 inference latency 需求低，每 2–4 层一 GPU。
  • Data parallel：gradient all-reduce 走 NCCL ring。
  • 同 switch 内 4 卡 GEMM tile sharding 利用 53 GB/s P2P 局部带宽。

不友好场景

  • Tensor parallel：all-gather 每 step 上百 GB，PCIe 会成为热点。
  • Expert parallel：MoE all-to-all 频繁小拷贝，需 NVLink。
  • 跨 switch 高频随机访问 · 必须走 CPU root，带宽减半。
全局小贴士： 在 RTX PRO 5000 集群上设计多 GPU 系统时，应主动选择 pipeline parallel 或 data parallel 而非 tensor parallel；若需 TP，考虑路由 GPU 在同一 PCIe switch 下以使用 53 GB/s 本层带宽，避免跨 switch 热点。

第 11 章 · Power, Efficiency & Optimization

功耗、效率与优化建议：MX-FP4 / Sparse INT8 6.9 TF/s/W · CUTLASS NVFP4 975 TFLOPS

RTX PRO 5000 72GB 官方 TDP 350 W，TC 满载仅 167–174 W，最高业务功耗 222 W（63% TDP，未触发 cap），稳态运行 boost 锁在 2580–2587 MHz，30s 满载最高温度仅 56 °C（热阈值的 62–67%，无 throttle）。 最佳能效负载为 MX-FP4 / Sparse INT8 TC：6.8–6.9 TFLOPS/W（FP16 的 4×、FP32 的 28×）；端到端 GEMM 上限为 CUTLASS NVFP4 = 975 TFLOPS（80.7% 效率）。FlashAttention 在 SM_120 上是 ALU-bound（softmax 占 63.1%），并非 TC-bound。

  • 能效冠军 — MX-FP4 / Sparse INT8 TC = 6.8–6.9 TFLOPS/W，FP16 的 4×、FP32 的 28×。
  • 散热余量 — 满载 167–174 W（TC）/ 222 W（max）远低于 350 W TDP，56 °C 远低于 83–90 °C 阈值，无 thermal throttle。
  • FA 瓶颈 — FlashAttention 在 SM_120 上是 ALU-bound（softmax `expf` 占 63.1%，SFU pipe），非 TC-bound。

11.9 关键优化建议清单（20 项）

  1. Block size：每 sub-core ≥ 2 warps（每 block ≥ 64 threads），触发 CGGTY latency hiding。
  2. Sparse MMA：必须使用 ::ordered_metadata，否则 9× 性能陷阱。
  3. 编译：FP6 / FP4 / MX 及 setmaxnreg 需 -gencode arch=compute_120a,code=sm_120a + CUDA 13.3。
  4. Atomic：优先 atomicAdd.f32（24 cyc）而非 .u32（45 cyc）；用 TMA reduce（105 cyc）替代 atomicAdd（45 cyc）做大规模 reduce。
  5. DRAM stride：避免 stride ≥ 48 KB（BW 降至 1/5）。
  6. Cache modifier：注意 ld.cg 在 SM_120 上绕过 L1 + L2 直达 DRAM。
  7. Cluster：max = 12（非 SM_100 的 16）；barrier 不随 cluster size 变。
  8. TMA：pipeline depth ≥ 32；multicast 不可用；im2col 不可用。
  9. Fast-math：对 sinf / cosf / logf 加速 4×+，对 division 加速 14×。
  10. FP64：1:64 throttle、非流水线、2 EU/SM — 不适合 FP64 密集计算。
  11. FP16x2 / FP32x2：无吞吐增益，仅节省寄存器。
  12. Warp scheduler：Round-Robin，0 cyc 切换，nanosleep 完全释放 dispatch slot。
  13. Stall counter：必须 ≥ instruction latency，否则静默数据损坏。
  14. Inter-SM：无 GPC 局部性，所有 SM pair 延迟均匀（~1240 cyc RTT）。
  15. Cross-GPU：atom.sys remote 1899 cyc（38.6× local），避免做同步。
  16. DVFS：1500–1800 MHz 是最佳能效点；boost 频率 +73% 吞吐但 +127% 功耗。
  17. Power：基础设施功耗 ~80 W 占总功耗 > 45%，减少活跃 GPC 比减少 per-SM 利用率更有效。
  18. match.any：370 cyc，应避免使用；用 match.all（24 cyc）或 vote+popc 替代。
  19. setmaxnreg：CUDA 13.3 可用，支持 warp specialization；≤ 4 warps 无竞争，16+ warps 有 hang 风险。
  20. CUTLASS NVFP4：975 TFLOPS — SM_120 推理场景终极选择（需 CUTLASS 4.5 + CUDA 13.3）。
