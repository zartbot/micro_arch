2.1.1 Shared Memory Controller: Decoupled Accept-Resolve Pipeline

SM_120 introduces a fundamentally new shared memory controller architecture that near-eliminates bank-conflict throughput penalties. Unlike prior architectures (SM_80–SM_90) where N-way bank conflicts caused proportional Nx serialization, SM_120 uses a decoupled accept-resolve pipeline that separates load acceptance from conflict resolution — the accept stage never stalls, regardless of pending conflicts.

Core Phenomenon: Throughput–Latency Asymmetry

The same 32-way bank conflict produces dramatically different effects on two metrics:

Architectural Model: Three-Stage Decoupled Pipeline

The asymmetry is explained by a three-stage pipeline where the accept and resolve stages are fully decoupled:

Stage Function Rate Governs which metric
ACCEPT Receives new load requests from the warp ~2 cyc/req, never blocked by pending resolve Throughput
CONFLICT RESOLVE Serializes conflicting bank accesses +2 cyc/way, deterministic Latency
DATA RETURN Delivers data to register file Indep loads: non-blocking; Dep chains: waits for resolve Exposes which metric

Key decoupling properties:

  • Accept never stalls: Even while Resolve is processing a 32-way conflict (62 extra cycles), Accept continues receiving new requests at full rate.
  • Per-warp isolation: Each warp (or SMSP) has an independent port — one warp's conflict resolution never blocks another warp's load acceptance.
  • Resolve absorbs multiple concurrent requests: Conflict resolution for multiple in-flight requests proceeds in parallel within the resolve stage.

Why throughput is nearly unaffected: Independent loads “complete” at the Accept stage — the subsequent Resolve and Return happen in the background without blocking the next load's issue. Only when the program needs the load's result (dependent chain) does the full serialized Resolve cost become visible.

Port contention model: The sharp transition between 16 and 32 warps (from 33.9 to 63.8 cyc/ld — almost exactly 2× at 1.88×) suggests the smem controller has a limited number of ports that can service concurrent warp requests. With 4 SMSPs per SM and each SMSP managing up to 12 warps:

2.1.3 L1 Cache Structure: Fully Associative with Coherence Overhead

The L1 data cache on SM_120 is unified with the texture/non-coherent cache path, with measured properties that constrain its microarchitecture:

  • Capacity: 89 KB effective (1 KB precision cliff, NCU confirms 88 KB). L1 + texture/NC share 100 KB physical SRAM with smem; carveout API is functional with 1:1 byte trade-off. L1 disabled entirely at 64+ KB smem allocation. [S24/case_1,2,3,4,5]
  • Associativity: ~712-way (= 89 KB / 128 B effective; up to 768-way at full 96 KB physical SRAM) — fully associative. Stride-scan probe (stride=128B baseline vs stride=4KB pathological set-conflict probe) shows identical cliff at N=720, ruling out any set-associative N-way design with N<712. [S8/case_74, S24/case_6]
  • Latency: 44 cyc for L1 hit (pointer-chase; 43.88-43.89 cyc constant across all access patterns and strides per [S26/case_7-9])
  • Smem latency: 33.89 cyc for identical access pattern
  • Delta: +10.1 cyc (29.8% overhead)

The 10-cycle L1-vs-smem gap reveals coherence protocol cost: Both L1 and smem access the same 100 KB physical SRAM array (confirmed by carveout experiments showing 1:1 displacement [S24/case_2,3,4]). The 10-cycle difference represents the L1 tag lookup + coherence check that smem bypasses entirely.

The fundamental architectural distinction is that shared memory uses software-managed addressing — the programmer explicitly controls which bank and offset to access, so the hardware needs only decode the address, select the bank, and read the SRAM. The L1 cache path, by contrast, must perform hardware tag matching to determine whether the requested address is present in the cache, plus a coherence state check to ensure the cached data is still valid (not invalidated by another SM's write). These two extra pipeline stages — tag CAM lookup and coherence protocol — account for the full 10-cycle overhead.

The decomposition of this overhead [SPECULATIVE — inferred from the latency delta between L1 and smem under identical access patterns, NOT directly measured per-stage]:

Component Estimated cost Rationale
Tag CAM lookup ~4–5 cyc ~712-entry fully-associative content-addressable memory; requires parallel comparison of the request address against all valid tag entries
Coherence state machine ~3–4 cyc MOESI-like protocol check — must confirm line is in Shared/Exclusive state and not pending invalidation from cross-SM coherence traffic
Multiplexer/hit-data select ~1–2 cyc Once the matching entry is identified, data routed from the correct SRAM way through a wide multiplexer to the output port

This decomposition is architecturally grounded: the tag CAM is the dominant cost because full associativity requires comparing all ~712 tag entries simultaneously (a wide parallel comparison network), while a set-associative design would limit comparisons to N ways within a single set. The coherence check cannot be overlapped with the tag lookup because the coherence state bits are stored alongside the tag — they must be read together. The mux/select stage is the final routing step after hit determination.

Critically, the SRAM read itself takes the same time in both paths — this is confirmed by the fact that both L1 and smem displace each other 1:1 in carveout experiments [S24/case_2,3,4], proving they access the same physical array. The 10-cycle gap is purely protocol overhead, not a difference in memory access time.

Full associativity is viable at this scale: ~712 entries (up to 768 physical) require a tag CAM of comparable depth. At 89 KB effective capacity, this is a reasonable area cost — well-precedented in the NVIDIA L1 lineage (V100, A100, H100 all use fully-associative L1 per Wong 2010 / Jia 2018 / Mei & Chu 2017) and < 0.001% of GB202's per-SM transistor budget. Full associativity completely eliminates conflict misses that plague set-associative designs; applications with strided access patterns (matrix column traversal, gather operations) benefit enormously. The empirical falsifiability of "true fully-associative vs hash-indexed N-way" is addressed by the stride-scan probe above ([S24/case_6]): a hash-indexed N-way with N<712 would necessarily cliff at N (=N-way) under stride=4KB; the observed stride-independent cliff at N=720 (=capacity) is consistent only with NUM_SETS=1.

Little's Law derivation for SM_120 TensorCore:

The MMA pipeline depth = Latency / Throughput = 29 / 15.88 = 1.83. This means only ceil(1.83) = 2 accumulator chains achieve 97% of peak throughput, and 4 chains reach 100%. The register file (255 regs) can hold 30× more chains than needed — the shallow pipeline depth (1.83) is remarkably generous compared to the ldmatrix pipeline (19.6).

The practical consequence: the register file's 255-register limit accommodates up to 60 independent accumulator chains (240 regs for accumulators + 12 overhead) before any spill occurs. Production GEMM kernels (CUTLASS, DeepGeMM) typically use 16–32 tiles, well within this safe zone.

Spill penalty mechanism: At 62+ chains, ptxas inserts STL/LDL (local memory spill/fill) instructions. Each spill pair occupies the LSU pipe, directly competing with ldmatrix for issue bandwidth. Practical impact on GEMM kernels:

  • Register budget rule: SM_120 allows 60 accumulator tiles (240 regs) without throughput loss. Production GEMM kernels (CUTLASS, DeepGeMM) typically use 16–32 tiles, well within the safe zone.
  • Spill is graceful, not catastrophic: Even at 64 chains (9100B spill), degradation is only 19%. The L1 cache efficiently absorbs spill traffic at single-warp occupancy. The primary cost is LSU pipe contention (STL/LDL competes with ldmatrix), not memory latency.
  • Occupancy vs tile size trade-off: At 252 regs/warp, only 1 warp fits per SMSP (255 max). Reducing to 128 regs allows 2 warps — doubling TC utilization through warp-level parallelism, which is more valuable than the marginal single-warp ILP beyond 4 chains.

2.2.1 L2 Internal Architecture: XOR-Hashed Set Indexing

The SM_120 L2 cache uses a fundamentally different indexing scheme from prior architectures: hardware XOR-based address hashing replaces traditional modular set indexing. This has profound implications for workload robustness.

Evidence for XOR hashing (four independent probes, all showing NO stride-dependent conflict):

  1. Stride sweep (Case 76): Pointer-chase at strides 64B through 4KB in a 4 MB buffer yields uniform 353-356 cyc/ld. Under modular indexing, specific strides would map to the same set, causing conflict misses. None observed.

  2. Wide stride sweep (Case 89): Strides from 1KB to 128KB in a 128 MB buffer produce uniform ~370 cyc access latency (CoV < 1%). The complete absence of stride-dependent anomalies across 7 decades of stride is impossible under modular indexing.

  3. Extreme conflict probe (Case 31, Test 1): 40 pointers at stride = 96MB (the full L2 size). Under modular indexing, all 40 pointers would map to a single set, overflowing any plausible associativity. Measured: constant 353-378 cyc — all L2 hits. This proves the addresses are NOT mapped to the same set.

  4. Dense conflict probe (Case 31, Test 2): 256 pointers at stride = 6MB. This 1.5GB address span would create 256-way set conflict under modular indexing. Measured: 362 cyc — still all L2 hits. Conclusive proof of hash-based distribution.

2.2.2 L2 Sector Architecture: 128B Tags, 32B Fetches

The L2 uses a sectored cache line design with 128B tag granularity but 32B transfer granularity:

This is architecturally significant — it suggests the GDDR7 memory controller scheduling algorithm interacts with the access pattern at this specific working set size, possibly due to row buffer hit rate changes when the stride wraps around at a DRAM-friendly boundary.

2.5 Memory Ordering, Coherence, and Cache Modifiers

2.5.1 Memory Consistency Model

[source: S8/case_64-71, S8/case_123-125, S7/case_41-42]

SM_120 implements a hierarchical relaxed memory model that is TSO-like at the hardware level, with five concentric ordering scopes and four primary synchronization mechanisms. The key architectural insight is that sequential consistency (SC) carries zero overhead compared to acquire-release (acq_rel) on SM_120 — the hardware already provides TSO-strength ordering, making SC effectively free at all scopes.

Five Ordering Scopes

Each scope defines a visibility domain — operations ordered at a given scope are guaranteed visible to all threads within that domain:

Scope Fence Cost Hardware Mechanism Visibility Domain
Thread 0 cyc Compiler barrier only (no SASS emitted) Single thread — program order
CTA 6–7 cyc Store buffer drain to L1 (MEMBAR.SC.CTA) All threads in one CTA (single SM)
Cluster 85–376 cyc CGA barrier fabric (UCGABAR_ARV) All CTAs in the cluster (2–8 SMs)
GPU 285 cyc GXBAR + L2 drain + ack from 12 partitions (MEMBAR.ALL.GPU) All 110 SMs on the GPU
System 920 cyc GPU drain + PCIe bridge ordering (MEMBAR.SC.GPU+SYS) CPU, other GPUs, all PCIe peers

The cost cliffs between scopes are dramatic: CTA→GPU is 40.7× (7 → 285 cyc), and GPU→SYS is 3.2× (285 → 920 cyc). The fundamental rule is: always use the narrowest scope that satisfies your visibility requirement.

Four Synchronization Mechanisms

SM_120 provides four distinct mechanisms for data synchronization, each with different trade-offs:

1. Fence Operations (fence.acq_rel, fence.sc) — order all prior memory operations within a scope without targeting a specific address: 

// CTA-scope fence: drains store buffer to L1 (7 cyc)
// SASS: MEMBAR.SC.CTA
asm volatile("fence.acq_rel.cta;" ::: "memory");

// GPU-scope fence: drains to L2, waits for ack (285 cyc)
// SASS: MEMBAR.ALL.GPU
asm volatile("fence.acq_rel.gpu;" ::: "memory");

// System-scope fence: GPU drain + PCIe ordering (920 cyc)
// SASS: MEMBAR.SC.GPU + MEMBAR.SC.SYS
asm volatile("fence.acq_rel.sys;" ::: "memory");

2. Acquire/Release Loads and Stores — ordering semantics attached directly to individual memory operations: 

// Acquire load: invalidates L1, forces L2 coherence check (393 cyc)
// SASS: LDG.E.STRONG.GPU
asm volatile(
    "ld.acquire.gpu.global.b32 %0, [%1];"
    : "=r"(val) : "l"(addr) : "memory");

// Release store: drains store buffer before this store (920 cyc)
// SASS: STG.E.STRONG.GPU + MEMBAR.ALL.GPU
asm volatile(
    "st.release.gpu.global.b32 [%0], %1;"
    :: "l"(addr), "r"(val) : "memory");

// Complete producer-consumer handshake pattern:
// Producer: st.release.gpu [flag], 1;   → 920 cyc
// Consumer: ld.acquire.gpu r, [flag];   → 393 cyc per poll iteration

3. Barrier Synchronization — explicit synchronization points where all participating threads must arrive: 

// Intra-CTA barrier: 1 cyc (32–768 threads), 16 cyc (1024 threads)
// SASS: BAR.SYNC.DEFER_BLOCKING (enables warp scheduling during wait)
__syncthreads();

// Cluster barrier (relaxed): 85 cyc, O(1) with cluster size
// SASS: UCGABAR_ARV + UCGABAR_WAIT
asm volatile("barrier.cluster.arrive.relaxed.aligned;" ::: "memory");
asm volatile("barrier.cluster.wait.aligned;" ::: "memory");

// mbarrier for TMA pipelines: arrive 31–34 cyc, try_wait 5 cyc (ready)
// SASS: MBARRIER.ARRIVE + MBARRIER.TRY_WAIT
// NOT-READY polling costs 101 cyc/iteration (6.9× vs ready)

4. Atomic Operations with Ordering — combine memory modification with ordering semantics: 

// Relaxed atomic: only RMW, no ordering (405 cyc L2 round-trip)
asm volatile("atom.global.relaxed.gpu.add.u32 %0, [%1], %2;"
             : "=r"(val) : "l"(addr), "r"(delta) : "memory");

// Release atomic: orders prior stores before this atomic (552–591 cyc)
// Used for grid sync counters
asm volatile("atom.release.gpu.global.add.u32 %0, [%1], 1;"
             : "=r"(val) : "l"(counter) : "memory");

// Fire-and-forget reduction: 5× faster throughput (10.34 vs 52.17 cyc/op)
// No return value → no scoreboard wait
asm volatile("red.global.add.u32 [%0], %1;"
             :: "l"(counter), "r"(delta) : "memory");

Production Synchronization Patterns

The following patterns combine the primitives above into complete synchronization idioms used in production GPU kernels:

Grid Sync (all SMs synchronize, ~1242 cyc constant across 2–110 SMs): 

// Grid synchronization via atomic counter + acquire polling
// Total round-trip: ~1242 cyc (~0.59 μs), constant with SM count
atom.release.gpu.global.add.u32 r, [grid_counter], 1;  // signal arrival
spin:
  ld.acquire.gpu.global.b32 r2, [grid_counter];         // poll completion
  setp.lt.u32 p, r2, num_blocks;
  @p bra spin;

TMA Pipeline Fence (CTA-local async ordering, 7 cyc — near-free): 

// Ensure TMA/cp.async completions are visible before smem reads
// SASS: FENCE.VIEW.ASYNC.S — only 7 cyc at CTA scope
asm volatile("fence.proxy.async.shared::cta;" ::: "memory");
// Now safe to read smem data written by TMA

// Global scope (163 cyc) — needed for cross-CTA async visibility:

// SASS: STG.E with no L1 allocate hint

2.5.1a Three-Domain Coherence Architecture

The fence cost hierarchy reveals that SM_120 implements three physically distinct coherence domains with dramatically different costs:

Store buffer decoupling evidence: The relaxed store at 4.8 cyc vs CTA-release store at 11.9 cyc (2.5×) proves the store buffer accepts writes at near issue rate. Async proxy separation: fence.proxy.async at 168 cyc (vs 285 for fence.acq_rel.gpu) proves that asynchronous copy operations (cp.async, TMA) are tracked via a separate completion mechanism that can be drained independently. Reduction fire-and-forget path: red.add at 10.34 cyc/op throughput vs atom.add at 52.17 cyc/op (5× faster) exploits the store-buffer-like fire-and-forget path. The L2 atomic unit accepts reduction requests at high rate as long as no return value is needed. The 378-cyc dependent atom.add latency represents the full L2 RMW round-trip, while red.add only pays the issue cost. This confirms the L2 atomic unit has separate accept and result-return ports.

Root-cause analysis per transition:

Fence CTA→GPU (40.7x): The independent variable is fence scope (CTA vs GPU). CTA-scope fence (7 cyc) stays entirely within the SM's local pipeline — it only orders operations visible within one cooperative thread array. GPU-scope fence (285 cyc) emits MEMBAR at GPU scope, which must drain the SM's store queue and force an L2 round-trip to ensure all prior writes are visible across all 110 SMs. The 40.7x cliff is the cost of crossing from SM-local to chip-wide L2 coherence.

Fence GPU→SYS (3.2x): The independent variable is again fence scope (GPU vs SYS). The system-scope fence adds PCIe/interconnect ordering on top of GPU-wide L2 drain. The incremental ~635 cyc (920−285) represents the round-trip latency for ensuring writes are visible to the CPU and other PCIe peers (~440 ns, consistent with PCIe coherence latency). The 3.2x ratio is smaller than the GPU/CTA ratio because PCIe overhead is additive, not multiplicative.

Store relaxed→release.gpu (191.7x): The independent variable is memory ordering qualifier (relaxed vs release.gpu). A relaxed store (4.8 cyc) fires-and-forgets into the SM's store buffer — the thread does not wait for the data to reach L2. A st.release.gpu emits STG.E.STRONG.GPU + MEMBAR.ALL.GPU per store, which drains the entire store queue and forces an L2 round-trip for every pending write before the release-store itself completes. The 191.7x cliff is the largest in the table because it combines store-buffer bypass latency (normally hidden) with GPU-scope fence overhead.

Load default→acquire.gpu (8.9x): The independent variable is coherence qualifier (default .ca vs .acquire.gpu). A default load (44 cyc) hits in L1. An acquire-qualified load forces L1 invalidation and an L2 coherence check (SASS suffix STRONG.GPU), converting what would be an L1 hit into an L2 round-trip at 393 cyc. The ~350 cyc acquire cost closely matches independently measured L2 hit latency (354 cyc), confirming the mechanism is L1 bypass. GPU vs SYS acquire differ by only ~1.5 cyc, indicating L1-to-L2 bypass dominates over scope extension.

Atomic relaxed→release.gpu (91.4x): The independent variable is ordering qualifier on atomics (relaxed vs release.gpu). A relaxed atomicAdd (5.7 cyc dispatch, ~378 cyc completion) performs a simple L2 read-modify-write. A red.release.gpu (521 cyc) adds ~520 cyc of GPU-scope ordering overhead: the release fence must drain all prior stores before the atomic itself can be issued through the coherent memory path. Consistent with GPU fence (285 cyc) + L2 atomic completion (~235 cyc).

LDG .ca→.lu (8.2x): The independent variable is cache eviction hint (.ca keep-in-cache vs .lu last-use). The .lu modifier (LDG.E.LU at SASS level) evicts the cache line from L1 after reading. Any subsequent re-access to the same address must go to DRAM (354 cyc) instead of hitting L1 (43 cyc). The 8.2x cliff is the L1-to-DRAM latency ratio, matching the memory wall measured elsewhere (§2.7.1).

STG .wb→.cg (0.08x, inverse cliff): The independent variable is cache write policy (.wb write-back vs .cg cache-global bypass). The .wb modifier writes through L1; a subsequent load-back must traverse L1 write-back coherence (~43 cyc). The .cg modifier bypasses L1 entirely — the store goes directly to L2, and a subsequent .cg load also bypasses L1, enabling store-to-load forwarding at L2 in just 3.59 cyc. Store-only throughput is identical (5.85 cyc/st) across all modifiers; the difference only manifests on store+load-back patterns where L1 coherence overhead is exposed.

Per-case scenario and root-cause analysis:

Raw 4B read (L2 wins 4.5x): Pointer-chase latency through a 256-entry linked list, measuring single-word dependent-read cost. DSMEM path uses ld.shared::cluster (SASS: LDS with cluster-scope address) to read from the peer CTA's shared memory; L2 path uses ld.global (SASS: LDG.E) hitting in L2 cache. L2-cached reads at 49.5 cyc benefit from the efficient L1→L2 hierarchy — the 1 KB pointer-chase array fits entirely in L2, and the GXBAR provides higher bandwidth than the SM2SM fabric. For unsynchronized single-word reads, L2 is structurally faster because DSMEM's interconnect traversal latency cannot be hidden.

Raw 128B read (L2 wins 4.4x): 128-step pointer-chase through 128B nodes (32 × u32 per node, first word is next-node index). DSMEM path uses generic pointers from cluster.map_shared_rank() (SASS: LD.E routed to DSMEM hardware); L2 path uses inline PTX ld.global.u32. DSMEM costs 659 cyc/step vs L2's 151 cyc/step. The ratio (4.4x) is nearly identical to the 4B case (4.5x), showing the ratio is stable across payload sizes — both paths pipeline the 31 independent loads within each node similarly. DSMEM's scaling from 225→659 cyc (2.9x for 32x data) and L2's from 49.5→151 cyc (3.1x) confirm significant ILP overlap within both pipelines, but the absolute gap remains because the DSMEM interconnect's per-access floor latency is fundamentally higher.

Write+read round-trip (DSMEM wins 2.5x): Full producer-consumer handoff with acknowledgment — CTA 0 writes 128B to its smem, both CTAs synchronize (3× barrier.cluster.arrive.relaxed + wait for data-ready / read-done / ack), CTA 1 reads via DSMEM. The L2 path uses st.global.release.gpu + ld.global.acquire.gpu flag polling. DSMEM costs 1201 cyc: 3 barriers × ~101 cyc = 303 cyc sync + ~225 cyc DSMEM reads + ~225 cyc ack read + smem writes + overhead. L2 costs 2956 cyc: 2× MEMBAR.ALL.GPU × ~300 cyc = 600 cyc fences alone, plus release stores, coherence propagation delay, polling loop iterations, and data reads. The crossover from L2-favored to DSMEM-favored occurs here because the MEMBAR.ALL.GPU (GPU-wide store-queue drain across 110 SMs, ~300 cyc each) dominates the L2 sync path, while barrier.cluster.arrive.relaxed (SASS: UCGABAR_ARV, ~101 cyc) operates only within the cluster scope, avoiding the GPU-wide flush entirely.

Sync notification only (DSMEM wins 26x): Pure synchronization with zero data transfer — repeated barrier ping-pong (DSMEM) vs release/acquire flag polling (L2). This is the most extreme ratio in the matrix. DSMEM costs 86 cyc per sync event: UCGABAR_ARV (~40 cyc) + UCGABAR_WAIT (~46 cyc), using the CGA hardware barrier circuit — a dedicated on-chip synchronization mechanism that does not touch the memory hierarchy at all. L2 costs 2242 cyc: each direction requires st.release.gpu (implying MEMBAR.ALL.GPU ~300 cyc + STG.E.STRONG.GPU) + L2 coherence propagation + ld.acquire.gpu polling (multiple LDG.E.STRONG.GPU iterations until the new value is visible) + reverse ack path. The MEMBAR.ALL.GPU alone at ~300 cyc is already 3.5× the entire DSMEM barrier cost. The .relaxed modifier on the cluster barrier is critical — a full barrier.cluster costs 385 cyc ([S7/case_53]) due to additional MEMBAR/ERRBAR instructions, whereas .relaxed eliminates those.

mbarrier bilateral (DSMEM wins 12.4x): Bilateral ping-pong using mbarrier hardware objects vs acquire/release polling. DSMEM path: mbarrier.arrive.shared::cluster (fire-and-forget via CGA interconnect) + mbarrier.try_wait.parity.shared (efficient local hardware poll). L2 path: st.global.release.gpu + ld.global.acquire.gpu polling loop. DSMEM costs 160 cyc bilateral (~80 cyc/direction), with only 0.7% jitter — the hardware-managed mbarrier is deterministic. L2 costs 1977 cyc bilateral (~988 cyc/direction, consistent with ~1000 ns L2 release/acquire from [S17/case_3]), with 9.2% jitter from the software polling loop's variable L2 latency. The mbarrier is lighter than the full cluster barrier (80 cyc/direction vs 101 cyc for barrier.cluster.relaxed) because try_wait.parity polls on local hardware state without issuing a full barrier instruction.

Data+signal 128B (DSMEM wins 3.5x): Realistic producer-consumer combining 128B data transfer with synchronization — CTA 0 writes 128B to its smem, 2× barrier.cluster.arrive.relaxed + wait (data-ready + completion), CTA 1 reads 128B from peer smem. L2 path: 128B global writes + release/acquire flag polling + data reads + ack. DSMEM costs 952 cyc/iter: sync component 2 × 86 = 172 cyc (18% of total) + data component ~780 cyc (32 DSMEM reads, partially pipelined). L2 costs 3312 cyc/iter: sync component 2× MEMBAR.ALL.GPU × 300 = 600+ cyc (18%+ just for fences, before counting polling and coherence delays) + data reads + stores + ack. Despite DSMEM's raw data reads being slower (780 vs ~400 cyc for L2 data reads), the 4× cheaper synchronization (172 vs 600+ cyc) more than compensates. Compared to Case 28 (3 barriers, 1201 cyc), saving one barrier + ack saves ~250 cyc, consistent with one barrier (86 cyc) plus one DSMEM ack read (~225 cyc).

st.async + mbarrier (DSMEM wins 8.4x): The most advanced DSMEM pattern — producer issues st.async.weak.shared::cluster.mbarrier::complete_tx::bytes (SASS: STAS), a true fire-and-forget async store that writes directly to the consumer's smem and decrements the mbarrier's transaction count upon hardware completion; consumer uses mbarrier.try_wait.parity to detect arrival. DSMEM producer issue: 2 cyc (fire-and-forget, 290× faster than L2's st.release.gpu at 580 cyc). DSMEM consumer wait: 240 cyc (cross-CTA transfer latency ~220 cyc + mbarrier notification overhead). L2 total: 2045 cyc (580 cyc release store + 1465 cyc acquire polling). The st.async pattern decouples producer progress from data transfer entirely — the producer thread continues immediately, and the hardware handles both the data movement and the completion notification. This is the foundation of production DSMEM pipelines (FlashInfer MLA, CUTLASS cluster GEMM). Critical ordering constraint: mbarrier.arrive.expect_tx must be called before st.async fires, otherwise the mbarrier never completes (deadlock).

Decision rule: Use DSMEM when data exchange requires synchronization (producer-consumer, tile sharing, pipeline stages) — the CGA barrier hardware is 12–26× cheaper than GPU-scope MEMBAR-based signaling. Use L2 for unsynchronized cross-SM reads where no ordering guarantee is needed — the GXBAR/L2 hierarchy provides 4.4× lower raw read latency. The crossover point is approximately one synchronization event per data transfer: as soon as even a single sync is included, DSMEM wins.

Tier 1 — Catastrophic (10x+, must avoid): Random DRAM access (86.7x) occurs when pointer-chase or gather patterns defeat the memory controller's bank-level parallelism entirely. Tier 2 — Significant (3–10x, design-level decisions): The L1→L2 memory wall (8.0x) is the dominant hierarchy boundary on SM_120 and the primary motivation for tiling strategies — keeping the working set below 89 KB is the single most impactful optimization for latency-sensitive code. DRAM coalescing (5.1x) and L2 BW amplification (4.17x) reward streaming access patterns that maximize spatial locality.

Tier 3 — Mild (1–3x, tolerable in most contexts): Bank conflict penalties on SM_120 are remarkably small thanks to the decoupled accept-resolve pipeline (§2.1.1) — the worst-case 32-way conflict adds only 1.43x throughput overhead and 2.62x latency overhead. Most remarkably, L2→DRAM latency is only 1.02x — GDDR7's closed-page architecture with 12-channel interleaving makes DRAM latency nearly indistinguishable from L2 for random access patterns.

Optimization implication: On SM_120, the traditional GPU optimization wisdom of "avoid bank conflicts at all costs" is less critical than ensuring coalesced DRAM access and keeping working sets within L1 capacity. The 8.0x L1→L2 cliff is the single most important performance boundary — it exceeds the combined penalty of 32-way bank conflicts (1.43x) and warp scaling (1.88x) by a factor of 3x.

Microarchitectural implication: The SM_120 coalescing unit has a coalescing window of 8 unique cache lines per warp request. Within this window (strides 1-8), all accesses are served in a single transaction with near-identical cost. Beyond this window (stride ≥16), the request is split into multiple rounds, doubling the latency. This is consistent with a coalescing buffer that can track 8 outstanding cache line addresses simultaneously.

2.8.2 The "Memory Wall" Location on SM_120

A key architectural insight emerges from comparing latency ratios across hierarchy levels:

Transition Latency Ratio BW Ratio Bottleneck Character
RF → Smem ∞ (0 → 34 cyc) — Finite vs infinite
Smem → L1 1.29× (34 → 44 cyc) ~0.26× Coherence overhead
L1 → L2 8.0× (44 → 354 cyc) 0.61× THE MEMORY WALL
L2 → DRAM 1.02× (354 → 362 cyc) 0.24× Near-transparent (random)

The SM_120 "memory wall" is at the L1-to-L2 boundary, not at the L2-to-DRAM boundary as on prior architectures. This is architecturally novel: the 96MB L2 with GDDR7's ~362 cyc latency effectively makes DRAM "feel like" an extension of L2 for random access patterns. The 8.0× cliff from L1 to L2 (44 → 354 cyc) is the dominant performance boundary.

Why L2-to-DRAM is nearly transparent: At 1.02× ratio for random pointer-chase, the DRAM latency (~362 cyc) is virtually identical to L2 latency (~354 cyc). This occurs because GDDR7 at 28 Gbps/pin with the memory controller's aggressive scheduling (closed-page auto-precharge, 4-way bank-group interleaving) delivers latency competitive with the on-chip L2 for random access — random walks naturally distribute across banks, avoiding same-bank serialization. The large 96MB L2 serves primarily as a bandwidth amplifier (4.17× over DRAM) rather than a latency reducer.

For streaming workloads: The story changes — L2 provides 5110 GB/s vs DRAM's 1227 GB/s (4.17×). The L2's value is in aggregate bandwidth, not per-access latency. Kernels should be designed to maximize L1 residency (working set < 96KB) as the primary optimization target, with L2 residency (working set < 96MB) as the secondary target.

2.10 Key Insights

KI-2.1: Bank Conflicts Fully Pipelined -- Near-Zero Throughput Penalty

Empirical Evidence: Smem bank conflict (32-way) adds only 1.43x penalty in throughput mode (2.23 to 3.19 cyc/op). Latency mode sees 2.62x (38.3 to 100.3 cyc for pointer-chase). Prior architectures show proportional serialization. [source: S8/case_54,63,119]

Theoretical Analysis: SM_120's shared memory controller pipelines multi-way bank conflicts rather than serializing them. The controller allows concurrent access to non-conflicting banks while replaying conflicting ones in subsequent cycles, hiding the replay latency behind the pipeline.

Microarchitecture Speculation [Confidence: H]: The smem arbiter has been redesigned with a pipelined conflict resolution stage that decomposes N-way conflicts into ceil(N/32) pipeline passes without stalling the issuing warp's throughput path. Patent US9639479 describes a shared memory arbiter with conflict resolution.

Implication for Optimization: On SM_120, smem bank-conflict avoidance is less critical than on prior architectures for throughput-limited kernels. XOR swizzle remains important for latency-sensitive paths (ldmatrix, pointer-chase) where the 2.62x latency penalty still matters.

KI-2.2: L2 XOR Hash Prevents All Systematic Partition Camping

Empirical Evidence: Uniform ~370 cyc/hop across 4KB-2MB strides. No conflict miss up to 256 pointers. Hash-based indexing prevents stride-dependent performance cliffs. [source: S8/case_76, S21/case_27,31]

Theoretical Analysis: XOR of multiple address bits distributes accesses uniformly even for power-of-2 strides. This eliminates the pathological behavior of modular interleaving where stride = MC_count x line_size causes all accesses to hit one partition.

Microarchitecture Speculation [Confidence: H]: NVIDIA moved from modular to XOR-based hashing between Ampere and Blackwell. Patent US10338919 describes memory interleaving with hash-based address distribution.

Update from Section 24: Hash bit 10 is the fundamental granularity of the XOR hash function, and a period-8 structure is confirmed [S24/case_23,24].

Implication for Optimization: Power-of-2 stride access patterns are safe on SM_120. The only remaining concern is stride=64B (single-sector) which creates artificial serialization within a single partition (163 GB/s), but this is an access granularity issue, not a camping issue.

KI-2.3: L2 Thrashing Cliff at 96MB Is Sharp (4.17x BW Drop)

Empirical Evidence: BW transitions from 5110 GB/s to 1227 GB/s at 96-112 MB working set. Persistence policy can protect 60MB of the 96MB. [source: S21/case_26,29]

Theoretical Analysis: Pseudo-LRU eviction cannot protect streaming + reuse patterns simultaneously. cudaAccessPolicyWindow is essential for large working sets that combine streaming reads with reused state.

Microarchitecture Speculation [Confidence: H]: The 96MB capacity with pseudo-LRU creates a sharp boundary because once the working set exceeds capacity, every new line evicts a potentially-needed line with no grace period.

Implication for Optimization: For KV-cache attention (working set often 64-256MB), use persistence hints to pin the reused portion. Monitor L2 hit rate via NCU lts__t_sectors_srcunit_tex_op_read_lookup_hit to detect thrashing.

KI-2.4: Software Prefetch Counterproductive -- HW Prefetcher Sufficient

Empirical Evidence: SW prefetch adds ~55-85% overhead for L2-resident data. No benefit at any lead distance. [source: S8/case_35,102,135]

Theoretical Analysis: SM_120's stream prefetcher aggressively fills ahead of the access stream. Software prefetch instructions compete with real loads for LSU issue slots (2.27 cyc each), worsening throughput without providing any latency benefit.

Microarchitecture Speculation [Confidence: H]: The hardware prefetcher detects straight-line streaming patterns and issues speculative loads into the L1. Adding explicit prefetch creates redundant requests that consume memory bandwidth and issue slots.

Implication for Optimization: Remove all __builtin_prefetch / prefetch.global from SM_120 kernels. Focus instead on ensuring coalesced access patterns that the HW prefetcher can predict.

KI-2.5: Smem 1.29x Faster Than L1 -- Explicit Management Always Wins

Empirical Evidence: Smem latency 33.89 cyc vs L1 44 cyc for identical pointer-chase patterns. L1/smem carveout API is functional with 1:1 byte trade-off from a 100 KB unified SRAM pool; L1 is disabled entirely at 64+ KB smem. [source: S8/case_10,57, S24/case_4]

Theoretical Analysis: Smem has a dedicated low-latency SRAM path; L1 shares tag lookup with the cache coherence protocol, adding ~10 cyc overhead.

Microarchitecture Speculation [Confidence: H]: L1 must perform tag comparison and coherence checks that smem (software-managed, no coherence) avoids. The 10 cyc difference is the coherence protocol overhead.

Implication for Optimization: Always prefer explicit smem staging over relying on L1 cache for reused data. The 1.29x latency advantage compounds across thousands of accesses in GEMM/attention mainloops.

KI-2.6: L1 Fully Associative Eliminates Conflict Misses

Empirical Evidence: Stride=4KB cliff at K matching capacity exactly ([S24/case_6] stride-scan: cliff at N=720 identical for stride=128B and stride=4KB, ruling out any N-way set-associative with N<712). No associativity-induced conflict at any access pattern. L1 effective capacity refined to 89 KB (1 KB precision cliff) [S24/case_1], NCU confirms 88 KB [S24/case_5]. [source: S8/case_74, S24/case_1,5,6]

Theoretical Analysis: Fully associative design is viable for the 89 KB L1. Eliminates pathological stride-based eviction patterns that plague set-associative caches.

Microarchitecture Speculation [Confidence: H]: ~712-way associativity = 89 KB / 128 B = 712 entries (effective; up to 768 entries when full 96 KB physical SRAM is allocated to L1 with smem carveout = 0). Implemented as a CAM (content-addressable memory) or highly-banked tag array. Area cost ~700×38×16T ≈ 425 K transistors — well-precedented in V100/A100/H100 L1 designs and < 0.001% of GB202's ~76 B-transistor per-SM budget.

Implication for Optimization: Stride-access patterns that cause conflict misses on set-associative caches (e.g., stride=N_ways x line_size) are safe on SM_120. No need for padding or offset tricks.

KI-2.7: Atomic Fire-and-Forget (red) 5x Faster Than atom

Empirical Evidence: red.add throughput 5.7-10.3 cyc/op vs atom.add 52.17 cyc/op dependent chain. BUT red.release.sys costs 521-947 cyc (69-126x). [source: S8/case_111,129]

Theoretical Analysis: red issues as fire-and-forget (no SB wait, no return value). The penalty of ordering comes from MEMBAR, not the reduction itself.

Microarchitecture Speculation [Confidence: H]: Reduction operations bypass the scoreboard entirely -- they fire into the L2 atomic unit and the warp continues immediately. Only when ordering guarantees are required does the MEMBAR drain cost manifest. Patent US20230315655A1.

Implication for Optimization: Use red.add (not atom.add) for accumulation where the return value is not needed (e.g., gradient accumulation, histogram). Reserve atom for read-modify-write where the old value is consumed.

KI-2.8: DSMEM Cross-CTA Latency 6.5x Local but Sync Cost Dominates

Empirical Evidence: DSMEM read 219-225 cyc (6.5x local smem 34 cyc). Both CTAs must stay alive -- premature CTA exit crashes DSMEM. [source: S8/case_58, S16/case_3]

Theoretical Analysis: Cross-CTA access traverses the cluster interconnect fabric (SM2SM crossbar) rather than the local SRAM path. The 220 cyc includes: request traversal (~85 cyc) + remote smem access (~25 cyc) + response return (~85 cyc) + pipeline overhead.

Microarchitecture Speculation [Confidence: H]: The DSMEM fabric is physically separate from the L2/GXBAR path. Patent US20230289189A1 Claim 6 and US20250173152A1 Claim 1 describe "SM-to-SM interconnect independent of the cache path."

Implication for Optimization: DSMEM is not a general-purpose fast memory -- it is a synchronization-optimized inter-CTA communication channel. Use it for tile-sharing with barriers (where the 85 cyc relaxed arrive is 26x cheaper than L2 atomic signaling), not for random cross-CTA reads.

KI-2.9: The "Memory Wall" on SM_120 Is at L1-to-L2, Not L2-to-DRAM

Empirical Evidence: L1 hit=44 cyc vs L2 hit=354 cyc gives 8.0x cliff. But L2 hit=354 cyc vs DRAM=362 cyc gives only 1.02x (nearly zero gap for random access). [source: S8/case_1,2,3,114, S26/case_7-9,11]

Theoretical Analysis: GDDR7 at 28 Gbps/pin with 384-bit bus provides 1344 GB/s theoretical -- close enough to L2 service rate that random-access patterns see nearly identical latency (~362 vs ~354 cyc). The 96 MB L2 acts as "warm DRAM" for most working sets.

Microarchitecture Speculation [Confidence: H]: The coalescing curve shows stride-16 at 19.6% -- the bottleneck is L1 sector miss to L2 round-trip, not DRAM latency. For GEMM, working sets typically fit L2 making kernels compute-bound, not DRAM-bound.

Implication for Optimization: The bottleneck classifier should check L1 working set FIRST, then L2 working set. Optimizing DRAM BW is rarely the right target on SM_120. Focus on L1 residency via tiling and smem staging.

KI-2.10: fence.acq_rel Cost Reveals MEMBAR.ALL.GPU as Universal Bottleneck

Empirical Evidence: CTA fence=7 cyc, GPU fence=285 cyc, SYS fence=920 cyc. The 285 cyc GPU fence dominates ALL cross-SM operations: full cluster barrier (371=285+86), st.release.gpu (920=285+STG), grid sync (1242~2x285+672). [source: S8/case_64-66, S16/case_5, S8/case_131]

Theoretical Analysis: MEMBAR.ALL.GPU must: (1) drain ALL pending stores from L1 to L2 via GXBAR, (2) wait for L2 ack from ALL 12 partitions, (3) invalidate L1 cache. This is structurally a GPU-wide broadcast+ack -- O(partitions) not O(1).

Microarchitecture Speculation [Confidence: H]: This is why cluster relaxed arrive (85 cyc) is 4.4x cheaper than full barrier (371 cyc) -- relaxed skips MEMBAR. It is also why DSMEM sync (86 cyc via SM2SM) beats L2 sync (2242 cyc via GXBAR+MEMBAR) by 26x. Patent US20230315655A1 proposes "0.5 RTT" composite write+arrive.

Implication for Optimization: ANY operation triggering MEMBAR.ALL.GPU costs >=285 cyc. Minimize GPU-scope fences by keeping data in CTA-local smem or using relaxed cluster barriers. The 40.7x CTA-to-GPU fence cliff is the single most important latency to avoid in kernel design.

2.12 TLB Hierarchy (Section 26 Results)

Section 26 (cases 15-20) characterized the SM_120 TLB hierarchy via pointer-chase latency sweeps across page counts and page sizes.

Parameter Value Source
L1 TLB capacity (4 KB pages) 712 entries, covering 2.85 MB [S26/case_15-16]
L1 TLB capacity (2 MB pages) >= 64 entries (lower bound; capacity not fully exhausted) [S26/case_17]
TLB miss penalty 327 cyc (fully hidden within L2 access latency of ~371 cyc) [S26/case_15-16]
TLB hierarchy Single-level (no L2 TLB detected up to 4096 pages / 16 MB) [S26/case_18]
Replacement policy Set-associative; gradual cliff from N=712 to N=1008 pages (not a sharp step) [S26/case_16]
4 KB vs 2 MB pages (WS=32MB) 8192 TLB entries vs 16 entries; 8.46x latency ratio (reflects L1-vs-L2 cache level, not TLB penalty per se) [S26/case_19-20]

Key findings:

  • Single-level TLB: No L2 TLB was detected. Probing up to 4096 pages (16 MB) showed no intermediate latency step between the L1 TLB hit and the L2 cache miss latency. The SM_120 TLB is a single-level structure with ~712 entries for 4 KB pages.

  • TLB misses are invisible as a separate bottleneck: The 327 cyc TLB walk penalty is fully subsumed by the L2/DRAM access latency (~371 cyc). On SM120, a TLB miss does not create an _additional penalty beyond the cache miss that typically accompanies it — the page-table walk completes within the time the memory subsystem already spends servicing the L2 access.

  • Set-associative replacement: The capacity cliff is gradual (N=712 to N=1008), not a sharp step at exactly 712 entries. This is consistent with a set-associative TLB where some sets overflow before others, producing a probabilistic eviction onset rather than a hard boundary.

  • Practical implication: TLB capacity is not a performance concern for typical GPU workloads on SM_120. The 712-entry TLB covers 2.85 MB with 4 KB pages, and huge pages (2 MB) extend coverage to >= 128 MB. For workloads with extremely sparse access across many pages, the TLB miss penalty is masked by the L2 access latency.

2.13 Summary

The SM_120 memory subsystem is characterized by three defining features: (1) a fully-pipelined shared memory controller that makes bank-conflict avoidance largely unnecessary for throughput-limited code, (2) a 96MB XOR-hashed L2 that eliminates partition camping and creates the primary performance boundary at the L1-to-L2 transition (8.0x), and (3) a dual-fabric on-chip network (SM2SM for DSMEM, GXBAR for L2) where the DSMEM path provides 26x cheaper synchronization despite higher raw read latency. The MEMBAR.ALL.GPU operation (285 cyc) is the universal bottleneck for all cross-SM communication, making CTA-scope operations (7 cyc) the strongly preferred default. The TLB is a single-level structure (712 entries for 4 KB pages) whose miss penalty (327 cyc) is fully hidden within L2 access latency.
