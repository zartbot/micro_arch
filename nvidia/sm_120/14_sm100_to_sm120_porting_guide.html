CHAPTER 14 · SM_120 · GB202 BLACKWELL

SM_100 → SM_120 Porting Guide

This chapter provides a systematic guide for porting GPU kernels from SM_100 (B200, data-center Blackwell with Tensor Memory and wgmma) to SM_120 (GB202, RTX PRO 5000 72GB without Tensor Memory). The SM_120 shares the same Blackwell generation but lacks key features present on SM_100: tcgen05/TMEM, wgmma, TMA multicast, 2-SM cooperative execution, and `cvt.rs` stochastic rounding. These are not incremental differences but fundamental architectural divergences that require full mainloop restructuring. All performance impact figures are measured on the RTX PRO 5000 72GB and traced to specific microbenchmark test cases.

14.3.2 Instruction Count Explosion

The change from wgmma to mma.sync fundamentally increases instruction count for the same computation:

Operation SM_100 (wgmma) SM_120 (mma.sync) Ratio
Instructions per m128n128k16 tile 4 wgmma 128 mma.sync 32x
Operand loads per tile 0 (descriptor feed) 128 ldmatrix --
Throughput per SM ~wgmma rate 15.88 cyc/mma at ILP>=4 --
Achieved TFLOPS/SM 28.1 [SPECULATIVE] 29.3 1.04x (SM_120 wins)

[SPECULATIVE] note on 28.1 TFLOPS/SM for SM_100: SM_100 hardware was not available for direct measurement in this campaign — all S18 cross-generation cases ([S18/case_2,4,7]) report ⚠️ Anomaly/Unsupported. The 28.1 TFLOPS/SM value is derived from NVIDIA-published B200 specs (132 SMs, ~3712 TFLOPS FP16-tensor at 1.965 GHz boost ⇒ 28.1 TFLOPS/SM = 3712 / 132); the 1.04× SM_120 advantage in the row above relies on this derivation, not on first-party measurement. Treat all SM_100 numerics in this chapter as [SPECULATIVE] unless explicitly cited to an S18 PASS case.

Despite the 32x instruction count increase, the SM_120 achieves slightly higher per-SM TFLOPS due to its higher clock frequency (2377 vs 2250 MHz baseline on B200). However, B200 has more SMs, yielding higher chip-level throughput.

14.4 Porting Case Studies

14.4.1 CUTLASS: sm100_gemm_tma_warpspecialized to sm120_gemm_tma_cooperative

Porting steps (not incremental; full mainloop restructure required):

  1. Replace wgmma with mma.sync: Restructure the entire mainloop from descriptor-based wgmma to register-based mma.sync. Changes tile shape from m64nXk to m16n8k, requiring 4x more loop iterations.

  2. Replace TM descriptor feed with ldmatrix + register staging: Insert explicit ldmatrix.sync.aligned.x4 instructions (49 cyc latency, 1.68 cyc/op throughput) before each mma.sync. Use XOR-swizzle (128B) to eliminate bank conflicts [S8/case_142: 4.23x benefit].

  3. Remove multicast; use ClusterShape<1,1,1>: TMA multicast is non-functional on SM_120. Set cluster shape to unity. If A-operand sharing is needed, use unicast + DSMEM redistribution (220 cyc per remote read) [S16/case_3].

  4. Remove warp specialization; switch to all-compute: Warp specialization adds 259 cyc/iter overhead with no TM bypass benefit. All-compute is 2.5-3.1x faster for TC-bound kernels [S9/case_33].

  5. Adjust tile size: Optimal tiles are 256x128x128 (MX-FP4) or 128x128xK (FP16), constrained by 99 KB smem [S9/case_136].

  6. Set pipeline depth to 3: Three-stage pipeline provides 1.92x speedup over serial; fourth stage adds only 5.5% [S10/case_82].

Verification: CUTLASS MX-FP4 on SM_120 achieves 1076.5 TFLOPS = 100.6% theoretical efficiency [S9/case_136].

Critical rule: Always prefer CTA-scope fences (7 cyc) over GPU-scope (285 cyc) when data is CTA-local. This provides 40.7x savings [S8/case_64-65]. On SM_120, the sys-scope penalty is far larger than on NVLink-equipped SM_100 due to the PCIe interconnect.

14.8 Multi-GPU Communication Porting

The SM_120 (RTX PRO 5000) uses PCIe 5.0 x16 interconnect, unlike SM_100 (B200) which has NVLink. This fundamentally changes multi-GPU kernel design:

Communication Pattern SM_100 (NVLink) SM_120 (PCIe) Recommended Approach
AllReduce ~900 GB/s 35.2 GB/s (NCCL) Use NCCL; avoid custom [S17/case_14]
Point-to-point ~900 GB/s 53 GB/s (raw P2P memcpy) P2P preferred over collective [S15/case_1]
AllToAll (MoE) NVLink-bounded 53 GB/s via raw P2P MoE expert parallelism viable [S17/case_16]
Cross-NUMA P2P N/A 41 GB/s (24% penalty) Pin experts to same NUMA node
Custom AllReduce Beneficial 2.4 GB/s (14x worse than NCCL) Never use custom AllReduce [S17/case_15]

Porting implication: Replace tensor parallelism (AllReduce-heavy) with pipeline parallelism or expert parallelism (point-to-point routing). The inter-stage latency for pipeline parallelism is 544 ns (min), 1376 ns (median) via griddepcontrol [S17/case_9].

Porting Checklist Summary

  1. Remove all wgmma/tcgen05 instructions; replace with mma.sync.aligned.
  2. Replace TM operand feed with ldmatrix.sync.aligned.x4 + XOR-swizzle.
  3. Remove multicast; set ClusterShape<1,1,1> or use unicast+DSMEM.
  4. Switch from warp specialization to all-compute paradigm.
  5. Limit pipeline depth to 3 stages (99 KB smem constraint).
  6. Replace cvt.rs with cvt.rn; handle stochastic rounding in software if needed.
  7. Add -gencode arch=compute_120a if using setmaxnreg.
  8. Replace custom AllReduce with NCCL; prefer P2P for MoE routing.
  9. Replace mbarrier-based intra-CTA sync with named barriers (14x cheaper).
  10. Minimize GPU-scope and sys-scope fences; prefer CTA-scope (40.7x cheaper).
