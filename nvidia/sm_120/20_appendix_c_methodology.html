Appendix C — Measurement Methodology

This appendix documents the measurement framework, timing methodology, known artifact sources, mitigation strategies, and per-chapter data confidence levels for the SM_120 microbenchmark campaign. All measurements follow the methodology established in Section 1 (Methodology Calibration) and validated across 1504 test cases. Understanding the measurement infrastructure is essential for interpreting results and assessing confidence bounds.

C.1 Measurement Framework

Timing Infrastructure

Component Implementation Notes
Clock source %clock64 register (64-bit cycle counter) Per-SM, monotonic
Start read CS2UR (Clock-to-Uniform-Register) Minimizes pipeline perturbation
End read CS2R (Clock-to-Standard-Register) Standard RF path
Overhead 3 cycles (empty-region baseline) Subtracted from all measurements
Resolution 1 cycle at GPU clock frequency ~0.39 ns at 2572 MHz boost

Source: section_1/01_clock64_overhead_calibration_260606_014930

Dual-Path Clock Read (SM_120 Specific)

SM_120 supports two clock read paths with different pipeline insertion points: 

CS2UR %clock64, %ur_start    // Uniform RF — lower perturbation for start
... measured region ...
CS2R  %clock64, %r_end       // Standard RF — conventional for end

This dual-path approach minimizes measurement overhead to exactly 3 cycles, verified against known-latency instruction sequences.

Iteration Protocol

Parameter Value Rationale
Warmup iterations 100+ Eliminates I-cache cold misses, TLB fills, JIT effects

Reported statistics Median, Mean, Stddev Median robust to outliers; mean for averaging
Outlier handling None (report raw) Outliers may indicate real microarchitectural events

Warp Control Strategy

Mode Configuration Use Case

Anti-Dead-Code-Elimination (Anti-DCE)

Problem: ptxas aggressively eliminates code with no observable side effects, even through asm volatile barriers.

Solution: Global memory store sink pattern:

// Anti-DCE: result stored to global memory
__global__ void kernel(uint32_t* sink, ...) {
    uint32_t result = 0;
    // ... measurement region ...
    sink[threadIdx.x] = result;  // Prevents DCE of entire chain
}

Verification: asm volatile alone is insufficient on SM*120 — ptxas still eliminates chains if the output register has no downstream consumer reaching a store. [Source: section_21/21.2_15_volatile_vs_asm_260608*\*]

Clock Pinning

# Pin GPU clock to boost frequency (eliminates DVFS drift)
nvidia-smi -lgc 2572,2572

# Verification: <0.3% drift over 60-second measurement windows

Source: section_3/08_clock_frequency_stability_260606_104041

SASS Verification

Every test case includes pre-measurement SASS verification:

# Extract SASS to verify instruction count and structure
cuobjdump --dump-sass <binary> | grep -c "<target_instruction>"

This ensures:

  1. The compiler did not eliminate the measured instruction chain
  2. No unexpected instructions were inserted (spills, moves, etc.)
  3. The dependency structure matches the intended measurement

Artifact Impact Mitigation Residual Uncertainty
ptxas reordering (max distance=20 instructions) Alters dependency chain structure; may break intended dep chains Inline asm + volatile + global store sink +/- 1-2 cycles
ptxas constant folding Eliminates entire computation chains when inputs are compile-time constants Pass all values as kernel arguments (runtime values) Eliminated
DVFS frequency drift Frequency changes during long measurement windows invalidate cycle counts nvidia-smi -lgc clock pinning; verified <0.3% drift <0.3%
%clock64 pipeline overhead Clock read consumes a pipeline slot and introduces 3-cycle measurement overhead Empty-region subtraction (measured 3-cycle baseline) +/- 2 cycles
L1 I-cache cold miss First execution of a code path incurs I-cache fill latency 100+ warmup iterations before measurement window Eliminated
L1 D-cache warmup First data access incurs cold miss 100+ warmup iterations with same access pattern Eliminated
Dead Code Elimination (DCE) Compiler removes computations with no visible side effect Global store sink (st.global.b32) on all computed values Eliminated
Scoreboard stall noise Scoreboard wait instructions may overlap with prior operation latency Long dependency chain amortization (100+ ops in chain) +/- 1 cycle
Single-warp measurement bias Single-warp tests cannot trigger throughput bottlenecks or inter-warp contention Separate latency tests (single-warp) and throughput tests (multi-warp) N/A (by design)
Register bank conflict Operand collector conflicts could add latency Operand collector masks conflicts (measured 0-cycle penalty) Eliminated on SM_120
TLB miss Large working sets may trigger TLB misses during measurement Pre-touch all pages during warmup; use huge pages where possible <1 cycle (rare)
Instruction cache aliasing Large code footprints may evict measured code from I-cache Keep measurement kernels compact (<1 KB instruction footprint) Eliminated

C.3 ptxas Compiler Behavior Catalog

The following ptxas behaviors were identified and characterized during the campaign:

Behavior Description Impact on Measurement Mitigation
Instruction reordering Moves independent instructions up to 20 slots from source position Can break intended dependency chains Verify SASS matches intent
Constant folding Eliminates chains with compile-time-known inputs Removes entire measurement region Runtime kernel arguments
Dead code elimination Removes code with no side effects Removes measurement target Global store sink
Register coalescing Merges register moves into consumer instructions May alter measured latency Verify via SASS inspection
Stall count assignment Assigns stall counts per instruction based on dependency analysis Generally near-optimal (0% overhead measured) Trust ptxas for production code
Loop unrolling Unrolls small loops May change instruction mix Explicit #pragma unroll control
Predication insertion Converts short branches to predicated code Alters control flow measurement Use asm goto for explicit branches
spill/fill insertion Adds memory operations when register pressure exceeds threshold Contaminates ALU latency measurements Keep register usage below spill threshold

C.7 Limitations and Scope

What This Methodology Can Measure

  • Instruction latency (cycle-precise for single-issue operations)
  • Pipeline throughput (operations per cycle at saturation)
  • Memory hierarchy latency and bandwidth
  • Synchronization costs
  • Feature presence/absence
  • Power consumption (at nvidia-smi sampling granularity)

What This Methodology Cannot Measure

  • Sub-cycle timing (e.g., intra-pipeline-stage details)
  • Internal microarchitectural state (e.g., reservation station occupancy)
  • Hardware event counters not exposed by NCU
  • Die-level physical layout or routing
  • Fuse configuration or disabled-but-present hardware
  • Exact transistor-level implementation of functional units

Measurement Uncertainty Budget

Source Contribution Cumulative
Clock read overhead (subtracted) +/- 0.5 cyc 0.5 cyc
Scoreboard noise +/- 0.5 cyc 1.0 cyc
DVFS residual +/- 0.3% 1.0 cyc + 0.3%
ptxas instruction insertion 0 (SASS verified) 1.0 cyc + 0.3%
Statistical noise (1000 iters) <0.1 cyc ~1 cyc + 0.3%

Total measurement uncertainty: +/- 1-2 cycles for latency measurements; +/- 0.5% for throughput measurements at saturation.

C.8 Notation Conventions

The following typography rules are applied throughout the report (Chapters 0–20) to keep numeric units, identifiers, and citations machine-parseable.

C.8.1 Cycle Unit

Token When to use Example
cyc (lowercase abbreviation) All inline numeric measurements, tables, plots, and machine-readable JSON 44 cyc, latency = 29 cyc, cyc/op = 4.0
cycle / cycles (lowercase noun) English prose where the word reads as a noun "the cycle counter", "600 cycles below the threshold"
Cycle / Cycles (Title Case) Only as a column header in a table or as the first word of a section heading \| ... \| Cycles \| ... \|, "### Cycle Budgeting"
Cyc (Title Case abbreviation) Only as a compact column header (e.g. Cyc/Op); never inline \| Min Cyc \| Cyc/Op \|

Forbidden: ALL-CAPS CYC/CYCLE; mixed forms like Cycle/op (use cyc/op); attaching s to the abbreviation (use cyc, never cycs).

C.8.1.1 Rounding Convention for Cycle Counts

Measured cycle values are reported using the following rounding rule:

Precision regime Rule Examples
Inline narrative / Quick Reference / KI titles Round to nearest integer when the fractional part is ≤ 0.5% of the integer value (e.g. 63.99 → 64; 28.95 → 29; 33.89 → 34) "DFMA = 64 cyc", "shfl = 29 cyc", "smem = 34 cyc"
Detailed measurement tables / S§ raw-data tables / AppB Preserve 2 decimal places when fractional precision matters for downstream models (e.g. throughput formulas, ILP saturation curves) 63.99 cyc, 28.95 cyc, 33.89 cyc
Chapter narrative when cross-referencing both Cite the integer form first, then disclose the precise value parenthetically "DFMA latency 64 cyc (precisely 63.99 cyc, [S5/case_48])"

The two forms are not in conflict; the choice depends on whether the consumer is a human reader (integer is clearer) or a performance model (fractional preserves R²). Cross-chapter discrepancies that fall within the rounding band are not bugs.

C.10 Presentation Conventions (S-01 / S-02 RESOLVED 2026-06-11)

This section captures cross-cutting presentation rules for cross-references, table numbering, and confidence tagging — acknowledging known structural inconsistencies in the v1 draft and codifying the target convention. Full reformatting of all 13,400+ lines of narrative is deferred as a separate refactor task; new content from 2026-06-11 onward follows these rules.

C.10.1 Cross-Reference Hyperlinks (S-01)

Current state: Most section/chapter cross-references in the narrative are plain text (e.g. "see §5.2.5") rather than Markdown hyperlinks.

Target convention (applied to all new content):

  • In-chapter section refs: [§N.M.K](#section-anchor) Markdown anchor link to chapter heading.
  • Cross-chapter refs: [Ch{N} §N.M.K](file:///…/N_chapter_title.md#section-anchor) absolute file URL link.
  • KI/KF/OQ refs: [KI-{chapter}.{seq}](#ki-N-M), [KF-{nn}](file:///…/17_cross_reference_map.md#kf-NN).
  • Plain text refs are tolerated in legacy content but flagged for refactor in final_reports/comments/comment_v1.md S-01.

Rationale: Qoder/VS Code Markdown previewer + GitHub renderer both support clickable anchor links. Preserving plain-text refs in legacy chapters avoids the 13,400-line cost of mass conversion while still providing modern UX for new readers.

C.10.2 Table Numbering (S-01)

Current state: Mixed conventions — "Table 1.1" (chapter.seq), "Table 1" (sequential), unnumbered captions, and ad-hoc bold labels ("Table 8.8").

Target convention: **Table {chapter}.{seq}: {Title}** zero-padded sequence within chapter (e.g. Table 8.8: Multi-Precision Roofline Parameters). Inline-rendered tables without a caption are tolerated for short auxiliary tables (≤5 rows). Anchored refs use Table-{chapter}-{seq} slug.

C.10.3 Identifier Unification (S-01)

Current state: KI, KF, OQ coexist with different scopes (see §C.9). Some readers find the per-chapter KI numbering confusing when a chapter is renumbered.

Decision: KI / KF / OQ remain as separate identifier systems with distinct semantics (per §C.9). They are not unified, because each serves a different purpose:

  • KI = chapter-internal narrative aggregation (re-numbered with chapter revisions)
  • KF = global, immutable atomic facts (master registry in Ch17)
  • OQ = hardware questions with resolution status (Ch16 catalog)

A forced unification would lose the per-chapter mnemonic value of KI-9.5 ("§9 insight 5") and break Ch17 KF-NN stability. The mapping documented in §C.9.2 is sufficient.

C.10.4 Confidence Tag Standardization (S-02)

Current state: [SPECULATIVE] (binary) and [Confidence: M/H/L] (graded) coexist; some claims have one, some both, some neither.

Target convention:

  • All speculative claims MUST carry both tags: [SPECULATIVE, confidence: M] (or H/L).
  • [SPECULATIVE] alone is shorthand for [SPECULATIVE, confidence: M] and should be expanded inline whenever the claim is the table cell anchor (not just narrative).
  • Confidence levels:
  • H (High): published vendor docs / first-party measurement extrapolation with strong physical grounding (e.g. SM_100 28.1 TFLOPS/SM derived from 3712 / 132).
  • M (Medium): cross-generation extrapolation, qualitative model, single-source measurement.
  • L (Low): open speculation, no measurement support, contradicts at least one observation.

Examples corrected in v1:

  • Ch1 §1.1.1 ASCII diagram "~120 SM slots" → [SPECULATIVE, confidence: M; see §1.1.2 harvesting model] (m-19 RESOLVED 2026-06-11)
  • Ch14 §14.4.2 NVLink barrier ~200 cyc / ~150 cyc rows → [SPECULATIVE] with footnote citing B200 docs as derivation source (m-17 RESOLVED 2026-06-11; confidence: M).
