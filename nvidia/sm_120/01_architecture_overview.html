CHAPTER 01 · SM_120 · GB202 BLACKWELL

Architecture Overview

This chapter establishes the complete architectural identity of the NVIDIA SM_120 (GB202, Blackwell) microarchitecture as implemented in the RTX PRO 5000 72GB. Through systematic microbenchmark probing (Sections 0-3), we characterize the chip topology (110 SMs across 11 GPCs), SM internal microstructure (4-SMSP single-issue, 6 independent pipelines), memory hierarchy (L1 through GDDR7 DRAM with hash-interleaved 12-MC subsystem), and the complete tensor core instruction set (mma.sync-only, no wgmma/tcgen05). We provide a detailed feature comparison with SM_100 (datacenter Blackwell) and identify six key architectural insights with microarchitecture-level analysis and optimization implications.

1.1.2 Topology Methodology and Confidence

The GPC topology was determined through multiple independent probes:

  1. TPC pairing (high confidence): All 55 TPC pairs at SM IDs (2k, 2k+1) confirmed via globaltimer coherence measurements showing <=3 cycle delta between paired SMs. [S3/case_5]

  2. Atomic contention ANOVA (medium confidence): Bimodal L2 affinity pattern (F=4.55, p<0.01) when grouping SMs into contiguous 10-SM clusters, consistent with 11 GPCs of 10 SMs each. [S3/case_5]

  3. CTA distribution (supporting): Stride-20 round-robin CTA scheduling pattern observed, consistent with GPC-level dispatch. [S3/case_5]

  4. L2 crossbar uniformity (constraint): L2 ping-pong measurements show a continuous gradient (674-833 cycles, 24% spread) with no bimodal gaps, proving the L2 crossbar masks physical GPC boundaries from the software perspective. This means GPC placement is architecturally transparent for memory access patterns. [S3/case_5]

Harvesting model [SPECULATIVE, confidence: M]: The full GB202 die likely contains 120 SM slots (6 full GPCs of 20 SMs each, or 12 GPCs of 10 SMs each). The RTX PRO 5000 activates 110 SMs by disabling 10 slots (8.3% harvest rate). The dense, contiguous virtual SM ID space [0-109] with %nsmid = 110 confirms driver-level ID compaction after harvesting -- no gaps exist in the virtual address space regardless of physical defect locations.

1.2.2 Pipeline Map

The SM_120 execution backend consists of 6 architecturally independent pipes, confirmed via co-issue overlap testing where all cross-pipe pairs achieve overlap ratios <= 1.16x (1.0x = perfect independence).

The diagram above illustrates the complete per-SMSP (Sub-Core) datapath architecture of SM_120. Each of the four identical SMSPs within an SM operates as a fully independent single-issue processor core. The pipeline begins at the Warp Pool, which holds 12 statically-bound warps (assigned via warpid%4 at block launch). The CGGTY Scheduler (Compiler Guided, Greedy, Then Youngest) selects the next eligible warp using compiler-inserted priority hints, with a 4-entry I-buffer providing limited lookahead. Instructions are decoded from their 128-bit fixed-width encoding, where the upper 64 bits carry scheduling metadata (4-bit stall counter, yield hint, scoreboard barrier IDs, and 4-bit RFC reuse flags) and the lower 64 bits encode the operation and operands. The Register File uses a 2-bank even/odd parity architecture (16,384 registers per SMSP, 2 read ports per bank) backed by a deep Register File Cache (≥24 entries/warp, LRU eviction) that eliminates bank-conflict penalties for all normally-compiled code. The Operand Collector performs 3-source reads per cycle; ptxas ensures mixed-parity register allocation so that the theoretical +1 cycle same-bank penalty never manifests in practice. After single-issue dispatch (1 instruction/cycle), the instruction is routed to one of six independent execution pipe backends: FMA-lite (FP32/FP16×2/BF16×2, 4-cycle latency, 1.08 cyc/op throughput), FMA-heavy (INT32/logic/shift, 2–5 cycles, IMAD achieves 2.0 ops/cyc via dual sub-pipe routing), XU/SFU (transcendentals, 18–44 cycles in three tiers with 99.96% overlap against FMA), LSU (load/store/atomic with 44-cycle L1 hit latency and a remarkably deep 12–13 entry MSHR queue enabling high memory-level parallelism), FP64 (64-cycle DFMA, also hosting DMMA at 579 cycles), and Tensor Core (29-cycle HMMA/QMMA/OMMA with 15.88 cyc/mma throughput, where MX block-scale adds zero overhead). The hybrid dependence tracking system combines compiler-trusted stall counters for fixed-latency ALU operations with 6 hardware scoreboard counters per warp for variable-latency memory operations, achieving area-efficient correctness without a full hardware scoreboard. All six pipe backends operate in parallel after dispatch, enabling rich co-issue: when the Tensor Core dominates (29-cycle MMA window), up to 27 non-MMA instructions from other pipes execute effectively for free.

FP32 + INT32 co-issue: The FMA pipe contains two subpipes (fmalite for FP32, fmaheavy for INT32/logic) that execute independently. Co-issue testing shows 1.01x overlap ratio (FFMA + IMAD), confirming concurrent FP32 + INT32 execution within the same SMSP clock cycle. [S1/case_4]

MUFU decomposition: XU (SFU) instructions internally generate fmalite micro-ops for range reduction and polynomial refinement, causing 5-16% interference when co-issued with FMA-bound workloads. This is a resource contention artifact, not a shared-pipe issue. [S1/case_4]

XU latency tier breakdown (see Ch5 §5.2.5): bare MUFU per op differs by ~2.4× across operations: Tier 1 = 40–44 cyc (RCP/RSQ/LG2 PTX-level with refinement wrappers); Tier 2 = 18–22 cyc (SIN bare 18 / COS 22 / TANH 18); Tier 3 = 18 cyc (EX2/LG2 bare). The single "44" in the pipe table reflects only the Tier-1 (worst-case PTX-level rcp); inline-throughput tools should use the per-operation breakdown.

LSU 40 vs 44 cyc note: The earlier 40 cyc figure came from [S1/case_4] pipeline-identification round-trip (which includes ld + dependent insn issue setup); the canonical L1-hit pointer-chase latency is 44 cyc [S26/case_7-9, S8/case_1] and matches AppB §B.6/B.9. Performance models should use 44 cyc.

1.3.3 Critical SM_120 Limitations

TMA Multicast Hardware Absent [confidence: H]: All TMA multicast operations on SM_120 are software-emulated via CALL.ABS.NOINC (function call to a runtime stub), resulting in 2000-8000x latency penalty compared to unicast TMA. This eliminates the primary operand-sharing mechanism for cluster-cooperative GEMM kernels. As a consequence, CUTLASS enforces ClusterShape<1,1,1> on SM_120, foregoing inter-CTA A-tile sharing entirely. Each CTA must independently load its operands from L2/DRAM, increasing memory traffic by a factor proportional to the cluster size that would otherwise have been used. [S10/case_28-32]

No wgmma/tcgen05: The absence of warpgroup MMA and Tensor Memory means SM_120 kernels are limited to 32-thread warp-level mma.sync operations with explicit shared memory staging. The SM_100 path (wgmma with register-to-shared async pipeline, 128-thread warpgroups) achieves higher arithmetic intensity per memory transaction and reduces synchronization overhead for large GEMMs. This is a fundamental architectural boundary, not a variant-gating decision -- all SM_120 targets (120, 120a, 120f, 121, 121a, 121f) uniformly reject wgmma and tcgen05. [S0/case_10, S2/case_2.1-3, S2/case_2.1-12]

1.6 Key Insights

KI-1.1: TMA Multicast Hardware Absent -- Defining SM_120 Limitation

Empirical Evidence: All TMA multicast operations on SM_120 are intercepted by a software stub (SASS: CALL.ABS.NOINC), with measured latency of ~4,300 cycles/byte -- a 2000-8000x penalty over unicast TMA (~2-4 cycles/byte). This penalty scales linearly with transfer size and is independent of cluster configuration. CUTLASS unconditionally sets ClusterShape<1,1,1> on SM_120 to avoid this path entirely. [S10/case_28-32, S16/case_16]

Theoretical Analysis: TMA multicast enables a single DRAM read to be broadcast to multiple CTAs' shared memory simultaneously within a cluster, reducing DRAM bandwidth consumption by a factor of N for N-way cluster sharing. Without it, each CTA in a cluster must independently load its A-operand tile, multiplying effective memory traffic by the cluster size. For a typical GEMM with ClusterShape<2,1,1>, this doubles A-tile bandwidth consumption.

Microarchitecture Speculation [Confidence: H]: The GB202 die omits the physical multicast steering logic within the TMA engine. This hardware, described in patent US20230289190A1 (Fig. 7), routes TMA responses to multiple destination shared memory partitions via a crossbar within the TMA unit. Its absence on GB202 is a deliberate die-area optimization for the consumer/workstation product -- the DSMEM unicast crossbar (which IS present, per US20250173152A1 section 6) handles point-to-point cross-CTA communication, but lacks the broadcast capability. The software emulation path (CALL.ABS.NOINC) implements multicast semantically by issuing N sequential unicast copies, explaining the linear cost scaling.

Implication for Optimization: Kernels targeting SM_120 should never use cluster sizes > 1 for GEMMs that rely on operand sharing via multicast. The optimal strategy is ClusterShape<1,1,1> with maximal per-CTA tile sizes to amortize the unavoidable per-CTA DRAM loads. For attention kernels where K-splitting uses clusters, SM_120 should use alternative parallelization strategies (e.g., split-K with separate reduction kernels rather than cluster-cooperative reduction). DSMEM unicast remains functional and efficient for point-to-point communication patterns (e.g., pipeline handoff between producer/consumer CTAs).

KI-1.2: Block-Scale (MX) MMA is Zero-Cost Hardware

Empirical Evidence: QMMA.SF (block-scale MMA with ue8m0 scale factors) has identical latency (28.78 cycles) and near-identical throughput (15.59 cyc/mma per [S9/case_91], vs 15.88 cyc/mma for dense QMMA per [S9/case_48-63]) to dense QMMA without scale factors. The 0.29 cyc gap (1.8% faster for QMMA.SF) is well within measurement noise and strengthens the zero-overhead conclusion: scale-factor handling adds no measurable cost. Scale factor register pressure is zero (scale registers are separate from data operands), alignment requirements are identical, and changing scale granularity from 16 to 128 elements produces no measurable timing difference. [S9/case_48-63 (dense), S9/case_91 (.SF); AppB §B.9 records 15.59 as the .SF-specific throughput]

Theoretical Analysis: The MX (Microscaling) format applies per-block exponent scale factors during the MMA accumulation step. If this scaling were implemented as a separate multiply-accumulate stage after the dot product, it would add pipeline stages and increase latency. The zero-cost observation implies the scale factor is applied inline within the reduction tree -- each partial product is multiplied by the shared exponent before summation, requiring only a shift operation (since ue8m0 is a pure exponent) that can be folded into the existing mantissa alignment logic.

Microarchitecture Speculation [Confidence: M]: The tensor core reduction tree includes dedicated scale-broadcast hardware (described in US12141082) that receives the ue8m0 scale values on a sideband path and applies them via barrel-shift during the accumulation phase. This adds no pipeline latency because the shift occurs in parallel with the carry-propagation step that already exists in the FP32 accumulator. The .SF suffix in the SASS opcode (QMMA.SF) activates this sideband path; without it (plain QMMA), the shift hardware is bypassed (implicit scale = 1.0 = zero shift).

Implication for Optimization: MX-format quantization (MXFP8, MXFP6, MXFP4) is free at the compute level on SM_120. The only costs are: (1) additional register space for scale factor operands in the PTX instruction (6 extra operands per MMA: scaleA, bidA, tidA, scaleB, bidB, tidB), and (2) slightly more complex shared memory staging to co-locate scales with data tiles. There is no reason to avoid MX formats for latency-sensitive inference workloads. The zero-cost nature makes MX-quantized models strictly preferable to dense formats at equivalent accuracy, since they achieve higher effective compute density (more elements per MMA at the same cycle cost).

KI-1.3: Strict 4-SMSP Single-Issue Architecture

Empirical Evidence: NCU performance counters show only smsp__issue_inst0 and smsp__issue_inst1 exist (no inst2 or higher). Per-SMSP IPC ceiling measured at 0.92 (theoretical 1.0, gap due to scoreboard stalls). The 4-SMSP aggregate IPC is 3.66 (99.1% of ideal 4.0 when accounting for the per-SMSP ceiling). The IPC plateau occurs at ILP=2 within each SMSP, consistent with a pipeline depth of 2 cycles for the dispatch-to-execute path. [S22/case_1-4, S3/case_9]

Theoretical Analysis: A single-issue scheduler can dispatch at most one instruction per cycle per SMSP. With 4 SMSPs, the theoretical SM-wide IPC is 4.0. Each issued instruction is routed to one of the pipe backends (FMA, XU, LSU, TC, FP64, Uniform). Multiple backends can be active simultaneously (since they are independent), but only one new instruction enters the pipe each cycle per SMSP. The ILP=2 saturation indicates a 2-stage front-end (decode + dispatch or fetch + decode) before the instruction reaches the execution unit.

Microarchitecture Speculation [Confidence: H]: The SM_120 SMSP is architecturally identical to the Hopper (SM_90) sub-core in terms of issue width. Patent US11768686 section 5.1 describes the "sub-core warp scheduler with single-issue dispatch" that has been the NVIDIA standard since Volta (SM_70). The smsp__issue_inst0 and smsp__issue_inst1 counters correspond to the decode slot (inst0) and the dispatch slot (inst1) -- not to dual-issue capability. The per-cycle flow is: select eligible warp -> read I-buffer entry -> decode -> dispatch to pipe backend.

Implication for Optimization: Instruction-level parallelism within a single warp provides no throughput benefit beyond ILP=2 (needed to fill the 2-stage pipeline front-end). Instead, throughput scaling comes exclusively from warp-level parallelism across the 4 SMSPs. This means: (1) unrolling loops beyond the point needed to expose 2 independent instructions is wasteful; (2) throughput-critical kernels should maximize active warps rather than per-warp ILP; (3) latency-critical single-warp kernels cannot exceed 1 useful instruction/cycle regardless of code scheduling. For Chapter 4-6 (instruction latency/throughput), this confirms the measurement model: dependent chains measure true latency, while 4+ concurrent warps measure throughput.

KI-1.4: GPC Topology -- Uniform L2 Hash Eliminates Partition Camping

Empirical Evidence: L2 cache access uses XOR-based hash interleaving across 12 partitions. A stride sweep from 64B to 16KB shows flat bandwidth (1056-1121 GB/s, CV ~2%) across the entire 128B-2048B range where modular interleaving would produce catastrophic drops. Specifically, stride=1536B (which equals 128B x 12, the modular-interleave alignment) shows 99.8% of baseline bandwidth. No stride-dependent anomaly exists in the 64B-32KB range. L2 latency is uniformly ~350-371 cycles regardless of access pattern. [S3/case_6, S3/case_7, S8/case_76, S8/case_115]

Theoretical Analysis: Modular interleaving selects the target MC/L2 partition as partition = (addr / interleave_size) % N_partitions. This creates systematic conflicts when access strides are multiples of interleave_size * N_partitions. XOR-based hashing instead computes the partition index by XOR-ing multiple address bit fields, distributing traffic uniformly regardless of stride patterns. The trade-off is slightly higher hardware complexity (XOR logic vs. simple modular) but complete elimination of partition camping.

Microarchitecture Speculation [Confidence: H]: NVIDIA transitioned from modular to XOR-based L2/MC hashing between Ampere (SM_80) and Hopper (SM_90), as described in patent US10338919. SM_120 inherits this design. The hash function likely XORs bits from multiple address regions (e.g., bits [11:6] XOR bits [17:12] XOR bits [23:18]) to compute the 4-bit partition index for 12 MCs. The specific hash polynomial is not recoverable from software measurements (by design -- all strides produce uniform distribution), but the BW flatness proves the hash has no short-period collisions.

Implication for Optimization: Partition camping is not a concern on SM_120. Kernel developers do not need to pad array dimensions, add address offsets, or use "swizzled" addressing patterns to avoid L2/MC conflicts. All access patterns from unit stride to large power-of-2 strides distribute traffic uniformly. The only bandwidth-relevant factor is spatial locality (L2 hit rate) and working set size relative to the 96 MB L2 capacity. The gradual BW decline at strides > 2.5KB (down to 678 GB/s at 16KB stride) is caused by L2/TLB/DRAM row-buffer effects, not partition conflicts.

KI-1.5: GDDR7 Near-Symmetric R/W with Closed-Page Policy and Bank-Level Parallelism

Empirical Evidence: Read bandwidth: 1231-1292 GB/s (91.6-96% of theoretical). Write bandwidth: 1171-1256 GB/s (87.1-93.4% of theoretical). Read/Write ratio: 1.02-1.07x (near symmetric). Single-warp DRAM pointer-chase shows a base latency of 370 cyc (stride >= 1280 B) and a +445 cyc penalty at stride < 640 B, attributable to closed-page same-bank consecutive-access serialization (not row-buffer hit/miss). Refresh-induced jitter shows P99/P50 ratio of 2.48x. The previously cited "344 vs 883 cyc page-hit/miss" pair from [S21/case_73] is a misnomer — see Theoretical Analysis below. [S8/case_81, S21/case_74-75, S24/case_8, S24/case_11]

Theoretical Analysis: GDDR7 uses independent read and write channels (unlike HBM which multiplexes R/W on shared channels), enabling near-symmetric bandwidth. The GB202 DRAM controller operates a closed-page (auto-precharge) row buffer policy [S24/case_11]: every access ends with auto-precharge, so consecutive accesses to the same bank must wait full tRP + tRCD + tCL (this is the +445 cyc penalty observed at small strides), while accesses to different banks overlap precharge with the next bank's activation. The "344 vs 883 cyc" pair from [S21/case_73] is a methodological artifact: it compared an L2-resident 400 KB random pointer-chase (344 cyc, L2 hit) against an L2-thrashing 128 MB sequential pointer-chase at stride=128 B (883 cyc, DRAM access with same-bank serialization). The 539 cyc gap is therefore L2-vs-DRAM plus same-bank serialization, not a row-buffer hit/miss differential, and the term "page hit/miss" is retired from this section.

Latency decomposition (single-warp DRAM access at stride=128 B, [S21/case_73] sequential):

When the next dependent access lands on a different bank (stride >= 1280 B), the tRP wait term collapses to ~0 (precharge of the previous bank overlaps with the new bank's ACT), recovering the ~370 cyc baseline observed in [S24/case_8].

Microarchitecture Speculation [Confidence: H]: The GDDR7 controller on GB202 implements a closed-page (auto-precharge) policy with bank-level parallelism as the primary throughput mechanism. Each access incurs full ACT + CAS regardless of preceding address; throughput scales by issuing concurrent requests across the 4-way bank groups per channel and 16 channels. The near-symmetric R/W bandwidth (1.02-1.07x ratio) is a departure from HBM-based systems (A100/H100) that typically show 10-15% R/W asymmetry due to shared channel arbitration. This symmetry simplifies performance modeling for SM_120 -- read and write bandwidth can be treated as equal for roofline analysis.

Implication for Optimization: (1) Place consecutive same-warp dependent accesses on different banks (use stride >= 1 KB or randomize within a bank-friendly pattern) to avoid the +445 cyc same-bank serialization. TMA tensor loads benefit not from "page hits" but from issuing many independent requests that saturate bank groups in parallel. (2) The near-symmetric R/W bandwidth means store-heavy kernels (e.g., activation checkpointing, KV-cache writes) do not suffer the write penalty seen on HBM systems. (3) The 2.48x P99/P50 refresh jitter means tail-latency-sensitive applications (serving) should use working sets that fit in L2 (96 MB) to avoid DRAM refresh pauses. (4) For bandwidth-bound kernels, the effective ceiling is ~1231 GB/s (91.6% of 1344 GB/s spec / 96.0% of 1283 GB/s operational, [S8/case_81] canonical) due to MC overhead and ECC tax.

1.8 Summary

SM_120 (GB202, RTX PRO 5000) implements a 110-SM, 4-SMSP-per-SM, single-issue architecture. Its defining architectural characteristics are:

  1. Warp-level MMA only: Full mma.sync coverage (HMMA/IMMA/DMMA/QMMA/OMMA families, 22+ shape/type variants, including sparse), but no wgmma or tcgen05 -- kernels must use explicit shared memory staging.

  2. No TMA multicast: Software-emulated with 2000-8000x penalty; cluster GEMM operand sharing is not viable; CUTLASS uses ClusterShape<1,1,1>.

  3. Rich sub-byte support: Native FP8/FP6/FP4 dense and MX block-scale MMA at zero additional cost over dense formats.

  4. Uniform memory subsystem: 12-MC GDDR7 with XOR-hash interleaving eliminates partition camping; 96 MB L2 with 12 uniform-latency partitions; near-symmetric R/W bandwidth.

  5. Deterministic timing: clock64 with 0-cycle overhead and 0.0135% frequency drift enables sub-ppm measurement precision for all downstream characterization chapters.

These findings establish the foundational architectural parameters that all subsequent chapters reference for latency/throughput normalization, occupancy modeling, and optimization guidance.
