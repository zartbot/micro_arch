CHAPTER 05 · SM_120 · GB202 BLACKWELL

Instruction-Level Characterization

This chapter presents a comprehensive per-instruction latency and throughput reference for all non-Tensor-Core compute instructions on NVIDIA SM_120 (GB202, Blackwell family; RTX PRO 5000 72GB). The characterization spans integer ALU (Section 4, 42 cases), floating-point ALU (Section 5, 104 cases), and logic/shift/conversion instructions (Section 6, 54 cases). All measurements employ single-warp execution (1 block, 32 threads) to eliminate scheduling noise, with dependent chains for latency and independent chains (ILP sweeps) for throughput. Each result reports the minimum over 1000+ measurement iterations after 200+ warmup iterations, with clock64 overhead (1 cycle) subtracted. SASS verification via `cuobjdump --dump-sass` confirms the actual instructions executed in all cases.

5.1 Integer ALU

The SM_120 integer pipeline operates through two primary subpipes: the fmalite pipe (handling IADD3 and lightweight integer operations) and the fmaheavy pipe (handling IMAD, shifts, and byte permutation). These two subpipes can co-issue with one another and with the FP32 pipe, enabling high aggregate throughput for mixed workloads.

SM_120 has no native integer division hardware. The div.u32 instruction is decomposed by ptxas into a 9-instruction software sequence:

  1. MUFU.RCP (reciprocal approximation via SFU) -- only for runtime divisors
  2. IMAD (Newton-Raphson refinement)
  3. IMAD.HI.U32 (quotient estimate)
  4. IADD (remainder computation)
  5. IMAD (remainder refinement)
  6. ISETP.GE.U32.AND (1st correction check)
  7. Predicated IADD (conditional correction)
  8. ISETP.GE.U32.AND (2nd correction check)
  9. Predicated IADD + SEL (final quotient selection)

The constant-divisor path saves ~13 cycles by using ptxas-precomputed magic-number reciprocals instead of runtime MUFU.RCP. All divisor values (powers-of-two or otherwise) produce identical constant-path latency of 47 cycles, confirming that ptxas always uses the magic-number approach regardless of divisor properties.

5.1.6 Microarchitectural Analysis: The Dual-Subpipe FMA Architecture

The INT32+FP32 co-issue measurement is the single most important finding for understanding the SM_120 compute pipeline. Case 32 provides definitive throughput-mode evidence (complementing Case 5's latency-mode result):

Test Total Ops Min Cycles cyc/op MIX/MAX Source
INT-only (ILP=4, IADD3) 1024 1024 1.000 — [S4/case_32]
FP-only (ILP=4, FFMA) 1024 1023 0.999 — [S4/case_32]
MIX 4INT+4FP 2048 1025 0.500 1.001 [S4/case_32]
MIX 8INT+4FP 3072 2049 0.667 1.000 [S4/case_32]

Why MIX/MAX = 1.001 proves separate subpipes, not a unified pipe: If IADD3 and FFMA shared a single issue port (as on many prior GPU architectures), the mixed workload would require 1024 INT + 1024 FP = 2048 issue slots → 2048 cycles (MIX/MAX ≈ 2.0). The measured 1025 cycles proves both pipes issue simultaneously. The 8INT+4FP test further confirms: the 4 FP chains (1023 cyc worth) complete entirely within the 8 INT chains' execution window (2048 cyc) — the FP pipe operates in parallel with zero interference.

Architectural explanation of the dual-subpipe design: The diagram above illustrates how each SMSP within the SM_120 contains a single warp-scheduler dispatch port that feeds into an FMA Group Opcode Selector MUX. This multiplexer routes each instruction to one of two physically independent execution pipelines based on its opcode class:

  • pipe_fma (fmalite) — handles lightweight integer operations (IADD3 at 2-cycle latency), logic (LOP3.LUT), comparisons (ISETP), and all FP32 arithmetic (FFMA, FADD). This is the "workhorse" pipe for address computation and floating-point data processing. It has a 4-stage pipeline depth for FFMA and requires ILP=4 for saturation.
  • pipe_fmaheavy — handles integer multiplication (IMAD at 4-cycle latency), funnel shifts (SHF), byte permutation (PRMT), packed FP16 FMA (HFMA2 with 2-cycle issue), and INT8 dot-product (IDP.4A). It saturates at ILP=2 with a shallower 2-stage pipeline.

The critical architectural insight is that these two subpipes execute completely independently — when one pipe is busy with an IADD3 chain, the other can simultaneously process an FFMA chain with zero mutual interference. This was verified by measuring MIX/MAX ratios of exactly 1.000–1.001 across multiple test configurations (Cases 5 and 32), proving the aggregate throughput reaches 2.0 ops/cycle when both pipes are saturated. The NCU hardware performance counters (smsp__inst_executed_pipe_fmalite and smsp__inst_executed_pipe_fma) independently confirm that IADD3 and FFMA are dispatched to separate physical pipes.

Architectural significance for GEMM kernels: This dual-subpipe design means that address computation (IADD3, LOP3 for pointer arithmetic) and data computation (FFMA for epilogue) can execute simultaneously within a single warp's instruction stream. A GEMM mainloop that interleaves IADD3 (tile address update) with FFMA (accumulator bias/activation) pays zero throughput penalty — both complete in the time of the slower pipe. This eliminates the traditional trade-off between "address compute cycles" and "useful FLOPs."

5.1.7 Integer Division: The Most Expensive ALU Decomposition

The div.u32 SASS decomposition (Case 8) reveals the compiler's software division algorithm on SM_120 hardware that has no dedicated integer divider. The 9-instruction sequence has a precisely measurable critical path:

How the division algorithm works: The diagram above shows the complete 9-step software division sequence that ptxas generates for div.u32 on SM_120. Since there is no dedicated integer division hardware, the compiler implements a reciprocal-multiply-correct algorithm:

  1. Reciprocal estimation (Step 1): The SFU's MUFU.RCP computes an approximate reciprocal of the divisor in floating-point. This single step dominates the entire critical path at ~18 cycles — consuming 33% of the total latency.
  2. Quotient computation (Steps 2–4): Newton-Raphson refinement (IMAD) followed by IMAD.HI.U32 extracts the upper 32 bits of the 64-bit product n × (1/d) to estimate the quotient. IADD then computes the initial remainder.
  3. Two-stage correction (Steps 5–9): Because the reciprocal approximation may be slightly imprecise, the algorithm performs two predicated correction passes. Each pass checks whether the remainder exceeds the divisor (ISETP.GE) and conditionally increments the quotient (@P IADD). This handles off-by-one and off-by-two errors inherent to fixed-precision reciprocal methods.

The 60-cycle runtime divisor cost is dominated by MUFU.RCP (step 1, ~18 cyc) and IMAD.HI.U32 (step 3, 9 cyc). The constant-divisor path at 47 cycles replaces the expensive MUFU.RCP with a precomputed magic-number multiplication, saving exactly the MUFU.RCP latency minus the simpler IMAD constant-multiply cost. All constant divisor values — whether powers-of-two or arbitrary — produce identical 47-cycle latency, confirming ptxas always uses the magic-number approach.

5.2 Floating-Point ALU

The SM_120 floating-point subsystem consists of three execution pipes: P0/FMA (FP32 and FP16/BF16 packed operations), P1/FP64 (double-precision), and P2/SFU (transcendental functions). All three can co-issue with one another and with the integer pipe.

Floating-point subsystem architecture overview: The diagram above illustrates the three independent floating-point execution pipes within each SMSP of the SM_120. Unlike architectures where FP and INT compete for shared issue bandwidth, SM_120 allows all three FP pipes to execute simultaneously with each other and with the integer pipes (fmalite + fmaheavy), enabling up to 5 concurrent operations per cycle from a single warp:

  • P0 / FMA Pipe (green): The primary compute workhorse handling FP32 (FFMA at 4-cycle latency, 4-stage pipeline, ILP=4 saturation) and packed FP16/BF16 (HFMA2). A critical finding is that HFMA2 provides no throughput advantage over scalar FP32 — it uses the same 32-bit datapath with 2-cycle time-multiplexing, yielding identical per-element throughput. SM_120 handles IEEE 754 subnormals at full speed with zero penalty. The PTX fma.rn.f32x2 instruction is lowered to 2 independent FFMA instructions with no native hardware support.
  • P1 / FP64 Pipe (purple): A dedicated but severely rate-limited double-precision unit. Each SMSP contains one non-pipelined FP64 EU that executes DFMA in 64 cycles with no internal overlap (ILP=2 yields zero speedup). At 4 EUs per SM (one per SMSP), the aggregate throughput is 0.0625 DFMA/cyc/SM — a 1:32 ratio versus FP32. However, the FP64 pipe co-issues perfectly with P0, meaning FFMA instructions execute "for free" during the 64-cycle DFMA shadow.
  • P2 / SFU Pipe (amber): The Special Function Unit computing transcendentals (MUFU.RCP, MUFU.SIN, MUFU.TANH, etc.) with a bare hardware latency of ~18-20 cycles. PTX-level SFU instructions appear more expensive (18-44 cycles) because ptxas inserts range-reduction wrapper sequences of 4-7 FMA/FSETP/FSEL instructions around the core MUFU operation. The tier differences (Tier 1: rcp/rsqrt at 40-44 cyc; Tier 2: sin/cos at 18-22 cyc; Tier 3: tanh/ex2 at 18 cyc) reflect wrapper complexity, not the MUFU unit itself. At saturation, 4 SFU units per SM provide ~1 op/cyc aggregate throughput.

The bottom bar chart shows the stark throughput disparity: FP32 and FP16 (per-element) are equal at 1.0x, while FP64 provides only 1/32x of FP32 throughput in real workloads. For half-precision compute acceleration, Tensor Cores (258x the ALU FP16 throughput) are the only effective path on SM_120.

Pipeline depth = exactly 4 stages: The textbook ILP scaling (1.00× → 2.00× → 3.67× → plateau) is perfectly consistent with a 4-stage pipeline:

Pipeline stage model and ILP scaling analysis: The figure above presents the SM_120 FP32 (FFMA) pipeline microarchitecture as measured by Case 23. The pipeline consists of four physical stages — (1) Operand Collect + Decode, (2) Multiply, (3) Align + Add, (4) Normalize + Round + Writeback — each consuming exactly 1 clock cycle for a total end-to-end latency of 4 cycles. The color-coded timeline diagrams in the middle section illustrate how instruction-level parallelism (ILP) directly controls pipeline occupancy:

  • ILP=1 (25% utilization, 3.98 cyc/op): With a single dependent chain, each FFMA must wait the full 4-cycle latency for its predecessor's result before entering Stage 1. Three out of four pipeline stages sit idle every cycle — the classic latency-bound regime.
  • ILP=2 (50% utilization, 1.99 cyc/op, exactly 2.00× speedup): Two independent chains alternate through the pipeline. While chain A's instruction is in Stage 3, chain B's next instruction enters Stage 1. The exact 2.00× speedup — with zero deviation — is strong evidence that the pipeline has no structural hazards, no write-back port conflicts, and no hidden bubbles. This is the hallmark of a clean in-order pipeline design.
  • ILP=4 (92% utilization, 1.09 cyc/op, 3.67× speedup): Four independent chains fill all four stages simultaneously, achieving theoretical peak throughput. The measured 92% utilization (not 100%) is entirely explained by the dispatch tax: out of 139 measured cycles for 128 FFMA operations, 11 extra cycles are consumed by measurement overhead (2 cyc for clock64) and loop control instructions (9 cyc for BRA/ISETP). The pipeline hardware itself is 100% saturated — every stage processes a useful FFMA operand on every cycle where an FFMA is dispatched.

The right panel confirms that no benefit exists beyond ILP=4: ILP=8 and ILP=16 both plateau at the same 1.09 cyc/op. This saturation point mathematically confirms the pipeline depth: depth = latency / issue_rate = 4 cyc / 1 cyc = 4 stages. The 7.9% dispatch tax (11/139 = 0.079) is an irreducible property of the measurement methodology — real kernels with unrolled loops and minimal control flow can approach the theoretical 1.00 cyc/op limit. Compared to Ampere/Hopper (which typically require ILP=4–8 for FP32 saturation due to deeper pipelines), SM_120's shallower 4-stage design requires only 4 independent accumulator registers per warp to reach peak throughput — a significant advantage for register-pressure-sensitive workloads like large-tile GEMM epilogues.

5.2.7 Microarchitectural Analysis: FP64 EU Architecture

SM_120 contains 4 independent FP64 execution units per SM, one per SMSP. Each EU executes the DFMA (double-precision fused multiply-add) instruction in 64 cycles as a non-pipelined operation — the EU is fully occupied for the entire duration and cannot accept a second instruction until the first completes. This design provides native double-precision hardware (not FP32 emulation), with a per-SM aggregate throughput of 0.0625 DFMA/cyc when all 4 EUs are utilized, yielding a FP64:FP32 throughput ratio of 1:32 in real workloads.

FP64 EU Topology

EU architecture details: The figure above shows the physical layout of the FP64 subsystem within one SM. Each of the four SMSPs contains a dedicated Warp Scheduler that dispatches DFMA instructions to its own private FP64 EU. Lane-Mask Dichotomy: Sparse vs Full-Warp Dispatch

A controlled experiment (8 kernel variants, single session) isolated the sole variable affecting the apparent FP64 contention: the warp lane mask. When all 32 lanes are active (the regime of all real DGEMM/DDOT workloads), 4 warps at 4W show exactly 0% contention. When only lane 0 is active (if (lane != 0) return; — a common microbenchmark pattern), the SMSP scheduler introduces ~12 cycles of sparse-mask dispatch overhead, inflating the per-warp DFMA cost from 64 to ~76 cycles (+19%).

Why is shfl.sync so expensive on SM_120 (29 cyc vs 2-5 cyc on prior architectures)?

On architectures prior to Volta (SM_70), warps executed in lockstep — all 32 threads were always at the same program counter. This meant shfl was a trivial register crossbar operation taking 2-5 cycles. Starting with Volta, NVIDIA introduced independent thread scheduling where threads within a warp can diverge and execute at different points. The .sync suffix on shfl.sync mandates a convergence barrier: before any data exchange can occur, the hardware must verify that all threads specified in the mask have actually reached the shuffle instruction. On SM_120, this convergence check dominates the latency budget (~8-10 cycles), followed by the actual 32-lane crossbar permutation (~12-14 cycles) and result writeback (~6-8 cycles). The three phases sum to the measured 28.95 cycles. Crucially, the mode selection (up/down/bfly/idx) adds zero extra cost — all modes traverse the same physical crossbar hardware with only the routing logic differing.

5.4.3 Architectural Implications

  1. Dual-pipe co-issue is the primary ILP mechanism: The confirmed perfect co-issue between INT32 (fmalite) and FP32 (fmaheavy) means that compiler-mixed workloads can achieve 2 ops/cycle without multi-warp concurrency. Kernel developers should interleave address arithmetic (IADD3) with FMA computation (FFMA) to exploit this free parallelism.

  2. Shallow pipeline depths favor low-ILP code: The INT32 pipe saturates at ILP=2 and FP32 at ILP=4. This is shallower than historical NVIDIA architectures (Ampere/Hopper typically require ILP=4-6 for INT and ILP=4-8 for FP). SM_120 trades pipeline depth for lower per-instruction latency.

  3. FP16/BF16 provides no throughput advantage over FP32: HFMA2 achieves exactly 1 FP16 element/cycle throughput, the same as scalar FFMA. This is a departure from architectures where FP16 offered 2x throughput. On SM_120, the half-precision advantage comes exclusively from Tensor Core operations, not the ALU pipe. [SPECULATIVE] This may indicate that the P0 pipe internally uses the same 32-bit FMA datapath for both FP32 and packed FP16, issuing HFMA2 across 2 cycles to process both lanes sequentially.

  4. FP64 is present but severely rate-limited: At 1:32 throughput ratio vs FP32 in real workloads (RESOLVED 2026-06-11; see §5.2.7), double-precision is intended only for occasional scalar computations (e.g., address calculation, accumulator precision) rather than bulk FP64 workloads. The non-pipelined nature means FP64 operations cannot be overlapped with one another, only with other pipe types.

  5. SFU latency is dominated by range-reduction wrappers: The bare MUFU instruction takes ~18-20 cycles, but PTX-level functions like rcp.approx.f32 cost 44 cycles due to subnormal handling and overflow protection wrappers inserted by ptxas. Performance-critical code should consider whether the full range-reduction is necessary or if direct MUFU SASS (via CuAssembler patching) could reduce latency.

  6. Warp shuffle is expensive: At 29 cycles per dependent shuffle, SM_120 is 6-14x slower for shuffle-based reductions compared to Ampere/Hopper. This strongly favors shared-memory-based communication patterns over shuffle chains for within-warp data exchange on this architecture.

5.5.1 SFU Pipeline Depth Reconciliation

A significant discrepancy exists between the initial SFU depth estimate and the ILP sweep data. Case 5.7/31 (SFU pipeline depth sweep) reveals:

Pipeline depth = latency / saturated_throughput = 46.8 / 8.43 ≈ 5.6

This contradicts the earlier estimate of depth=2. The resolution lies in understanding what the "SFU pipeline" actually comprises at the SASS level:

Why does a "single" transcendental instruction need 7 SASS operations?

The MUFU (Multi-Function Unit) hardware computes reciprocals via a minimax polynomial approximation over a fixed-point lookup table. The LUT is indexed by the mantissa bits of the input and returns a pre-computed coefficient set, enabling a ~18-cycle polynomial evaluation that achieves the IEEE 754 .approx precision guarantee (~1 ULP error). However, this LUT has a restricted input range — it only handles normalized FP32 values in [1.0, 2.0). Inputs outside this range (subnormals near zero, very large values near overflow) would produce incorrect results. The ptxas compiler therefore wraps the bare MUFU instruction with a range-reduction envelope:

  1. Range detection (FSETP ×2): Test if the input is subnormal or near overflow using predicate comparisons.
  2. Scale factor selection (FSEL ×2): Conditionally select a pre/post-scale factor using the predicates. If the input is in-range, the scale factor is 1.0 (no-op).
  3. Pre-scaling (FMUL): Multiply the input by the scale factor to bring it into the LUT's valid range.
  4. Core computation (MUFU.RCP): Hardware reciprocal lookup + polynomial evaluation (~18 cycles on the dedicated XU pipe).
  5. Post-correction (FMUL): Multiply the raw result by the inverse scale factor to produce the correctly-scaled final answer.

This 5-phase algorithm ensures IEEE-compliant behavior across the entire FP32 dynamic range while keeping the hardware LUT compact (only ~1K entries). The same wrapper pattern applies to all MUFU variants (RSQ, LOG2, EX2, RCP) — only the core LUT differs.

Architectural reconciliation: The apparent contradiction between "depth=2" (bare MUFU) and "depth=5.6" (ILP sweep) resolves cleanly once we distinguish two pipeline models. The bare XU pipe has ~2 internal stages (18c latency / 8-9c issue rate), meaning only 2 MUFU instructions can be simultaneously in-flight within the unit. But the composite PTX pipeline — comprising 4 wrapper instructions (FSETP, FSEL, FMUL ×2) on P0 pipes plus the MUFU on XU — creates a 5-6 stage interleaving opportunity. At ILP=8, the wrapper FMA/FSEL instructions of chain N execute on the P0 pipe concurrently with the MUFU.RCP of chain N-1 on the separate XU pipe. This inter-pipe parallelism is the source of the 5.6× speedup, and it saturates when the XU issue slot (8.43 cyc) becomes the bottleneck — at that point, ~5 chains are simultaneously active across different stages of the composite pipeline.

Optimization implication: For latency-critical paths that can tolerate the reduced precision, bypassing the wrapper via direct MUFU SASS (using CuAssembler binary patching) saves 28 cycles per operation (46.8c → 18c, a 2.6× improvement). This is safe when inputs are guaranteed to be in the normal FP32 range [~2-126, ~2126], which covers the vast majority of practical kernel operands (e.g., softmax denominators, normalization factors).

5.5.2 HFMA2 Two-Cycle Issue: Time-Multiplexed Half-Precision

The measured 2-cycle issue interval for HFMA2 (Case 5.7/39, confirmed by Case 5.9/55) reveals the internal structure of the packed FP16 datapath. Key evidence:

  1. HFMA2 latency = FFMA latency = 4 cycles: Both share the same 4-cycle pipeline stages.
  2. HFMA2 throughput = 2 cyc/op: Saturates at ILP=2 (not ILP=4 like FFMA).
  3. Mixed HFMA2+FFMA time = sum of both (Case 55: 194 cyc measured vs 192 cyc predicted by serialization): They compete for the same pipe slot.
  4. Per-element FP16 throughput = per-element FP32 throughput: No packing speedup.

Inferred mechanism: The P0 (fmaheavy) pipe has a single 32-bit FMA datapath. HFMA2 processes the two packed FP16 lanes sequentially by occupying the pipe for 2 consecutive cycles:

Time-division multiplexing mechanism: The pipeline timing diagram (top-left) shows how a single HFMA2 instruction occupies the P0-heavy pipe for 2 consecutive cycles. In Cycle 0, the low half f16[0] enters Stage 1 of the 4-stage FMA pipeline; in Cycle 1, the high half f16[1] follows into Stage 1 while f16[0] advances to Stage 2. Both halves complete processing and become available as a packed result register at Cycle 4. The pipe is "busy" (cannot accept new instructions) during Cycles 0-1, giving the measured 2-cycle issue interval. This is fundamentally different from SM_80/SM_90 (Ampere/Hopper), where a dedicated 2×16-bit wide FMA datapath processes both f16 lanes in parallel in a single cycle, achieving 2× throughput over scalar FP32.

Three independent measurement confirmations establish this model beyond doubt: (1) ILP sweep shows HFMA2 saturates at ILP=2 while FFMA saturates at ILP=4 — consistent with pipeline depth = latency/issue = 4/2 = 2; (2) Mixed HFMA2+FFMA execution (Case 55) produces 194 cyc vs 192 cyc predicted by pure serialization, confirming they compete for the same pipe slot with zero overlap; (3) Per-element throughput is identical — HFMA2 delivers 2 f16 elements per 2 cycles = 1 elem/cyc, exactly matching FFMA's 1 f32 per 1 cycle.

Implications for kernel optimization: On SM_120, the packed FP16 ALU path offers zero throughput advantage over scalar FP32. The only benefit of HFMA2 over two scalar FFMA instructions is code density (1 PTX instruction vs 2, potentially better I-cache behavior). For any workload requiring FP16 compute throughput, Tensor Cores are the exclusive viable path — a single MMA instruction processes 4096 FP16 operations in 15.88 cycles, yielding 258 FP16 ops/cyc, which is 258× faster than the ALU FP16 path. This architectural choice reflects NVIDIA's design philosophy for the consumer Blackwell die: dedicate silicon area to Tensor Cores rather than duplicating FP16 ALU datapaths.

5.5.4 The Register Renaming Artifact in SFU Measurements

A critical methodological insight from Case 5.1/01: ptxas applies register renaming to PTX inline-asm chains even when using a single +f constraint. For the PTX sequence: 

  // Intended: single chain (depth=64)
  tanh.approx.f32 %f0, %f0;  // 64 iterations
  tanh.approx.f32 %f0, %f0;
  ...

ptxas generates SASS using 4 distinct destination registers (R3, R4, R5, R6), creating 4 interleaved sub-chains of 16 operations each. The measured "per-op latency" of 17.7 cyc actually reflects the inter-chain issue rate (pipeline throughput), not the true single-instruction latency.

Corrected SFU latency model:

  • Measured at PTX level with single-variable chain: 17.7 cyc (reflects 4-way renaming × throughput)
  • True bare MUFU latency (estimated from ILP-1 pointer-chase): ~18-20 cyc
  • Composite PTX rcp/rsqrt/lg2 latency (including wrappers): 40-47 cyc

This artifact means that SFU latency measurements from PTX-level inline asm are lower bounds of true single-instruction latency when the measurement chain is long enough for ptxas to apply multi-register renaming. The most reliable measurements use pointer-chase through MUFU (where renaming cannot help) or CuAssembler-patched single-register SASS.

5.5.6 Pipeline Depth Summary Model

Pipeline depth model: The figure above consolidates all instruction-level measurements into a unified pipeline characterization. The critical insight is that pipeline depth (the number of independent chains needed to saturate throughput) is not equal to latency — it is determined by the formula depth = ceil(latency / issue_interval). This distinction is architecturally significant: FFMA has latency=4 and issue_interval=1, giving depth=4 (need 4 independent chains). IMAD also has latency=4, but its half-rate issue (2 cyc/issue on fmaheavy) gives depth=2 — only 2 chains are useful, meaning register pressure for IMAD loops is half that of FFMA loops despite identical latency.

Co-issue architecture: The bottom-left panel quantifies the co-issue relationships between all major pipe pairs. Every measured combination achieves ≥92% overlap (with most at exactly 100%), confirming that the SM_120 SMSP functions as a multi-pipe in-order processor: a single warp scheduler dispatch port feeds multiple physically independent backends (P0-lite, P0-heavy, P1, P2, CVT, LSU) that execute in parallel. This means a single warp can sustain up to 5 operations per cycle when the instruction mix covers multiple pipes — the key ILP mechanism on SM_120 without requiring multi-warp occupancy.

Practical implications for compiler/kernel optimization: (1) For GEMM inner loops, interleave IADD3 address math with FFMA compute to exploit the fmalite↔fmaheavy 100% co-issue. (2) For IMAD-heavy workloads (e.g., integer quantization), only 2 chains are needed to saturate — allocating more ILP wastes registers. (3) The non-pipelined FP64 EU (depth=1) means DFMA cannot overlap with itself at all — only cross-pipe co-issue with P0/P2 hides its 63-cycle latency. (4) SHFL's depth=7 means 7 independent shuffle chains would saturate the crossbar — but dependent reduction chains (the common case) get only 1 chain and pay the full 29-cycle latency per stage.
