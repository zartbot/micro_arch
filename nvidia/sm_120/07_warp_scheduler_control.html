Warp Scheduler & Control

This chapter characterizes the SM_120 warp scheduler, instruction pipeline, scoreboard architecture, register file, I-cache hierarchy, and occupancy model. The SM_120 employs a Compiler Guided Greedy Then Youngest (CGGTY) scheduling policy with zero-cycle context switch, a hybrid dependence model combining 4-bit stall counters (fixed-latency) with 6 hardware scoreboard counters (variable-latency), and a 128-bit fixed-width instruction encoding that embeds scheduling metadata directly into the control word. Key findings include: the stall counter is a correctness requirement (not a hint) with silent corruption on under-specification, ILP=4 achieves 92% peak throughput with a single warp, and the I-cache cliff at 8-16KB constrains GEMM unroll factors. 7.0 Pipeline Stage Model

Pipeline Stage Model — Architectural Walkthrough

The SM_120 front-end-to-writeback path is organized as a classic 6-stage in-order pipeline, replicated identically across all four SMSP (sub-partitions) within each SM. The design philosophy emphasizes compiler-managed dependence tracking and 0-cycle resource contention elimination, pushing complexity out of hardware and into the 128-bit SASS control word. Below is a stage-by-stage walkthrough of the per-SMSP datapath.

Stage 1 — Fetch: Each SMSP owns a private L0 I-cache (~4 KB), backed by a shared L1 I-cache (~64 KB/SM) across the four SMSPs. A stream prefetcher — capable of looking ahead ≥256 instructions — completely masks L0 cache-line boundaries for straight-line code, delivering a sustained fetch rate of 1 instruction per cycle with zero boundary stalls [S11/case_67, S13/case_6]. The penalty for exceeding L0 capacity is modest in the transition zone (8–16 KB → +6% cost at +0.06 cyc/insn) but degrades sharply beyond 16 KB (see §7.8). Code-location switches — branches to non-prefetched addresses — incur a fixed +18.5-cycle redirect penalty, making the prefetcher optimized exclusively for linear instruction streams. Cold-start fill from L2 proceeds at ~50.6 B/cyc.

Stage 2 — Decode: SM_120 uses a fixed-width 128-bit (16-byte) instruction encoding. The decoder extracts the 41-bit compiler-embedded control word from each instruction, which encodes: a 4-bit stall counter (stall[3:0]), a yield bit (yield[4]), write/read barrier indices (wrbar[10:5], rdbar[16:11]), and a write-back barrier mask (wtmask[22:17]). This control word is the backbone of the CGGTY scheduling model — it tells the hardware exactly how many cycles to wait before the next issue, which scoreboard barriers to set and wait on, and whether to yield the dispatch slot. Because the encoding is fixed-width, decode is a single-cycle operation with no variable-length penalty.

Stage 3 — Schedule: The CGGTY (Compiler-Guided Greedy-Then-Youngest) scheduler evaluates all resident warps each cycle in a deterministic four-step sequence: (1) decrement all stall counters, (2) build the eligible set (stall==0 ∧ SB-wait clear ∧ ¬yield ∧ ¬barrier-blocked), (3) greedy continuation of the previously-issued warp if still eligible, (4) youngest (smallest warpID) tiebreak if greedy fails. The maximum dispatch rate is 1 instruction per cycle per SMSP (4 per SM aggregate), and there is no back-to-back same-warp issue — the per-warp IPC ceiling is 1.0 [S11/case_49]. This stage is where the hybrid dependence model bifurcates: if the stall counter is non-zero, the scheduler simply decrements it and moves on; if stall==0, the hardware scoreboard is consulted as a fallback for variable-latency operations (LDG, LDSM, TMA, cp.async).

Stage 4 — Operand Collect: The Operand Collector (OC) is a decoupling buffer between the 256 KB register file and the execution backends. It features ≥3 read ports and eliminates bank-conflict penalties entirely (0-cycle penalty, confirmed by [S12/case_2]) by spreading same-bank reads across internal cycles, transparent to the scheduler. The OC also houses a Register File Cache (RFC) with ≥24 entries per warp using LRU eviction [S12/case_6], capturing recently-read operands to avoid redundant bank accesses. Bypass forwarding is effective for results completing 0–2 cycles ago, reducing effective RAW latency to 2 cycles for IADD3 [S12/case_9]. WAR hazards incur 0 extra cycles because the OC has already latched the old value before writeback occurs [S12/case_8].

Stage 6 — Writeback: Results are committed to the register file via a dedicated write port in 1 cycle. The operand collector's bypass network ensures that consumers reading a result 0–2 cycles after writeback can be serviced directly from the OC without a full register-file round-trip. The writeback stage also updates scoreboard barrier tracking: if the instruction carries a wrbar field, the corresponding SB counter is released, potentially unblocking a waiting consumer instruction.

Pipeline-Level Architectural Properties:

  • In-order issue, out-of-order completion: Instructions are issued in program order per warp, but different backends complete at different times. The scoreboard manages cross-backend synchronization for variable-latency operations, while the stall counter handles fixed-latency dependencies.
  • No pipeline interlocks for fixed-latency ops: The hardware does not verify RAW hazards for ALU/FP/TC instructions — it trusts the compiler-set stall counter. This is the source of both the area savings (41 bits/warp of software state vs 256+ bits of hardware scoreboarding) and the silent-corruption risk when stall values are set below write-back latency (see KI-7.1).
  • 0-cycle context switch: Because all warp contexts are simultaneously resident in the RF, the scheduler can switch warps every cycle with zero bubble. This makes TLP (thread-level parallelism) a free latency-hiding mechanism — the only cost is RF capacity allocation.
  • ILP-first saturation: A single warp at ILP=4 achieves 92% of peak FMA throughput, making per-warp ILP the primary compute-saturation lever. Occupancy (TLP) is primarily a memory-latency-hiding tool — adding warps beyond ILP=4 saturation provides zero compute throughput gain (see §7.9 for the ILP×TLP iso-throughput model).

7.1 Scheduling Policy

7.1.1 CGGTY Decision Flow

CGGTY (Compiler-Guided Greedy-Then-Youngest) is the warp-scheduling policy used by every SMSP on SM_120. Unlike classical hardware-only schedulers, CGGTY pushes a substantial portion of the dependence-tracking responsibility into the compiler via a 41-bit-per-warp software state that is embedded directly into the 128-bit SASS control word (stall counter, yield bit, write/read barriers, SB wait-mask). Each cycle the SMSP scheduler executes a deterministic four-step decision pipeline:

  1. Decrement stall counters — every warp's 4-bit stall counter is decremented by one (saturating at zero). This is the only mutation the scheduler performs unconditionally per cycle, regardless of which warp ends up being issued.
  2. Build the eligible set — a warp is eligible iff stall == 0 ∧ SB-wait clear ∧ ¬yield ∧ ¬barrier-blocked. The four conditions are AND-combined per warp; the first three are compiler-set fields in the control word, the fourth is hardware-tracked from bar.sync / __syncthreads arrivals. If the eligible set is empty the SMSP issues an idle bubble for that cycle.
  3. Greedy continuation — if the warp issued in cycle T-1 is still in the current eligible set, it is reissued; this captures intra-warp ILP and locality without any tiebreak logic. Empirically this path dominates: the measured W3/W0 = 3.96 ratio (out of an expected 4.0) confirms greedy is taken on roughly 99% of cycles where it is admissible [S11/case_28].
  4. Youngest tiebreak — if greedy fails (the previous warp left the eligible set), the scheduler picks min(eligible_set.warpID). The "youngest" name refers to smallest warp ID, which on SM_120 corresponds to the warp most recently bound to the SMSP via the static SC = ⌊warpid / 4⌋ mapping. Picking by ID requires zero extra hardware state — no aging counters, no LRU stack — making CGGTY substantially cheaper to implement than GTO (Greedy-Then-Oldest, used on Volta/Ampere, which must track per-warp dispatch timestamps).

The selected warp is dispatched into the appropriate execution backend (FMA/ALU/SFU/LSU/TC) and its instruction's stall and barrier fields are written back into the per-warp scheduler state for the next cycle.

The architectural payoff is twofold: (a) 41 bits/warp of compiler-provided state replaces ~256 bits/warp of hardware scoreboarding, freeing significant die area per SMSP; (b) 0-cycle context switch — because all warp state is RF-resident, the scheduler can cycle-by-cycle rotate between warps without any save/restore overhead. The tradeoff is that incorrectly set stall counters cause silent stale reads (KI-7.1) — there is no hardware safety net for wrong compiler hints.

Hot-warp bias (Case 25): CGGTY is not perfectly fair across warps. As warp count grows, greedy increasingly favors whichever warp last held dispatch — bias ratio rises from 1.0× at 4W to 2.0× at 32W. This is by design: greedy maximizes per-warp ILP utilization and operand-collector reuse (RFC hits), at the cost of slightly less inter-warp fairness. The static SC binding (Case 34) means the bias is contained within each SMSP — no warp can starve a warp on a different SMSP.

Anti-starvation (Case 24): Despite the greedy preference, CGGTY is not pathologically unfair. A warp with 10× the work of its siblings completes in only 6.7× the time — i.e., it does receive ~50% more cycles than its work-share alone would mandate, demonstrating partial anti-starvation. Combined with the youngest tiebreak (which lets newly-eligible warps preempt the dispatch slot when greedy fails), CGGTY converges to round-robin-like behavior whenever the dominant warp stalls.

The redux.sync vs shfl tree comparison (Case 9) is architecturally revealing — the two approaches solve the same problem (warp-wide integer reduction) via fundamentally different hardware paths, with a 3.77× latency gap:

The SM_120 SMSP contains a dedicated parallel reduction tree — a hardware unit that can sum all 32 per-lane values into a single scalar in approximately 45 cycles without traversing the warp crossbar. This tree is activated by the REDUX.SUM.S32 SASS instruction (the lowering target of redux.sync.add.s32), which executes as a single instruction and writes its result directly to a Uniform Register (UR). Because URs are shared across all lanes in the warp, the result is implicitly broadcast — no additional shfl or vote instruction is needed to distribute the scalar.

In contrast, the software butterfly tree using shfl.sync.bfly requires 5 stages (log₂32 = 5), each consisting of a SHFL.BFLY + IADD pair (10 instructions total). Every stage traverses the warp crossbar to exchange values between lanes at stride offsets of 16, 8, 4, 2, and 1, costing 34 cycles per stage. The total 170-cycle latency reflects 5 × 34 = 170 cyc of serialized crossbar traffic. The key tradeoff: after each stage, every lane holds a partial sum — the intermediate values are exposed in per-lane vector registers (R0..R31), enabling prefix-scan and exclusive-scan patterns that redux.sync cannot support.

Architectural implications: The 3.77× speedup (45 vs 170 cyc) confirms that the reduction tree is a physically distinct datapath from the warp crossbar — it does not internally execute 5 shfl-like stages. Rather, it performs a tree-structured parallel summation in dedicated hardware, likely using a logarithmic-depth adder tree that compresses 32 inputs to 1 output in a single instruction's latency window. The 45-cycle latency includes operand collection from 32 lanes, tree traversal, and UR writeback — a remarkable single-instruction aggregate throughput of 32 additions in 45 cycles (~0.71 add/ns at 2.1 GHz). For GEMM/attention kernels that need warp-wide reductions (e.g., online softmax normalization), redux.sync should be preferred whenever the result is a scalar consumed uniformly by all lanes; shfl remains necessary only when per-lane partial results are needed for scan-type algorithms.

The 2-way → 4-way cost explosion (3× → 30.6×): This super-linear scaling is NOT simply "4 paths executed serially" (which would be 4×). A naive serialization model predicts 4 paths × 2.98 cyc/path = 11.92 cyc/iter — only 4.0× baseline. The measured 91.03 cyc/iter is 7.6× higher than this prediction, leaving 79.11 cyc unaccounted for. This excess decomposes into three compounding microarchitectural factors:

Factor 1 — Binary-search branching structure (≈ 22 cyc, 24%)

ptxas does not compile a 4-way switch (tid % 4) into a balanced binary tree. Instead it emits a linear if/else-if chain, biased toward path 0: 

LEVEL 1:  ISETP.EQ  P, tid%4, 0   →  @P  BRA path0   ; BSSY/BSYNC pair
LEVEL 2:  ISETP.EQ  P, tid%4, 1   →  @P  BRA path1   ; BSSY/BSYNC pair
LEVEL 3:  ISETP.EQ  P, tid%4, 2   →  @P  BRA path2   ; BSSY/BSYNC pair
DEFAULT:                              BRA path3

Each level costs roughly 6–8 cycles of branch resolution plus the matched BSSY.RECONVERGENT / BSYNC.RECONVERGENT reconvergence-stack push/pop. With 4 leaves the chain reaches 3 levels deep, so the structural overhead alone is 3 × 7 ≈ 21 cyc. This cost grows logarithmically with path count: 16-way switch (§7.4.3) requires ~4 levels → 28 cyc, contributing to the 197.9× cost at 16-way.

Why a chain instead of a tree? Because ptxas optimizes for the common case — when most lanes hit path 0, the chain short-circuits at level 1. A balanced tree would amortize cost but penalize the common case.

Factor 2 — Unrolled code explosion → L0 I-cache thrash (≈ 50 cyc, 55% — DOMINANT)

The #pragma unroll 128 body multiplied by 4 path bodies × 4 ops per path × 16 B per SASS instruction yields a code footprint of: 

128 iterations × 4 paths × 4 ops × 16 B/insn  ≈ 32 KB  (unrolled body)

This exceeds the L0 I-cache (4 KB) by 8×. Cross-referencing the I-cache cliff measured in §7.8 ([S21/case_32-35]):

Code size Steady-state cost Regime
≤ 8 KB 1.08 cyc/insn L0-resident
16 KB 1.57 cyc/insn L0→L1 spillover
32–64 KB 4.30 cyc/insn L1 thrash

A 32 KB body sits squarely in the L1-thrash regime. Every L0-miss/L1-replay adds ~3 extra cycles per fetched instruction; aggregated over the inner loop, this dominates the divergence cost — empirically accounting for ~5-10× of total cost per the original characterization.

This is the key insight of CGGTY divergence cost on SM_120: divergence does not just serialize lane-paths, it also fragments the I-fetch stream and breaks streaming-prefetcher locality, which compounds with the branching overhead. Loop not unrolling (or partial unroll with #pragma unroll 4) would shrink the body to fit L0 and recover most of the lost performance — at the cost of higher loop-counter overhead.

Factor 3 — Reconvergence synchronization under ITS (≈ 7 cyc, 8%)

Volta-onward GPUs use Independent Thread Scheduling (ITS): each divergent lane group can issue independently from the other groups in the same warp. However, every BSYNC.RECONVERGENT instruction forces a rendezvous: each group halts at its BSYNC until all groups in the warp have reached the same reconvergence stack entry. The total wait time per BSYNC equals the slowest group's path time minus the average — i.e., synchronization cost grows with path-length variance, not with mean path length.

For our 4-way case with equal path lengths, the per-BSYNC wait is small (~1–2 cyc), but the reconvergence stack must support 4 simultaneously-active divergence frames (≥ 8 levels deep per §7.4.4), and each BSSY/BSYNC pair adds a fixed pipeline overhead (~4 cyc) regardless of group balance.

7.4.6 BSSY/BSYNC Reconvergence Architecture

SM_120 uses Blackwell's Independent Thread Scheduling (ITS) reconvergence model with BSSY/BSYNC — a per-warp hardware stack that manages divergent thread group rendezvous without compiler-emulated barrier tracking:

The BSSY.RECONVERGENT instruction acts as a stack push — it records the reconvergence PC onto a dedicated per-warp hardware stack before entering a divergent region. When the predicated branch @P BRA splits the warp into independent thread groups, each group receives its own active PC and can issue instructions autonomously under ITS. The BSYNC.RECONVERGENT instruction acts as a stack pop and rendezvous — it forces every divergent group to halt until all groups have reached the same reconvergence stack entry, at which point the warp reunifies and resumes execution from a single PC. The pure BSSY+BSYNC pair overhead is approximately 4 cycles when no actual divergence occurs (i.e., all lanes take the same path); when divergence is real, the BSYNC wait time equals max(group_times) − avg(group_times), meaning synchronization cost grows with path-length variance, not with mean path length.

The reconvergence stack is ≥8 levels deep ([S13/case_20]), allowing up to 8 levels of nested divergent branches. Crucially, the compiler reuses barrier slot B0 for all nesting levels — the hardware manages the stack implicitly per-warp, so ptxas does not need to track or assign distinct barrier IDs for each nesting depth. This B0 reuse is possible because BSSY/BSYNC operations are always properly nested (LIFO order), and the hardware stack naturally enforces the matching between each BSSY push and its corresponding BSYNC pop.

ptxas compilation aggressiveness: The compiler converts nearly all control-flow patterns to @P / @!P predication, completely bypassing the BSSY/BSYNC mechanism for ~95% of divergent code. Under predication, both the true and false paths execute through the same pipeline, with the predicate mask controlling which lanes' results are committed — this is why predicated divergence cost is split-ratio independent (§7.4.2). BSSY/BSYNC is emitted only when warp-level collective operations (e.g., shfl.sync with partial masks, redux.sync, vote.ballot.sync) require full warp convergence to produce correct results. This aggressive predication strategy minimizes reconvergence overhead at the cost of executing both paths — a tradeoff that favors compute-bound kernels where predicated instructions are cheap, but may hurt memory-divergent code where both paths issue separate loads (§7.4.4).

Divergent loop model: Divergent loops where each lane iterates a different count follow the max(iter) model with 2.0-2.7× overhead (Case 24). The cost is determined by the lane with the most iterations — minimizing max(iter) across lanes is more important than minimizing total work. This is a direct consequence of the BSYNC rendezvous semantics: the warp cannot proceed past the loop exit until the slowest lane finishes all its iterations.

7.7 Register File and Scoreboard

7.7.1 Register File Architecture

The SM_120 register file is a 256 KB SRAM organized as 65,536 × 32-bit entries, partitioned equally across 4 SMSPs (16,384 entries = 64 KB each). Each SMSP's register file is further divided into 2 banks using simple parity banking: even-numbered registers (R0, R2, R4, ...) reside in Bank 0, and odd-numbered registers (R1, R3, R5, ...) reside in Bank 1. This R%2 scheme ensures that a typical 3-operand instruction (e.g., FFMA Rd, Ra, Rb, Rc) with consecutive register numbers naturally spreads its reads across both banks.

Physical organization and allocation granularity

Register allocation is performed at 8-register granularity per thread — i.e., ptxas can only assign register counts in multiples of 8 (24, 32, 40, ..., 256). The hardware enforces a minimum of 24 registers per thread; even when the user requests maxreg=16, ptxas silently raises the allocation to 24 ([S21/case_111-112]). Each SMSP supports up to 12 warp slots, and total SM capacity is 48 warps. The occupancy formula is: 

warps/SM = min(48, floor(65536 / (regs_per_thread × 32)))

Bank-conflict elimination and the operand collector

Unlike earlier GPU generations where bank conflicts caused 1-cycle replay penalties, SM_120 achieves 0-cycle bank-conflict penalty ([S12/case_2]). This is accomplished by the Operand Collector (OC) — a decoupling buffer between the register file and the execution backends. The OC has ≥3 read ports (labeled A, B, C) and can service multiple operands per cycle from the same bank by spreading reads across consecutive cycles internally, transparent to the scheduler. Key OC capabilities:

  • RFC (Register File Cache): ≥24 entries per warp with LRU eviction policy. Recently-read register values are cached in the OC, eliminating redundant bank accesses for operands reused within a few instructions ([S12/case_6-7]).
  • Bypass forwarding: Results from instructions completing 0–2 cycles ago can be forwarded directly to consumers without a full register write-back + re-read round-trip. This reduces effective RAW latency to 2 cycles for IADD3 ([S12/case_9]).
  • WAR hazard absorption: Write-After-Read hazards incur 0 extra cycles because the OC has already captured the old operand value before the write-back occurs ([S12/case_8]).

7.8 I-Cache Architecture

Three-level instruction cache hierarchy

The SM_120 instruction fetch subsystem implements a three-level cache hierarchy optimized for high-throughput sequential instruction delivery:

  • L0 I-Cache (~4 KB per SMSP, private): Each of the 4 SMSPs has its own L0, providing single-cycle instruction delivery for code that fits within ~256 instructions. Per-SMSP isolation means warps on different SMSPs never contend for L0 capacity.
  • L1 I-Cache (~64 KB per SM, shared): Shared across all 4 SMSPs, organized in 128-byte lines (8 instructions per line at 16 B/insn). When code exceeds L0 but fits L1, the penalty is modest: +6% over the L0-resident baseline.
  • L2 / DRAM: For cold-start or extremely large code bodies, instructions are fetched from the unified L2 or DRAM at 3.46 cyc/insn (+220% penalty). Cold-start fill rate is ~50.6 B/cyc from L2.

Stream prefetcher mechanism and effectiveness

A stream prefetcher sits between the PC register and L0, looking ahead ≥256 instructions. For straight-line code, the prefetcher completely eliminates cache line boundary effects: sequential sweeps from 64 B to 2048 B show exactly zero timing variation (all 1.000 cyc/FFMA). This confirms the prefetcher can sustain full-rate delivery as long as the code path is linear.

The prefetcher's limitation is code-location switches: each jump to a non-prefetched address incurs +18.5 cycles of redirect penalty. This makes the prefetcher optimized exclusively for single linear streams — heavily-branching code (e.g., 16-way switch/case) cannot benefit from prefetching and instead compounds its divergence cost with I-cache misses (§7.4.1 Factor 2).

Hit/miss characteristics and performance impact

I-cache independence from data/constant caches

Section 24 (Cases 25–27) conclusively demonstrates that the instruction cache and constant cache occupy separate physical SRAM arrays. All three interference directions — constant pressure on I-cache, I-cache pressure on constant latency, and simultaneous pressure on both — show zero cross-interference. This means kernel designers need not worry about constant-memory-heavy prologues evicting instruction lines or vice versa.

7.11 Key Insights

KI-7.1: Stall Counter = Correctness Requirement, Not Performance Hint

Empirical Evidence: Setting stall < write-back latency via CuAssembler causes silent stale register reads -- no error, no trap. Sharp boundary at stall=4 for FFMA. Stale values cascade: 15 OOM error at 2-hop. Only stall=0 triggers HW scoreboard fallback. [source: S11/case_132-141]

Theoretical Analysis: NVIDIA's software-managed dependence model trades hardware complexity for compiler trust. The HW does NOT verify RAW hazards for fixed-latency instructions. 41 bits/warp of software state replaces 256+ bits of hardware scoreboard. Patent US11442795.

Microarchitecture Speculation [Confidence: H]: The dual mechanism (stall counter for fixed-latency, SB for variable-latency) is an area optimization. Full hardware scoreboarding for all instructions would require tracking 256+ register destinations per warp. By trusting the compiler for predictable-latency operations, NVIDIA saves significant die area per SM.

Implication for Optimization: CuAssembler binary patching must NEVER decrease stall counters below the instruction's write-back latency. The only safe direction is increasing stall (wastes cycles but never corrupts). stall=0 is always safe (triggers HW fallback). The optimization headroom from stall tuning is real but narrow (6.1% for HMMA chains).

KI-7.2: CGGTY Scheduler with 0-Cycle Context Switch

Empirical Evidence: Greedy dispatch confirmed via CuAssembler tracing. W3/W0 ratio=3.96 (expected 4.0). Context switch = 0 cycles (1-16 warps, RF-resident). I-buffer ~4 entries/warp. [source: S11/case_28,47]

Theoretical Analysis: All warp state is RF-resident -- no context save/restore needed. The scheduler simply changes its warp selection pointer. This enables cycle-by-cycle warp switching without pipeline bubbles.

Microarchitecture Speculation [Confidence: H]: The 0-cycle switch is possible because the register file holds all warp contexts simultaneously, and the operand collector can deliver operands from any warp's registers without additional setup.

Implication for Optimization: Warp diversity is free. Having warps at different pipeline stages (some computing, some waiting for memory) does not incur switching overhead. Heterogeneous workloads within an SM benefit from this property.

KI-7.3: Scoreboard Depth 6 SB with Sharp Cliff

Empirical Evidence: 6 SB counters per warp. 1-5 LDG all complete in 17 cyc (L1 hit). Sharp cliff at N=5->6 (17->761 cyc, 44.8x). 5 fully-overlapping SB counters confirmed. Effective outstanding depth ~10-11 before throughput doubles. [source: S11/case_43-44, S12/case_10,24]

Theoretical Analysis: SB counters track only variable-latency producers. The 6-SB design is a deliberate trade-off: sufficient for typical GEMM mainloops (3-4 outstanding loads) but limiting for memory-intensive kernels.

Microarchitecture Speculation [Confidence: H]: The 6-SB limit constrains GEMM mainloop ILP: LDG ILP=4 is optimal because ILP=8 exhausts SB. For FlashAttention, the softmax+MMA+LDG mix uses 3-5 SB for loads, leaving 1-3 for ldmatrix.

Implication for Optimization: Count SB usage in the mainloop. If total > 5 outstanding variable-latency ops, expect degradation. ptxas's co-consumed-load grouping (sharing 1 SB for loads consumed together) is the primary mitigation.

KI-7.4: I-Cache Cliff at 8-16KB Constrains GEMM Unroll Factor

Empirical Evidence: 1-8KB: 1.08 cyc/FFMA. 16KB: 1.57 (1.45x). 32-64KB: 4.3 cyc/FFMA (4.0x). Per-SMSP L0 ~4KB confirmed. [source: S21/case_32-35, S13/case_5]

Theoretical Analysis: L0 I-cache ~4KB per SMSP provides zero-overhead fetch for straight-line code. Large unrolled loops (>8KB) spill to L1 (~64KB/SM shared across 4 SMSPs), causing 45% throughput loss.

Microarchitecture Speculation [Confidence: H]: The stream prefetcher completely masks L0 boundaries for sequential code but cannot help when working set exceeds L0. The 4KB/SMSP size is a direct constraint on unroll factor: at 16B/instruction, 4KB holds 256 instructions.

Implication for Optimization: GEMM mainloop unroll factor should target <256 instructions (4KB) per path. K-unroll of 4 with 128x128 tile typically generates ~200-300 instructions -- right at the boundary. Profile with NCU smsp__inst_executed_pipe_cbu to detect I-cache pressure.

KI-7.5: ILP Dominates Over Occupancy -- 1 Warp at ILP=4 = 92% Peak

Empirical Evidence: Single-warp FFMA at ILP=4 achieves 1.08 cyc/op (92% of 1.0 theoretical). Adding warps provides zero throughput gain once ILP=4 is reached. Iso-contour: (16w,ILP=1) = (8w,ILP=2) = (4w,ILP=4). [source: S11/case_122, S21/case_53]

Theoretical Analysis: With pipeline depth 2-4, ILP=4 fully saturates the FMA backend. Additional warps only help for memory-latency hiding, not compute throughput.

Microarchitecture Speculation [Confidence: H]: SM_120 is fundamentally "ILP-first" for compute. The 4-deep FFMA pipeline combined with single-issue means 4 independent operations saturate one SMSP. The occupancy benefit is exclusively for hiding variable-latency operations (memory, barriers).

Implication for Optimization: High ILP (4+) with moderate occupancy (16-32 warps) is optimal. 48W regression (+19% for mixed workloads) is caused by L1/L2 cache contention, not compute starvation.

KI-7.6: Descriptor-Based Addressing Is the Only Global Memory Mode

Empirical Evidence: All ld.global/st.global lower to desc[UR][R.64] addressing on SM_120. Zero overhead: descriptor vs pointer gives 0 cyc delta. [source: S21/case_118-119]

Theoretical Analysis: SM_120 uses descriptor-based addressing exclusively. The uniform register holds the descriptor base; the vector register holds the offset.

Microarchitecture Speculation [Confidence: H]: Descriptors enable hardware bounds checking and type information without additional instructions, replacing the traditional pointer+offset model.

Implication for Optimization: No performance concern from descriptor-based addressing. The UR allocation for descriptors is handled by ptxas transparently.

KI-7.7: 24-Register Floor Enforced by ptxas

Empirical Evidence: maxrregcount<24 is silently raised to 24. ptxas exceeds maxreg=24 internally (uses 28-30) and prefers restructuring over spilling. Spill onset at 32 accumulators: 5.23x slowdown. [source: S21/case_111-115]

Theoretical Analysis: The 24-register floor accommodates SM_120's minimum ABI: frame pointer, stack pointer, descriptor registers, and predicate spill slots.

Microarchitecture Speculation [Confidence: H]: The minimum register set includes: R0 (zero/reserved), R1 (stack pointer), 2-4 descriptor registers (URs map to R-space for dispatch), and sufficient space for the compiler's scheduling window.

Implication for Optimization: Do not set maxrregcount below 32 for performance-critical kernels. The sweet spot is typically 64-96 registers (32-20 warps occupancy) for compute-bound GEMM.

KI-7.8: Stall Counter "Trust Gap" Makes CuAssembler a Double-Edged Sword

Empirical Evidence: stall=2 on a 4-cyc FFMA -> silent stale read (127.0 vs 511.0). Cascades to 15 OOM error at 2-hop. ptxas achieves near-optimal stall for GEMM mainloop (+0% improvement) but is 6.1% suboptimal for HMMA chains (stall=11 vs optimal stall=12). [source: S11/case_132-141, S21/case_82]

Theoretical Analysis: The optimization headroom from CuAssembler is real but narrow (6%), and the risk of silent corruption from wrong stall values is catastrophic.

Microarchitecture Speculation [Confidence: H]: The safest CuAssembler optimization is SB counter grouping (3 cyc savings, no corruption risk). For stall tuning, only INCREASING stall is safe; DECREASING below write-back latency is always unsafe.

Implication for Optimization: An automated agent MUST NOT adjust stall counters unless it can formally verify against the instruction latency table. The recommended approach: (1) SB grouping (safe), (2) yield bit manipulation (safe), (3) stall increase for power savings (safe), (4) stall decrease only with formal verification (dangerous). [S26/case_57] independently confirms Ch12's finding: stall=12 is optimal for HMMA dep-chains, 33.8% faster than ptxas default stall=11 at ILP=2. Three regimes observed: scoreboard-limited (stall<=11), optimal (stall=12-13), issue-rate-limited (stall>=14).

KI-7.9: 6-SB Scoreboard Is the Hidden ILP Ceiling

Empirical Evidence: Cliff at N=5->6 (44.8x). In MMA+LDG 3-way interleave, SB cliff at N=5 outstanding LDG. [source: S11/case_43-44,114]

Theoretical Analysis: The 6-SB limit constrains the number of independent memory operations per warp. Combined with the LDG ILP=4 sweet spot (ILP=8 degrades), this creates a hard ceiling on memory-level parallelism per warp.

Microarchitecture Speculation [Confidence: H]: The ILP ceiling explains why GEMM mainloops use 3-stage pipelines (not 5-6): each stage requires 1-2 SB counters for its outstanding loads, and 3 stages x 2 SB = 6 SB fully utilized.

Implication for Optimization: Kernel designers should count SB usage: each outstanding LDG/LDSM/cp.async/TMA consumes 1 SB. Total > 5 causes degradation. Manual SB assignment via CuAssembler can improve by ~3 cyc/group.

KI-7.10: SM_120 Is Fundamentally "ILP-First" for Compute

Empirical Evidence: (1W, ILP=4) = (4W, ILP=1) = (2W, ILP=2) for FFMA throughput. 48W regression +19% for mixed workloads from cache contention. [source: S21/case_53, S11/case_122]

Theoretical Analysis: The 4-deep FMA pipeline combined with single-issue means ILP=4 saturates per-SMSP throughput. Additional warps provide: (1) memory latency hiding, (2) cache contention (negative), (3) barrier amortization. Net effect: moderate occupancy (16-32W) optimal for most workloads.

Microarchitecture Speculation [Confidence: H]: This is fundamentally different from Hopper (SM_90) where 2-partition SMs benefit more from TLP. SM_120's 4-SMSP single-issue design rewards deep unrolling (high per-warp ILP) over high occupancy.

Implication for Optimization: Optimal strategy: high ILP (4+) with moderate occupancy (16-32 warps). Avoid 48W for mixed workloads. For TC-bound: scale linearly to 48W (TC saturates at 4W per SM regardless). For memory-bound: 24-32W optimal (L2 contention at 48W).

7.12 Summary

The SM_120 warp scheduler and control architecture is characterized by three key design choices: (1) a trust-the-compiler dependence model that trades hardware scoreboard complexity for 41 bits/warp of software-managed state, (2) zero-cost context switching enabled by RF-resident warp contexts, and (3) an ILP-first throughput model where pipeline depth (not occupancy) determines compute saturation. The 6-SB scoreboard and 8-16KB I-cache cliff are the primary resource constraints that kernel developers must design around.
