CHAPTER 11 · SM_120 · GB202 BLACKWELL

Optimization Guidance

This chapter synthesizes the measured microarchitectural data from Chapters 1--10 into actionable optimization guidance for three production kernel frameworks: CUTLASS, DeepGeMM, and FlashInfer. Unlike generic GPU optimization guides, every recommendation is backed by cycle-level measurements from the SM_120 (GB202, RTX PRO 5000 72GB) microbenchmark suite, enabling quantitative cost-benefit analysis for each optimization decision. The chapter establishes three distinct performance regimes (TC-bound GEMM, DRAM-bound GEMV, and mixed attention), provides a priority-ordered optimization checklist with expected benefits, and documents framework-specific patterns validated against SM_120 hardware characteristics.

11.0 SM_120 GEMM Data Path (G-S-R-TC)

The critical data path for matrix multiplication on SM_120 traverses four stages: Global Memory, Shared Memory, Register File, and Tensor Core. SM_120 lacks Tensor Memory (TM) and wgmma instructions, making this path strictly smem -> ldmatrix -> RF -> mma.sync.

11.1.3 SM_120-Specific Optimizations

1. Tile re-partitioning with 128B swizzle. The optimal pipeline for feeding MMA operands from shared memory on SM_120 is: smem (128B swizzled) -> ldmatrix.sync.aligned.x4 -> HMMA. This yields 29.1 cyc/tile with ldmatrix latency fully hidden behind MMA execution. The .trans qualifier on ldmatrix is free (0 cyc overhead), enabling efficient B-matrix transposition without explicit permutation. [source: S10/case_17]

2. Pipeline depth: 3-stage optimal. Triple-buffered cp.async pipelines achieve 1.92x speedup over serial execution. A 4th stage provides only +5.5% marginal benefit, which is not worth the additional 33KB smem consumption (reducing occupancy from 2 to 1 block/SM in many configurations). [source: S10/case_82]

3. Warp specialization: marginal benefit without TM. On SM_120, warp specialization (2 producer + 2 consumer warps) provides only 5.8% speedup for TMA-bound tiles (1139 vs 1206 cyc). All-compute configurations achieve 2.5-3.1x better throughput for TC-bound workloads. Specialization adds a constant 259 cyc/iter overhead from mbarrier synchronization between warp roles. Named barriers (15 cyc) should replace mbarriers (211 cyc) for intra-CTA synchronization wherever TMA completion tracking is not required. [source: S23/case_33-38, S9/case_33]

4. Epilogue conversion. BF16 output packing via F2FP.BF16.F32.PACK_AB costs 11.8 cyc/tile, which can be overlapped with the next tile's prologue when STG fire-and-forget is used (40.9% overlap achieved). [source: S10/case_108]

11.2.2 SM_120-Specific Impact

DeepGeMM's NVLink barrier mechanism (red.release.sys.global.add) is designed for NVLink-connected systems where inter-GPU signaling costs approximately 200 ns. On SM_120 (PCIe-only topology), the same pattern costs substantially more:

Operation Cost (cyc) Cost (us @ 2.5 GHz) Source
red.release.gpu 521 0.21 S8/case_129
red.release.sys 947 0.38 S8/case_129
Round-trip signaling ~14400 5.76 S8/case_130
Custom AllReduce bandwidth -- 2.4 GB/s S17/case_15
NCCL AllReduce bandwidth -- 35.2 GB/s S17/case_15

The 14x gap between custom AllReduce (2.4 GB/s) and NCCL (35.2 GB/s) makes DeepGeMM's nvlink_barrier.cuh mechanism non-viable on SM_120. NCCL is the only viable collective communication path for this platform.

11.2.3 Recommended Adaptations

  1. Replace NVLink barriers with NCCL collectives. The PCIe round-trip cost (5.76 us) makes custom AllReduce infeasible.
  2. Retain st.bulk.weak for smem initialization. The 533-6322x speedup over memset is platform-independent.
  3. Use cluster_arrive_relaxed where semantics allow. The 4.4x cost reduction (85 vs 371 cyc) is unconditional.
  4. Prefer tensormap.replace(addr/dim) over stride. Address and dimension updates (2 cyc) are 21x cheaper than stride updates (43 cyc).

11.3.3 Recommended Optimizations for FlashInfer on SM_120

  1. Replace shfl_down-based rowsum with MMA-based rowsum. Setting B=all-ones and computing a single MMA yields 16 partial sums simultaneously. Per-row cost: 1.81 cyc (MMA) vs 20 cyc (shfl_down) = 11x per row, 96x batched across 16 rows.

  2. Exploit SFU+MMA independence for free softmax. The SFU pipe (EX2, RCP) and TC pipe (HMMA/QMMA/OMMA.SF) are fully independent on SM_120. Up to 7 EX2 operations execute for free during each HMMA cycle, enabling softmax computation at zero additional cost during MMA-bound tiles.

  3. Use FP4 KV cache where accuracy permits. FP4 provides 1.64x speedup over FP8 by reducing bandwidth demand. The MX block-scale format (FP4 E2M1 with UE8M0 scale factors) maintains acceptable accuracy for inference KV caches.

  4. Page size 64 for paged KV attention. Measured optimal throughput occurs at page_size=64 (6.50 GB/s, 3.1x over ps=1). L2 hash-based indexing eliminates conflict misses for random page access patterns.

11.4 Key Insights

KI-11.1: CUTLASS MXFP4 Achieves 100.6% Efficiency -- Best-Optimized Kernel Path

  • Empirical Evidence: CUTLASS block_scaled_gemm OMMA cooperative_o with 256x128x128 tile peaks at 1076.5 TFLOPS (100.6% of 1070 theoretical at nominal 2.377 GHz baseline / 90.8% of 1185 hardware peak at sustained 2.572 GHz boost, V-02 RESOLVED 2026-06-11). Super-theoretical against the nominal-clock baseline due to clock boost (2572 vs 2377 MHz nominal). All 588 shapes supported with consistent performance. [source: S9/case_136]
  • Theoretical Analysis: Two distinct theoretical baselines exist for SM_120 MXFP4 — (a) nominal-clock baseline at 2.377 GHz ≈ 1070 TFLOPS (CUTLASS published reference, used in vendor literature for cross-generation comparison); (b) hardware-peak baseline at sustained boost 2.572 GHz ≈ 1185 TFLOPS (110 SM × 4 SMSP / 15.59 cyc × 16384 FLOP/MMA × 2.572 GHz, recorded in AppB §B.15 mxfp4_theoretical_at_2572_boost). Use baseline (a) when comparing to CUTLASS published reference points (yields the headline 100.6%); use baseline (b) for hardware-utilization analysis (yields the more conservative 90.8%, which matches FP16 cuBLAS efficiency at 92.4%). The 100.6% "super-theoretical" reading is therefore a clock-baseline artifact, not a measurement anomaly.
  • Microarchitecture Speculation [Confidence: H]: CUTLASS 4.x OMMA cooperative kernel fully exploits SM_120's unified TC pipeline by maintaining ILP>=4 across all MMA formats. The 256x128x128 tile maximizes smem utilization (99KB) and register budget while maintaining sufficient pipeline depth.
  • Implication for Optimization: This kernel represents the achievable ceiling for SM_120 MMA throughput. Any custom kernel should benchmark against CUTLASS MXFP4 cooperative as the reference point. Performance below 90% of this baseline indicates suboptimal pipelining or resource utilization.

KI-11.2: DeepGeMM NVLink Barrier Costs 521-947 cyc -- PCIe-Bound on SM_120

  • Empirical Evidence: red.release.gpu = 521 cyc; red.release.sys = 947 cyc. Custom AllReduce achieves only 2.4 GB/s vs NCCL 35.2 GB/s (14x gap). Grid sync via L2 atomic = 224 ns O(1). [source: S8/case_129-130, S17/case_14-15]
  • Theoretical Analysis: DeepGeMM's nvlink_barrier.cuh assumes NVLink signaling latency (~200 ns). On PCIe SM_120, the same pattern costs 5.76 us round-trip due to PCIe protocol overhead and non-coherent memory semantics.
  • Microarchitecture Speculation [Confidence: H]: The 14x performance gap between custom and NCCL AllReduce arises because NCCL uses kernel-level pipelining and GPU Direct RDMA optimizations that amortize PCIe protocol overhead, while DeepGeMM's barrier-based approach serializes the latency.
  • Implication for Optimization: On SM_120, always use NCCL for collective communication. DeepGeMM's NVLink barrier pattern is strictly non-portable to PCIe-only topologies.

KI-11.4: FlashInfer MMA-Based Rowsum 96x Faster Than shfl_down Per-Row

  • Empirical Evidence: MMA with B=all-ones: 7.25 cyc/mma for FP8 (1.81 cyc/row across 16 rows per MMA). shfl_down for single row: 174 cyc/warp (log2(32) = 5 shuffle stages). Per-row: MMA 1.81 vs shfl 20 cyc = 11x. Batched 16 rows: 96x. [source: S9/case_160-161]
  • Theoretical Analysis: MMA amortizes the TC pipeline cost across 16xK elements per instruction. Each shfl_down stage costs ~4 cyc/op with 5 stages required for full warp reduction, totaling 20 cyc per row. MMA computes 16 rows in a single 29-cyc instruction.
  • Microarchitecture Speculation [Confidence: H]: The TC pipeline processes the row reduction as a side effect of the standard MMA datapath -- no additional hardware is required. The B=all-ones trick converts the reduction into a matrix-vector multiply where each output element is the sum of the corresponding row.
  • Implication for Optimization: Replace all shfl-based row reductions in attention kernels with MMA-based rowsum. This is particularly impactful for online softmax where row sums must be computed every tile iteration.

KI-11.5: Warp Specialization Marginal on SM_120 Without TM

  • Empirical Evidence: Warp-spec 2+2 configuration: 5.8% faster for TMA-bound tiles (1139 vs 1206 cyc). All-compute configuration: 2.5-3.1x better for TC-bound workloads. Constant 259 cyc/iter sync overhead from mbarrier. [source: S18/case_09, S9/case_33]
  • Theoretical Analysis: Without Tensor Memory, both producer and consumer warps must traverse the same smem->RF->TC path. Specialization adds synchronization overhead (mbarrier 211 cyc) without the TM bypass benefit that makes warp specialization profitable on SM_100.
  • Microarchitecture Speculation [Confidence: H]: SM_120 lacks any bypass path from TMA to TC. Without a dedicated operand buffer (such as Tensor Memory on other architectures), the producer warp's contribution is limited to issuing TMA descriptors — a task that can equally well be handled by the same warps that consume the data.
  • Implication for Optimization: On SM_120, prefer all-compute GEMM configurations. Use named barriers (15 cyc) instead of mbarrier (211 cyc) for any remaining intra-CTA synchronization. Reserve warp specialization only for workloads where TMA issue rate is the actual bottleneck (rarely true given the 4.4x data-starvation ratio).

KI-11.6: Named Barriers Preferred for Intra-CTA Sync

  • Empirical Evidence: Named barrier round-trip: 15 cyc. mbarrier round-trip: 211 cyc. mbarrier.arrive alone: 31 cyc (2x the full named barrier cost). [source: S23/case_38]
  • Theoretical Analysis: Named barriers use the lightweight BAR.SYNC hardware mechanism. mbarrier requires smem-based phase tracking with three distinct operations: SYNCS.EXCH + SYNCS.ARRIVE + SYNCS.PHASECHK.
  • Microarchitecture Speculation [Confidence: H]: The 14x cost differential arises because named barriers use a dedicated hardware unit (the barrier synchronization unit) that resolves in a single scoreboard check, while mbarrier must perform multiple shared memory transactions to update phase counters and check arrival status.
  • Implication for Optimization: Use mbarrier ONLY for TMA completion tracking (where expect_tx semantics are required) or cross-CTA cluster semantics. All other intra-CTA synchronization should use named barriers. This applies particularly to warp-specialized kernels where the synchronization overhead directly impacts mainloop throughput.

KI-11.7: SM_120 Optimization Has Three Distinct Regimes with Different Critical Paths

  • Empirical Evidence: GEMM at M=N=K>=2048 is TC-bound (73% efficiency). GEMV at M=1 is DRAM-bound (98% BW efficiency). FlashAttention at d=128 is thin-K-GEMM-bound (46% efficiency). Each has a DIFFERENT optimization priority. [source: S19/case_1,17,62]
  • Theoretical Analysis: The three regimes map to different entries in the optimization checklist:
  • TC-bound GEMM: Priority = ILP>=4 (#1, 1.81x) -> swizzle (#2, 4.23x) -> 3-stage pipeline (#3, 1.92x) -> precision downgrade (#4, 2-4x)
  • DRAM-bound GEMV: Priority = coalescing (#7, 5.1x) -> vectorize LDG.128 (#6, 1.20x) -> quantized weights (2.38x) -> batching (linear amort.)
  • Mixed attention: Priority = MMA-based rowsum (#14, 96x) -> selp mask (#15, free) -> operand reuse -> Br tile optimization
  • Microarchitecture Speculation [Confidence: H]: The regime classification depends on the arithmetic intensity (ops/byte) of the kernel relative to SM_120's roofline: TC peak = 1070 TFLOPS (FP4), DRAM BW = 1231 GB/s. The crossover occurs at ~870 ops/byte for FP4, ~217 ops/byte for FP16. Large GEMM exceeds this threshold; GEMV falls far below.
  • Implication for Optimization: The bottleneck classifier must correctly identify the regime BEFORE selecting optimization strategies. Applying TC-bound optimizations (ILP, swizzle) to a DRAM-bound GEMV wastes effort; applying memory optimizations (coalescing) to a TC-bound GEMM is irrelevant.

KI-11.8: The "No wgmma/No TM" Constraint Makes CUTLASS SM_120 Kernels Fundamentally Different from SM_100

  • Empirical Evidence: CUTLASS SM_100 uses sm100_gemm_tma_warpspecialized with wgmma descriptors, TM buffer, and multicast. CUTLASS SM_120 uses sm120_gemm_tma_cooperative with mma.sync, smem buffer, and ClusterShape<1,1,1>. [source: S14/case_13,14, S9/case_33]
  • Theoretical Analysis: The porting delta is NOT incremental -- it requires a full mainloop restructure:
  • wgmma -> mma.sync changes tile shape from m64nXk to m16n8k (4x smaller per instruction, 4x instruction count)
  • TM (256KB) -> smem (99KB) reduces buffer capacity 2.6x, limiting pipeline depth
  • multicast -> unicast+DSMEM adds 220 cyc per cross-CTA tile read
  • warp-specialization -> all-compute is 2.5-3.1x better on SM_120
  • Microarchitecture Speculation [Confidence: H]: SM_120's GB202 die is a different physical design from B200's GB202 (SM_100). The absence of Tensor Memory and wgmma is an architectural choice for the desktop/workstation segment, trading peak throughput density for simpler programming model and reduced die area.
  • Implication for Optimization: When porting ref_code/ kernels from SM_100 to SM_120, do NOT attempt incremental changes. Instead: (1) Start from CUTLASS SM_120 cooperative template, (2) port the algorithmic structure (tile sizes, reduction pattern), (3) re-tune ILP, pipeline depth, and tile shape for the mma.sync path. The measured expectation is: CUTLASS MXFP4 SM_120 = 100.6% efficiency, confirming this baseline is achievable.

11.5 Register Allocation Guidelines

Register pressure is a critical optimization variable on SM_120. The following constraints bound the design space:

Parameter Value Source
Total RF per SMSP 16384 x 32-bit Architecture
ptxas minimum floor 24 registers S21/case_111
ptxas exceeds maxreg to avoid spills Uses 28-30 when maxreg=24 S21/case_112
Spill onset threshold 32 accumulators per thread S21/case_113
Spill cost 0.32-0.43 cyc/spill-op S21/case_113-114
Spill penalty (worst case) 5.23x slowdown S21/case_113
launch_bounds impact Zero if regs already fit S11/case_124
setmaxnreg.dec cost 60 cyc one-time S12/case_13

Recommendations:

  1. Target <= 128 registers per thread for occupancy >= 2 blocks/SM. This leaves headroom for ptxas's 24-register floor and restructuring preferences.
  2. Avoid exceeding 32 accumulator registers in any inner loop. Beyond this threshold, ptxas is forced to spill (233 STL + 231 LDL observed), causing 5.23x slowdown.
  3. Use setmaxnreg.dec in epilogues where the accumulator register budget is no longer needed. The 60-cyc one-time cost is negligible compared to the occupancy benefit.
  4. Do not rely on launch_bounds for performance tuning. If registers already fit within the target, launch_bounds has zero measured impact.

11.6 Memory Access Pattern Recommendations

Coalescing and Vectorization

Pattern Measured BW / Cost vs Optimal Source
Stride-1 (fully coalesced) 858 GB/s 100% S21/case_1,5
Stride-16 (worst case) 169 GB/s 19.6% S21/case_5
LDG.E.128 (vectorized) 1.27-1.31x L2 -- S21/case_117
Misaligned access penalty <0.8% Negligible S21/case_3
Column-major GEMM tile 6.25% sector util 16x waste S21/case_4,8
STG.cg (L1 bypass) 3.59 cyc/store 12x vs default S8/case_95

Recommendations:

  1. Always ensure stride-1 (coalesced) access for global loads. The 5.1x penalty at stride-16 is catastrophic for bandwidth-bound kernels.
  2. Manually vectorize to float4/int4. ptxas does NOT auto-vectorize (confirmed: 8 scalar LDG remain 8 separate transactions). Manual float4 provides 1.27-1.31x benefit for L2-resident data.
  3. Use STG.cg for write-only output. L1 cache bypass reduces store issue cost by 12x (3.59 vs 42.78 cyc).
  4. Misalignment is a non-issue on SM_120. Arbitrary alignment offsets cause less than 0.8% penalty -- no need for alignment padding.
  5. Stage column-major data through shared memory. Direct column-major access wastes 87.5-93.75% of L2 bandwidth. Shared memory staging with XOR-swizzle eliminates this waste.

11.8 Compilation and Methodology Notes

All optimization recommendations in this chapter are derived from measurements under the following conditions:

  • Platform: SM_120a (RTX PRO 5000 72GB, GB202, 110 SMs, compute capability 12.0a)
  • Toolchain: nvcc CUDA 13.3, -O3 -std=c++17 -lineinfo -gencode arch=compute_120a,code=sm_120a
  • Clock pinning: nvidia-smi -lgc to suppress DVFS; measured clock ~2077-2497 MHz
  • Measurement: Single-warp <<<1, 32>>> for latency; multi-warp/multi-block for throughput
  • Timer: clock64() bracket; median over 1000+ iterations; 100+ warmup
  • CuAssembler validation: Cases involving stall counters used binary patching of control codes
  • NCU cross-validation: Sector counts, L2 BW, and throughput metrics cross-validated with Nsight Compute
