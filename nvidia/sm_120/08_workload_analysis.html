Workload Analysis

This chapter characterizes the NVIDIA SM_120 (GB202, RTX PRO 5000 72GB) under representative production workloads spanning the compute-bound, memory-bound, and synchronization-bound regimes. Through a systematic 588-shape GEMM sweep across 18 library/precision combinations, attention decomposition benchmarks, and full-chip power profiling, we establish the achievable throughput envelope, identify structural efficiency ceilings, and derive actionable optimization guidelines. The measured data demonstrates that SM_120 achieves up to 1076.5 TFLOPS (CUTLASS MXFP4, 100.6% of theoretical peak) for compute-bound workloads, 879 GB/s (98% of DRAM peak) for memory-bound GEMV, and reveals that FlashAttention efficiency is structurally capped at 46-61% by thin-K/thin-N GEMM shapes rather than softmax computation.

8.1.3 Chip-Level Efficiency Highlights

The 588-shape sweep reveals several notable efficiency characteristics:

  • CUTLASS MXFP4 achieves 100.6% of theoretical peak (1076.5 vs 1070 TFLOPS). The slight super-theoretical measurement arises from clock boost above the pinned frequency during sustained computation. This confirms that the CUTLASS cooperative kernel with 256x128x128 tiles and OMMA.SF.16864 instructions fully saturates the tensor core array.

  • cuBLAS FP8 achieves 102% efficiency (551.1 vs 536 theoretical) at large shapes. The same clock-boost effect produces the slight super-theoretical reading. cuBLAS FP8 requires cublasLt API and cannot run at M<8 due to implicit-scaling constraints.

  • Hand-tuned OMMA.SF sustains 99.0% of theoretical (1068.5 vs 1079.1 TFLOPS) at chip-level with 110 SM x 220 blocks x 50K iterations. This represents the absolute hardware ceiling for MX-FP4 computation on this chip.

  • Sparse MXFP4 peaks at 1318 TFLOPS but achieves only 61.7% efficiency. Tile alignment constraints (K=256 minimum) penalize small-K shapes, suggesting significant room for tile-size optimization in the CUTLASS sparse MXFP4 path.

  • Grouped GEMM MXFP4 (1036.4 T) achieves 96.9% of single-GEMM MXFP4 (1076.5 T). The 3.7% overhead from group dispatch is near-negligible, validating grouped GEMM as an efficient MoE execution strategy.

8.1.5 3-Stage Software Pipeline

The optimal software pipeline configuration for SM_120 GEMM is a 3-stage design achieving 1.92x speedup over unpipelined execution. The structure overlaps TMA loads with MMA computation across three shared memory buffers, with each stage consuming one buffer while the next is being filled asynchronously:

The pipeline operates by dedicating three shared-memory buffers to interleave data movement with computation. In the steady state, while the tensor cores compute on buf[i], the TMA engine is simultaneously loading buf[i+1] from DRAM. The third buffer (buf[i+2]) acts as a decoupling buffer that absorbs latency variation between TMA loads (479 cyc/tile) and the variable MMA compute phase. The fundamental constraint is the 4.4x data-starvation ratio: TMA delivers data at only 6.5 B/cycle while the tensor cores can consume at 28.9 B/cycle, making the pipeline load-latency-bound rather than compute-bound in the innermost loop. This starvation ratio is the primary reason that DRAM latency hiding requires approximately 64 FMA operations to overlap with the canonical 362-cycle DRAM access latency ([S26/case_10-12]).

Synchronization overhead is the largest structural efficiency loss: mbarrier arrive/wait and fence.proxy.async operations collectively consume 34% of the mainloop (546 out of 1615 cycles). The GPU-scope fence dominates at 163 cycles per invocation, establishing an irreducible overhead floor. The CTA-scope fence (7 cycles) is negligible by comparison. Adding a 4th pipeline stage yields only +5.5% additional speedup because the marginal benefit of deeper buffering is offset by increased shared-memory pressure — at 3 stages, the smem budget is already consuming 99 KB (3 x 33 KB/buffer), leaving minimal room for the CTA's other storage needs within the 228 KB per-SM shared memory capacity.

8.2.1 Microarchitectural Analysis: The L2 Caching Sweet Spot

The GEMV performance data (Case 17) reveals a precision-dependent L2 caching threshold that fundamentally determines decode inference throughput. The SM_120's 96 MB L2 cache creates a binary bandwidth regime: when the weight matrix (N × K × bytes_per_element) fits entirely within L2, effective bandwidth jumps 2–3× above DRAM peak; once the matrix exceeds L2 capacity, bandwidth drops to the DRAM ceiling of ~860 GB/s.

The architecture creates a sharp cliff at the L2 boundary: FP16 at N=K=4096 produces a 32 MB weight matrix that fits comfortably in the 96 MB L2, delivering 1832 GB/s (2.1× DRAM peak). Doubling to N=K=8192 inflates the matrix to 128 MB — exceeding L2 capacity — and bandwidth collapses to 860–879 GB/s (DRAM-saturated). INT8's 1-byte elements shift this cliff dramatically: the same N=K=8192 shape produces only a 64 MB matrix that still fits in L2, achieving 2780 GB/s (3.2× DRAM peak). This is the microarchitectural justification for W4/W8 weight quantization in decode-heavy inference — the throughput gain comes from L2 caching behavior, not from faster compute hardware.

A notable anomaly emerges in the BF16 vs FP16 comparison: at identical N=K=4096 (both 32 MB, comfortably in L2), BF16 GEMV achieves 2297 GB/s while FP16 delivers only 1832 GB/s — a 25% gap. Since both precisions use the identical HMMA.16816 hardware pipe, this difference is purely attributable to cuBLAS selecting a more L2-optimized kernel for BF16 GemmEx, likely employing different tiling granularity or more aggressive prefetch depth. The practical implication for deployment: BF16 weights are strictly preferable to FP16 for decode workloads on SM_120, gaining 25% bandwidth at zero accuracy cost (the mantissa precision difference is irrelevant for inference-only weights after training convergence).

8.3.1 FlashAttention Bottleneck Decomposition

The per-tile cycle budget for FlashAttention on SM_120 (d=128) reveals that the dominant bottleneck is thin-K GEMM efficiency, not softmax. Each attention tile processes ~520 cycles of work decomposed into four pipeline stages — Q×K^T MMA, online softmax, Score×V MMA, and accumulator reformat — with the critical insight that the SFU executing softmax co-issues entirely free alongside the tensor core pipe.

The cycle budget reveals a counter-intuitive architecture: while softmax nominally occupies 25% of wall time (130 cycles), the MUFU.EX2 instruction (18 cycles) and surrounding ALU operations execute entirely within the XU/SFU pipe, which co-issues at 0% overhead with the TC pipe (Chapter 3, §3.4.3). The real bottleneck is the Q×K^T GEMM at only 46% tensor core efficiency — each m16n8k16 MMA processes 16 of 128 K-elements, creating an 8-iteration dependent chain where accumulator feedback forces ILP=1 within the K-dimension. The three efficiency killers are: (1) tile setup overhead (mbarrier + ldmatrix per K-step), (2) zero K-reuse (operands loaded 8× but each used only once per MMA), and (3) smem-to-RF transfer cost of 11 cycles per MMA × 8 MMAs = 88 cycles overhead on 127 cycles of useful compute — a 69% overhead fraction that is structurally unavoidable at K=128.

This analysis establishes that FlashAttention efficiency on SM_120 is structurally capped by head_dim: at d=128 the K-loop has only 8 iterations (46% eff), at d=256 it would double to 16 iterations (~60–70% projected), while standard K=4096 GEMM achieves 95–100% via full K-reuse amortization. The SM_120's lack of Tensor Memory (TMEM) bypass — present on SM_100a via tcgen05 — adds +27.5% overhead on the smem→RF path, further penalizing the short K-loop. Optimization efforts should focus on maximizing the MMA-to-overhead ratio (larger tiles, more ILP from M/N parallelism) rather than softmax fast-math, which provides negligible benefit since the computation already runs free in otherwise-idle pipe slots.

8.3.2 Microarchitectural Analysis: Why Thin-K GEMMs Are Structurally Inefficient

The Q×K^T GEMM efficiency of only 46% at K=128 (Case 62) is not a software limitation — it is a structural consequence of the mma.sync m16n8k16 tile shape interacting with the SM_120 pipeline. The 8 dependent MMAs forming the K-loop accumulator chain force ILP=1 within the K-dimension, while three per-step overhead sources consume 69% of useful compute time.

The fundamental mechanism is the K-loop dependency chain: each mma.sync.m16n8k16 instruction processes 16 elements of the K-dimension and writes its result to an accumulator register that the next MMA reads as input. At K=128, this creates an 8-iteration serial chain where each MMA must wait for its predecessor's 29-cycle latency — yielding 232 cycles per output tile at ILP=1. While M/N-dimension parallelism (launching multiple output tiles) can raise ILP to 4 and reduce per-tile cost to 127 cycles, the three per-K-step overheads remain inescapable: (1) tile setup (mbarrier + ldmatrix) executes once per K-step regardless of tile count, (2) zero K-reuse means operand data is loaded 8 times but consumed only once per MMA with no register temporal locality, and (3) the smem→RF transfer path costs 11 cycles per MMA × 8 = 88 cycles overhead on 127 cycles of useful compute — a 69% overhead fraction that no software optimization can eliminate at K=128.

The efficiency scales directly with K-dimension depth: at K=4096 (256 MMAs per tile), these same fixed overheads amortize over 32× more compute iterations, achieving 95–100% efficiency. This explains why attention's Q×K^T (K=head_dim=128) is structurally capped at 46% while the Score×V multiplication achieves 61% — its effective K equals the block column count Bc (typically 64–256), providing a longer reduction chain. However, Score×V is still limited by its small output N dimension (=head_dim=128), which constrains tile parallelism. Increasing head_dim to 256 would double the K-loop to 16 iterations, projecting 60–70% efficiency, but this is an algorithmic change outside the hardware's control.

8.3.3 Softmax Is Architecturally Free: The SFU Co-Issue Window

The online softmax computation (max reduction → subtract → exp2 → multiply scale) costs 130 cycles per tile in wall-clock time. However, the SM_120 co-issue architecture allows these operations to execute entirely within MMA latency stall windows: the SFU (MUFU.EX2, 18 cycles) co-issues at 100% with TC operations, while FMA instructions achieve 93.6% overlap — making the net softmax overhead effectively zero.

The mechanism exploits SM_120's multi-pipe architecture: during each 29-cycle MMA latency window (while the tensor core computes but results are not yet available), the warp scheduler can dispatch independent instructions to the FMA and XU/SFU pipes at zero additional cost. The softmax operations — MUFU.EX2 (exp2, 18 cycles in XU/SFU), sub.f32 and mul.f32 (4 cycles each in FMA) — execute precisely in these idle slots. The co-issue matrix from Chapter 3 §3.4.3 confirms: XU+Tensor = 0% overhead (perfect overlap), FMA+Tensor = 6.4% overhead (93.6% overlap). This means the 130 cycles of softmax wall-clock time contributes only ~0.5 visible cycles per row — a 0–3% net overhead rather than the 25% that a naive analysis might suggest.

This architecture has a profound implication for kernel optimization: softmax fast-math variants (approximate exp2, reduced-precision accumulation) cannot improve FlashAttention throughput because the standard MUFU.EX2 path is already executing for free in an otherwise-idle pipe. Reducing the SFU instruction from 18 to 8 cycles gains nothing when both latencies fall entirely within the 29-cycle MMA stall window. Optimization effort should instead target the thin-K GEMM bottleneck (§8.3.2) which is the actual throughput limiter — the softmax computation is architecturally invisible.

8.5.1 Microarchitectural Analysis: The Roofline Defines Three Workload Classes

The roofline data from Case 25, combined with the M-scaling curves, definitively establishes three workload classes on SM_120, each with a distinct optimization strategy. The classification is determined by the relationship between a kernel's arithmetic intensity and the precision-specific crossover AI*, but a critical third class emerges when thin dimensions prevent full TC utilization regardless of the AI position.

The three classes correspond to distinct hardware regimes: Class 1 (compute-bound) occurs when AI exceeds the precision-specific crossover — the tensor core pipeline at 15.88 cyc/MMA throughput becomes the sole bottleneck, and optimizations target ILP, operand reuse, and tile sizing while memory bandwidth is irrelevant. Class 2 (memory-bound) applies when AI falls below crossover with large data volumes — DRAM bandwidth at ~860 GB/s (or L2 at 2–3× amplification for cached matrices) determines throughput, and the levers are weight quantization, L2 residency engineering, and batching. Class 3 (mixed/thin-dimension) is the most insidious: the kernel's AI suggests it should be memory-bound, yet the actual bottleneck is structural TC underutilization from thin K or M×N dimensions. FlashAttention epitomizes this class — its AI of ~50 places it below the FP16 crossover of 220, but the true limitation is the 46% TC efficiency from K=128 thin tiles, not insufficient memory bandwidth.

The attention paradox: FlashAttention's per-tile arithmetic intensity (~50 FLOP/byte) places it below the FP16 crossover (220), classifying it as memory-bound. Yet the bottleneck is NOT DRAM bandwidth — it is thin-K TC underutilization. This violates the simple roofline model because the roofline assumes perfect compute efficiency. When TC utilization is only 46% (due to K=128 thin tiles), the effective compute ceiling drops to 0.46 × 271 = 125 TFLOPS, which shifts the effective crossover to AI ~102. At AI~50, attention is thus operating at ~50% of its (reduced) compute ceiling — correctly classified as "structurally inefficient" rather than simply "memory-bound."

KI-8.2: Attention Limited by Thin-K/Thin-N GEMMs, Not Softmax

  • Mechanism: Small-M GEMM is launch-latency dominated, not compute-limited. Each cuBLAS call incurs 2-5 us of launch overhead that dominates when the actual compute takes only 3-5 us.
  • Implication: Persistent kernel scheduling with work-stealing eliminates per-expert launch overhead. Grouped GEMM with batch>=8 tokens per expert is the minimum for reasonable efficiency. Router GEMM should be fused with the preceding layer.
  • Confidence: H

KI-8.5: All Normalization+Activation Fusions Save 1.3-1.9x from Eliminating Memory Round-Trips

  • Phenomenon: Fused RMSNorm+SiLU 1.28-1.41x faster; fused QK RMSNorm+RoPE 1.5-1.9x; fused LayerNorm+SiLU 1.5x; fused bias+activation+residual 2.4-3.7x. All are memory-BW-bound.
  • Mechanism: Each unfused kernel requires one full DRAM round-trip (read input, write output). Fusion eliminates intermediate materializations. The benefit is proportional to the number of eliminated memory passes, not compute savings.
  • Implication: Always fuse normalization, activation, and residual operations. The fusion benefit is purely from bandwidth savings, making it universally beneficial regardless of sequence length or model size. The 2.4-3.7x speedup for bias+activation+residual (3 operations -> 1 kernel) confirms the linear scaling of fusion benefit with operation count.
  • Confidence: H

  • Phenomenon: Manual split-K always performs worse than standard cuBLAS. Split-16 loses 30-60%. Even thin M=64 shapes achieve 70 TFLOPS without splitting.
  • Mechanism: cuBLAS internally optimizes K-splitting via stream-K strategies. External split-K via multiple API calls adds atomic reduction overhead and wave quantization penalties that exceed any load-balancing benefit on the 110-SM chip.
  • Implication: Never manually split-K via multiple cuBLAS calls on SM_120. For custom kernels requiring explicit split-K, use single-launch kernels with shared-memory reduction rather than multi-launch approaches.
  • Confidence: H

KI-8.7: Causal Mask (selp) is Free; Additive Mask Costs +40.6%

  • Implication: All attention implementations should use predicated (selp-based) causal masking. Additive masking should only be used when relative position biases require actual floating-point addition.

  • Cross-reference: The 46% Q x K^T efficiency is caused by thin-K: with K=128, each m16n8k16 MMA processes only 16 of 128 K-elements, requiring 8 sequential MMAs with limited inter-tile parallelism. The softmax SFU (MUFU.EX2, 18 cyc) runs 100% free during MMA stalls (TC+XU 0% overhead, [S22/case_50]), so softmax is NOT the actual bottleneck despite occupying 25% of wall time.
  • Implication: Optimizing FlashAttention on SM_120 should prioritize: (1) maximizing Q x K^T tile efficiency via larger Br tiles (more rows per MMA batch), (2) using MMA-based rowsum (96x faster than shuffle-based reduction) instead of scalar warp-reduction, (3) using selp causal mask (free) instead of additive mask (+40.6%). Softmax SFU optimization (fast_math) has minimal impact because EX2 already co-issues free with MMA.
  • Confidence: H

8.9 Fallacies and Pitfalls

Following CAQR §1.12, this section catalogs common misconceptions refuted by SM_120 measurements.

Fallacy: "FP16 ALU packing provides 2× throughput over FP32." FALSE on SM_120. HFMA2 achieves 1 FP16 element/cycle = identical to FP32 FFMA rate. The 2-cycle issue interval time-multiplexes both FP16 lanes through the same 32-bit datapath. FP16 throughput advantage comes exclusively from Tensor Cores (258× ALU rate). [source: [S5/case_39], [S9/case_1]]

Fallacy: "Higher occupancy always improves performance." FALSE. Single warp at ILP=4 achieves 92% of peak FP32 throughput. 48 warps can regress 19% for mixed workloads due to L1/L2 cache contention. [source: [S11/case_122], [S21/case_53]]

Fallacy: "Softmax is the FlashAttention bottleneck." FALSE. NCU profiling shows softmax/rescale produces 49% of FlashAttention instructions (FMA+ALU), but SFU (MUFU.EX2) co-issues 100% free with TC operations. The actual bottleneck is thin-K GEMM at 46% efficiency. [source: [S25/case_12], [S19/case_62]]

Pitfall: "Using asm volatile with memory clobber prevents DCE." FALSE. ptxas eliminates asm volatile("" :: "f"(val) : "memory") entirely. Must use global memory store for anti-DCE. [source: [S21/case_15]]

Pitfall: "Setting stall counter below write-back latency merely reduces performance." FALSE. It causes silent data corruption — no CUDA error, no trap. Only stall=0 triggers the hardware safety net. [source: [S11/case_132]]

Fallacy: "TMA multicast works on all Blackwell GPUs." FALSE. GB202 (SM_120) lacks the LRC multicast crossbar. All multicast calls are SW-emulated at 2400-7500× penalty via CALL.ABS.NOINC. [source: [S10/case_28]]

8.10 Summary

SM_120 demonstrates a well-defined performance hierarchy across workload types:

  1. Compute-bound GEMM achieves 94-101% of theoretical peak for large shapes (M>=1024) across all precisions, with CUTLASS MXFP4 slightly exceeding theoretical due to clock boost effects.

  2. Memory-bound GEMV saturates DRAM at 98% utilization for N>=8192, with L2 providing 1.5-1.7x amplification for smaller working sets.

  3. Attention workloads are structurally limited to 46-61% efficiency by thin-K GEMM shapes, not by softmax computation.

  4. Synchronization overhead ranges from 5% (MoE dispatch at production scale) to 20.6% (DeepGemm pipeline at moderate compute loads).

  5. Power efficiency favors tensor core operations: MX-FP4 achieves 12.0 TFLOPS/W while FP32 FFMA achieves only 0.83 TFLOPS/W -- a 14.5x efficiency gap. [SPECULATIVE] This suggests that future architectures will further widen the TC/CUDA-core efficiency gap, potentially deprecating CUDA-core FP32 paths entirely for AI workloads.

The structural ceiling imposed by the smem-to-RF data path (KI-8.8) represents the single most impactful architectural limitation of SM_120 compared to SM_100. The lack of Tensor Memory forces all operands through a serialized ldmatrix-to-register-to-MMA pipeline, consuming 27.5% of each MMA's cycle budget. Operand reuse remains the primary lever for approaching theoretical peak on this architecture.
