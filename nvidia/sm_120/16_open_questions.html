CHAPTER 16 · SM_120 · GB202 BLACKWELL

Open Questions

This chapter catalogs unresolved questions identified during the first-round SM_120 (GB202) microbenchmark campaign (Sections 0-23, 1504 cases) and documents the results of the targeted follow-up experiments (Section 24, 31 cases) designed to resolve them. Of the 8 original open questions testable on SM_120 hardware, **all 8 are now fully resolved** — OQ-3 (FP64 EU placement) was reclassified from PARTIALLY RESOLVED to RESOLVED on 2026-06-11 after the lane-mask dichotomy experiment ([temp/fp64_eu_dichotomy_test{,_v2}.cu]) decisively showed 4 per-SMSP EUs (case_16 was correct; case_50/13/17 were sparse-lane-mask measurement artifacts). The follow-up campaign yielded several architectural discoveries that update the findings in Chapters 2, 3, 5, 7, and 9.

16.1 Resolved Questions

OQ-1: L1 Cache Actual Capacity and Dynamic Partitioning [RESOLVED]

Original conflict: 48 KB from working-set sweep [S8/case_72] vs ~80-96 KB from fine-grained sweep [S8/case_84].

Resolution (Section 24.1, Cases 1-6):

The L1 data cache has an effective capacity of 89 KB under default configuration, sharing 100 KB of unified SRAM with texture and non-coherent caches. The 48 KB figure from [S8/case_72] reflected a coarse sweep granularity; the true cliff is at 89 KB with 1 KB precision.

Measured architectural parameters:

Parameter Value Source
L1 effective capacity (default, no smem) 89 KB (cliff at 90 KB, 1 KB precision) [S24/case_1]
L1 hit latency 43.86-43.99 cyc (constant 32-89 KB) [S24/case_1,5,6]
Physical SRAM per SM 100 KB (102,400 bytes via cudaGetDeviceProperties) [S24/case_4]
L1 + texture sharing 1:1 byte displacement — 24 KB tex reduces L1 from ~88 KB to ~64 KB [S24/case_2]
L1 + NC sharing Same SRAM — displacement at WS + NC ~ 96 KB [S24/case_3]
Carveout API effect Functional — 1:1 byte trade-off, cliff shifts 88→56→<32 KB as smem 0→32→64 KB [S24/case_4]
L1 disabled threshold 64+ KB shared memory completely disables L1 [S24/case_4]
NCU validation Zero capacity misses for WS <= 88 KB; 415 capacity misses at WS = 96 KB [S24/case_5]
Associativity Fully associative (>= 710-way) — cliff is stride-independent [S24/case_6]

Detailed L1 capacity curve (pointer-chase, 1 KB steps):

Working Set (KB) cyc/chase Level
32-89 43.86 L1 hit (flat)
90 48.60 Transition onset
92 73.41 --
96 131.18 --

128 274.16 L2 deep

Texture displacement curve (24 KB tex1Dfetch co-load):

WS (KB) No Texture (cyc) With 24KB Texture (cyc) Delta
48 43.99 44.44 +1.0% (within L1)
64 43.99 44.33 +0.8% (within L1)

88 43.99 281.55 +540% (fully spilled)

Microarchitectural model: SM_120 has a unified 100 KB SRAM bank per SM that is dynamically partitioned among L1 data cache, texture cache, non-coherent cache, and shared memory. The cudaFuncSetAttribute(MaxDynamicSharedMemorySize) API controls the partition at coarse granularity (~32 KB steps). When shared memory is 0 KB, L1 gets ~89 KB (the remaining ~11 KB is consumed by tag storage, metadata, and reserved lines). At 64+ KB shared memory, L1 is completely disabled (minimum SRAM for L1 operation is ~36-48 KB). Texture and NC loads share the L1 allocation with 1:1 displacement — each KB of texture footprint reduces L1 by 1 KB. The cache is fully associative, eliminating stride-based conflict misses.

Impact on performance model: The analytical model should use 89 KB as the default L1 capacity (not 48 KB or 96 KB). For kernels using texture or NC loads, reduce by the concurrent texture/NC footprint. For kernels requesting >32 KB shared memory, use the measured carveout cliff table.

OQ-2: GDDR7 Bank Group Structure [RESOLVED]

Original question: Exact bank count, group count, row buffer policy unknown. Only 3x stride conflicts confirmed.

Resolution (Section 24.2, Cases 7-12):

GDDR7 on SM_120 uses 4-way bank group interleaving at 128B (cache-line) granularity with a closed-page (auto-precharge) row buffer policy. The earlier "3x conflict" finding is explained: 3 is not a multiple of 4, so 3×128B strides create bank-group conflicts.

Measured architectural parameters:

Parameter Value Source
Bank group count 4 per 1024B page [S24/case_10]
Interleave granularity 128B (cache-line aligned) [S24/case_7,10]
Optimal stride Multiples of 512B (4 cache lines) → full BW ~345 GB/s [S24/case_7]
Worst-case stride Non-multiples of 4×128B → 50-78 GB/s (6.9x degradation) [S24/case_7,10]
Row buffer policy Closed-page (auto-precharge) — seq/random ratio = 1.00 at all strides [S24/case_11]
Same-bank consecutive-access penalty +445 cyc (2.2× base) for strides < 640B — closed-page tRP wait, not row-buffer miss [S24/case_8]
Same-bank footprint (latency-effective) ~1 KB (transition at 640-1280B; above this, consecutive dependent accesses land on different banks) [S24/case_8]
Base DRAM latency 370.3 cyc (~177 ns at 2.092 GHz) [S24/case_8,11]
Channel BW uniformity CoV = 0.3% across all strides — L2 XOR hash eliminates all imbalance [S24/case_12]
Conflict mechanism L2 partition camping (not DRAM timing) — 128x DRAM traffic amplification at conflict strides [S24/case_9]

Complete stride-BW mapping (single-SM streaming read):

Stride (B) BW (GB/s) Bank Group Status
16 310 L2 spatial reuse (intra-cacheline; DRAM is closed-page so no row-buffer hit)
128 161 1-way bank spread
256 203 2-way bank spread
384 (3×128) 69 Bank group conflict
512 (4×128) 327 Full 4-way spread
768 (6×128) 62 Bank group conflict (worst)
896 (7×128) 57 Bank group conflict
1024 (8×128) 345 Full saturation
1536 (12×128) 213 Partial (12 = 4×3, mixed)
2048 339 Full saturation
4096 339 Full saturation

Bank group conflict mechanism: NCU profiling ([S24/case_9]) reveals that conflict strides cause L2 hit rates to drop from 99.4% (stride=1024B) to 4.6% (stride=384B) and 21.3% (stride=768B), with DRAM traffic amplification of 128x. The dominant BW loss is L2 partition camping (the XOR hash function maps conflict strides to the same subset of partitions), not direct DRAM bank-group timing effects.

Closed-page policy evidence: Single-warp pointer-chase shows identical latency (~371 cyc) for sequential and random access at all strides from 256B to 128KB (ratio = 1.00 ± 0.001). No row-buffer hit benefit exists. The +445 cyc penalty at strides < 640B [S24/case_8] is closed-page same-bank consecutive-access serialization — when the next dependent access targets the same bank, it must wait for that bank's tRP (precharge) to complete before re-activation, adding ~tRP cycles on top of the base 370 cyc ACT+CAS. The transition at 640-1280 B reflects the stride at which consecutive pointer-chase steps begin landing on different banks, exposing bank-level parallelism. This is not a row-buffer hit/miss effect (which cannot exist under closed-page) and not a bank-group conflict (which is a throughput, not single-warp-latency, phenomenon).

Caveat — Case 11 working-set bound: Although Case 11 allocates a 128 MB buffer, its sequential/random ring is capped at n_nodes <= 65536 and only 1024 chases are measured per trial, so the effective unique cache lines touched per trial is ~1224 × 128 B ≈ 156 KB, which fits entirely in the 96 MB L2. The "ratio = 1.00" thus reflects L2-resident behavior, not pure DRAM behavior, and on its own does not rigorously prove closed-page. The closed-page conclusion is more strongly supported by Case 8: under an open-page policy, small strides (<1 KB, mostly same-row) would yield the lowest latency due to row hits; the observed inversion (small strides are slowest at +445 cyc) is the signature of closed-page + same-bank serialization. A future follow-up should re-run Case 11 without the n_nodes cap so that ring length truly exceeds L2 capacity.

OQ-4: Warp Scheduler Priority Transition Details [RESOLVED]

Original question: CGGTY confirmed but transition mechanism under mixed stall types unclear.

Resolution (Section 24.4, Cases 18-21):

Measured scheduler behavior:

Property Finding Source
ALU+MEM priority ALU warps (4-cyc stall) always dispatched over MEM warps (~300 cyc). FFMA throughput unaffected (-0.4%). MEM warps benefit from co-scheduling (-19.7% latency) [S24/case_18]
Fair round-robin Completion time ratio tracks work ratio within 4%. cyc/iter stable at ~34 cyc across all warps at 1:2:3:4 imbalance [S24/case_19]
DEFER_BLOCKING Confirmed: barrier-blocked warp does NOT impede dispatch. W1 throughput = 1.000x baseline while W0 blocked at bar.sync. Barrier cost = 7 cycles [S24/case_20]
SB exhaustion switch Zero-overhead: ALU warp FFMA = 4.00 cyc (exactly baseline) when co-warp is SB-blocked with 6 outstanding LDGs [S24/case_21]

Scheduler priority model (updated from Chapter 7):

CGGTY Decision Flow (refined by Section 24):
  1. Greedy: continue current warp if eligible
  2. If current warp is ineligible (stall/SB/barrier):
     a. Immediately switch to any eligible warp (0 cycles)
     b. Among eligible: lowest warpID first (youngest)
     c. Barrier-blocked warps are deprioritized (DEFER_BLOCKING)
     d. SB-blocked warps are instantly deprioritized (no delay)
  3. Mixed stalls: short-stall (ALU 4-cyc) warps effectively have higher
     re-eligibility frequency than long-stall (MEM 300+cyc) warps
  4. Anti-starvation: inherent in round-robin for homogeneous ALU (4% tolerance)

OQ-5: L2 Hash Function Reverse-Engineering [RESOLVED]

Original question: Uniform XOR hash confirmed but exact function unknown.

Resolution (Section 24.5, Cases 22-24):

Hash function characterization:

Parameter Value Source
Number of partitions 12 (one per MC) [S24/case_23,24]
Hash type XOR-based [S24/case_23]
Lowest active hash bit Bit 10 (1024B granularity) [S24/case_23,24]
Period in N×128B stride sweep 8 (spikes at N=8,16,24,32) [S24/case_24]
Power-of-2 camping None (1 KB to 256 KB all safe) [S24/case_23]
Worst adversarial BW 49 GB/s (at strides 1408B, 1664B = 11×128, 13×128) [S24/case_24]
Uniform distribution BW 236 GB/s [S24/case_24]
Adversarial degradation 4.77x (not 12x — hash prevents single-partition concentration) [S24/case_24]
Worst concentrates on ~2.5 partitions (not 1) [S24/case_24]

Practical implications: Power-of-2 strides are always safe. Strides that are non-multiples of 8×128B (1024B) may cause up to 4.8x BW degradation. The worst-case adversarial stride still achieves 2.5x above the theoretical single-partition floor, proving the XOR hash cannot be fully defeated by stride-only attacks. For kernel optimization, ensure access strides are multiples of 1024B to guarantee uniform partition distribution.

OQ-7: Constant Cache and I-Cache SRAM Sharing [RESOLVED]

Original question: Whether constant cache and I-cache share physical SRAM.

Resolution (Section 24.6, Cases 25-27):

Constant cache and I-cache use separate physical SRAM on SM_120. Three independent interference tests all confirm zero cross-pressure effects:

Test Const Effect on I-cache I-cache Effect on Const Interference
Case 25: Const saturation (64 KB LUT) Overhead ratio 2.603, stable across all iters -- None
Case 26: I-cache pressure (32 KB code) -- 0.0% delta at all LUT sizes (4-64 KB) None
Case 27: Simultaneous saturation 44,360 cyc (combined) / 44,296 cyc (const-only) = 1.001x -- None

Additional finding: Effective constant cache capacity is approximately 4-8 KB (34 cyc hit at 4 KB, rising to 99.4 cyc at 64 KB).

OQ-8: Per-SM Power Gating Granularity [RESOLVED]

Original question: Whether individual pipes (TC, SFU, LSU) within an SM can be independently power-gated.

Resolution (Section 24.7, Cases 28-31):

SM_120 uses clock gating, not power gating at the functional-unit pipe level. Pipe switching is instantaneous with zero startup latency.

Per-pipe power characterization:

Pipe W/SM (incremental) Total at 110 SMs Source
FMA (FFMA) 0.23 ~25.3 W [S20/case_12]
TC (HMMA) 0.15 ~16.5 W [S20/case_12]
LSU (LDG/STG) 0.065 ~7.1 W [S24/case_30]
SFU (MUFU) ~0.23 (= FMA) ~25.3 W [S24/case_28]

Power model (step function, not linear):

P_total = P_dvfs_idle (13.3W)
        + P_clock_domain (2.6W, when clocks locked)
        + P_activation (44.7W, step function when ANY SM active)
        + P_per_SM (<3 mW/SM, below nvidia-smi resolution)

Clock gating evidence ([S24/case_31]): FMA pipe latency = 4.00 cyc immediately after 0, 10, 100, 1000, or 10000 HMMA iterations of TC warmup. Zero startup penalty rules out power gating (which would show 10-100 ns ramp-up).

TC+FMA co-issue power ([S24/case_29]): Adding FMA alongside TC increases total power by only 0.40 W for 110 SMs (0.004 W/SM). The idle FMA pipe is clock-gated during TC-only operation and resumes instantly when needed.

16.2 Resolved on Follow-up

OQ-3: FP64 EU Placement [RESOLVED 2026-06-11 — 4 per-SMSP, all-lane regime decisive]

Original question: 2 FP64 EU/SM confirmed, but placement (per-SMSP vs shared) unclear.

Final answer: 4 independent FP64 EUs per SM, one per SMSP. The earlier "2 shared EU" reading was a sparse-lane-mask measurement artifact, not real EU sharing. Decisive evidence: a controlled lane-mask dichotomy experiment on SM_120 server1 (2026-06-11, 8 kernel variants, [temp/fp64_eu_dichotomy_test{,_v2}.cu]) where the only varied parameter is the lane mask:

Lane mask 1W cyc/DFMA 4W cyc/DFMA 4W contention Agg @ 4W (DFMA/cyc/SM)
lane==0 only 63.99 75.99 +18.8% 0.0526
all 32 lanes (real DGEMM) 63.99 63.99 0% 0.0625 = 4/64

#pragma unroll was tested orthogonally (256 vs 1000 DFMAs, with vs without unroll, all 4 cells × 2 lane modes = 8 kernels) and is a null variable. The lane mask is the sole determinant.

Reinterpretation of S24 cases:

Case Lane mask 4W result Real meaning
[S5/case_50] lane 0 only +19% contention Sparse-mask dispatch overhead (NOT EU sharing)
[S24/case_13] lane 0 only +19% contention Sparse-mask dispatch overhead (NOT EU sharing)
[S24/case_17] lane 0 only NCU cycles_active 4.76× vs inst 4.00× SMSP scheduler arbitrating sparse warps (NOT EU sharing)
[S24/case_16] all 32 lanes 0% contention Correct measurement of true 4-EU architecture

The original conflict was an algebraic coincidence: 4 / 76 = 0.0526 ≈ 2 / 64 = 0.0312 × 1.69. When case_50/13 assumed the 2-shared-EU model and computed "effective EU count" from agg = 0.0526, they got ≈2 — but the real cause was 4 EUs each running with a +19% scheduler tax under sparse mask, not 2 EUs at full speed.

Updated downstream model:

  • FP64 EU count: 4 per-SMSP
  • All-lane DFMA throughput: 0.0625 DFMA/cyc/SM = 4/64
  • FP64:FP32 ratio (real-workload): 1:32 dep-chain / 1:32 cuBLAS-pipelined (not 1:64)
  • SM_120 FP64 dep-chain peak (110 SMs, 2.1 GHz): 28.9 GFLOPS
  • SM_120 cuBLAS DGEMM peak: 0.96 TFLOPS [S18/case_3] (ILP-pipelined via tensor-core-style accumulator chains lifts dep-chain bound by ~33×)

Updated chapters:

  • Ch5 §5.2.4 / §5.2.7 — EU table + ASCII diagrams reflect 4-per-SMSP architecture and lane-mask dichotomy explanation.
  • Ch3 SM Microarchitecture — FP64 EU count consistent at 4 per SM.
  • Ch9 Power model — FP64 active power computed against 4-EU model.

Audit trail: see Ch5 §5.2.7 "Decisive Lane-Mask Dichotomy Experiment (2026-06-11)" for the full 8-kernel data table and ASCII diagrams of dispatch paths in both regimes.
